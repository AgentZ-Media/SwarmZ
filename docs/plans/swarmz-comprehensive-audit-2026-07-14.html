Anchored no-follow I/O

Agent output is not authority

CAPABILITY · ✓

Fail-closed authority

GIT · ✓

Repository code cannot hijack backend Git

OUTWARD · ✓

GitHub double gate

Default-off Rust-Gate, Drain-Lock, Credential-Redaction und eine zusätzliche Mission-/Autonomie-Grenze schützen Push, PR und Kommentare.

Delivered feature system

Das Feature-Set für große Vibe-Projekte ist implementiert.

Die Bausteine arbeiten als ein System: Intake erzeugt durable Tasks; Scheduler und Envelope begrenzen Attempts; Evidence Gates verifizieren; der Integration Train erzeugt den belastbaren Gesamtstand; Insights und Replay machen das Ergebnis erklärbar.

01 · FOUNDATION

Mission graph + scheduler

Persistente Tasks, Abhängigkeiten, Attempts, Priorität und Acceptance Criteria. Ein Queue-Scheduler aktiviert nur so viele Worker, wie Budget, System und Konfliktlage erlauben.

  • Bulk import: Text, Markdown, CSV, GitHub Issues
  • Critical Path, Waves, Pause/Resume/Cancel
  • Retry, Timeout, Stuck Detection; jeder Retry ist ein frischer Worker
  • Report Schema v2 mit Task-/Attempt-ID und SHA-Evidence
02 · SAFETY

Mission envelope

Einmalige, sichtbare Nutzerfreigabe statt alle fünf Turns manuell zu re-armen.

  • Task-, Token-, Zeit- und Parallelitätslimit
  • erlaubte Tools / Netzwerk / GitHub
  • Stop bei Regression, Konflikt oder Failure
  • bestehender Breaker bleibt Notbremse
03 · CONCURRENCY

Conflict radar

Konflikte vor dem Schreiben verhindern, nicht erst beim Merge entdecken.

  • File-intent + Worktree-Leases
  • Pfad-/API-Overlap-Heatmap
  • Contract Tasks vor Consumer Tasks
  • Baseline Drift und Semantik-Warnungen
04 · DELIVERY

Integration train

Ein „fertiger“ Worktree ist noch kein fertiges Produkt. Ein dedizierter Integrationscheckout erzeugt den belastbaren Gesamtstand.

  • Rebase/cherry-pick/merge queue
  • gestufte Regression pro Änderung
  • Checkpoints + automatischer Rollback
  • Conflict Resolver Agent + Human Gate
05 · QUALITY

Evidence gates

„tests_pass: true“ darf nicht allein die Wahrheit sein.

  • zentrale Acceptance Commands
  • unabhängiger Reviewer/Tester
  • Clean-checkout + combined regression
  • Exitcodes, Logs, Artifacts und CI-Link
06 · OPERATIONS

Runtime environments

Worktrees brauchen reproduzierbare Laufzeitkontexte, ohne Profile, Personas oder das alte Terminal-Grid zurückzubringen.

  • Setup-/cleanup-Skripte und Statuskarten
  • Port-, DB- und Schema-Isolation
  • Secret-Mapping ohne Secret-Kopie
  • Background commands mit bounded output
07 · KNOWLEDGE

Playbooks + task roles

Wiederholbare Qualität ohne dauerhafte Worker-Persönlichkeiten. Rollen sind begrenzte Briefings pro Attempt; nur der Orchestrator lernt.

  • Architect / Implementer / Tester / Security als Task-Rolle
  • Repo-Skills und Mission Templates
  • Mission Insights + Failure Clustering
  • user-approved Orchestrator learning updates
08 · SCALE

Multi-root missions

Ein Feature läuft oft durch Backend, Web, Mobile und Infrastruktur.

  • mehrere Repositories pro Mission
  • Cross-repo Dependencies
  • versionierte API Contracts
  • koordinierte PR-/Release-Reihenfolge
09 · CONTROL

Audit + replay

Jeder Status muss erklärbar und nach einem Neustart rekonstruierbar sein.

  • Append-only Events + Outbox
  • Prompt, Approval, Diff, Commit, Test
  • Exportierbarer Mission Report
  • Reconcile-Ansicht nach Crash/Update
