Product plan

GPT Realtime 2 Orchestration Layer for SwarmZ

Ziel: SwarmZ wird von einem Terminal-Tiler zu einer lokalen Agenten-Leitstelle. Der Nutzer spricht ein Ziel aus; der Orchestrator kennt alle Fenster, liest Kontext, verteilt Arbeit, sendet Prompts, stoppt riskante Aktionen und konsolidiert Ergebnisse.

Surface Voice + chat overlay above all panes
Model GPT Realtime 2 for low-latency speech and tool calls
Trust model Local-first state, explicit approvals for destructive actions
First milestone Read-only commander, then controlled write actions

North Star

No more prompt wrangling

The user says the outcome. SwarmZ turns it into scoped agent instructions.

Every pane has context

Each agent pane exposes runtime, cwd, git state, usage, terminal summary, last prompt and risk state.

One commander, many agents

The orchestrator can spawn panes, send prompts, read outputs, pause, close and request review.

Safety stays visible

Yolo agent execution is allowed, but destructive app-level actions still flow through an approval lane.

Product Flow

1. Speak Goal capture

"Build X, split it across three agents, review before PR."

2. Plan Task graph

Orchestrator proposes workers, scopes, branches and acceptance checks.

3. Act Tool calls

Create panes, worktrees, prompts, pauses, reads and close requests.

4. Observe Live context

Terminal streams are summarized into per-pane and global memory.

5. Decide Review lane

User approves risky operations, PR creation and final cleanup.

Context Model

The hard part is not sending prompts. The hard part is keeping a compact, truthful model of what every terminal is doing without flooding Realtime context. SwarmZ should store raw terminal scrollback locally, but send the model only a distilled state object plus targeted excerpts on demand.

Layer Stored locally Sent to Realtime Update trigger
Pane identity agent id, workspace, cwd, runtime, branch, worktree metadata always in compact state pane create/move/close
Terminal activity raw chunks + parsed recent lines ring buffer status, last command, last assistant/user prompt, error flags PTY output debounce
Semantic summary rolling summary per pane and per workspace summary + open questions + next action idle transition, large output, explicit read
Code state git status, changed files, diff stats, local-only commits file list + high-risk diff excerpts on demand git poll / worktree events
Global memory task graph, decisions, approvals, rejected actions active objective + current graph + constraints every orchestration action

Tool Surface For The Orchestrator

{
  "list_panes": "return compact state for every visible and hidden pane",
  "read_pane": "return recent terminal excerpt or semantic summary",
  "create_agent": "runtime, cwd, profile, startup, worktree options",
  "send_prompt": "paste text into a pane; optional submit",
  "interrupt_agent": "send Ctrl-C or escape sequence",
  "close_agent": "request normal close flow, never force-delete silently",
  "focus_pane": "switch workspace and focus pane",
  "create_worktree": "explicit branch/copy-env action",
  "git_snapshot": "repo status, changed files, ahead/dirty risk",
  "request_approval": "ask user before destructive or irreversible action"
}

Implementation Architecture

Frontend
Commander overlay

A slim global overlay in the title-bar layer with push-to-talk, transcript, suggested plan, active task graph and approval prompts. It should not cover terminal work unless the user opens the full view.

State
orchestration slice

Store `taskGraph`, `paneSummaries`, `approvals`, `commanderSession`, and `toolCallLog` outside the core agent lifecycle to avoid bloating `store.ts` further.

Backend
Realtime session bridge

Rust owns API keys and websocket/session setup. JS sends local tool results and audio frames; Rust brokers model events, transcripts and tool-call requests back to the app.

Context
terminal summarizer

Capture recent lines in `term-host`, classify blocks as user input, assistant output, shell command, test output, error, or permission prompt. Summaries update only when the pane goes idle or output crosses a size threshold.

Phased Delivery

Phase Scope Definition of done
MVP 0: context inventory Expose pane list, recent lines, summaries, git state Debug drawer shows what the commander would know
MVP 1: read-only voice commander Realtime voice, answer questions about current panes "What is agent 2 doing?" works without sending prompts
MVP 2: controlled actions Create panes, focus panes, paste prompts, no destructive actions User can say "ask Codex in a new worktree to review this branch"
MVP 3: task graph Planner creates worker/reviewer/tester nodes and tracks completion Fleet overview shows orchestration status, not just terminal status
MVP 4: PR workflow Diff review, test execution, PR draft creation, cleanup proposal Orchestrator can prepare a PR but asks before push/cleanup

Safety Rules

Always ask

Delete worktree, kill busy pane, push, force-remove branch, clear history, overwrite settings.

Ask when uncertain

Prompt submission to the wrong pane, huge paste, command that mentions secrets, network credentials.

Allowed directly

Read pane summaries, focus panes, create plain panes, create clean worktrees, paste draft prompts.

Open Questions

Question Recommended answer
Can Realtime read raw terminal scrollback? No by default. It gets compact summaries and can request excerpts from a named pane.
Where do summaries live? Tauri store for resumable commander memory; raw scrollback remains xterm/local only.
Does the commander use MCP? Yes for external tools; app-internal control should be native tools first, MCP-exportable later.
Can agents spawned manually be discovered? Yes. The orchestration state subscribes to the existing store, so every pane is visible.
