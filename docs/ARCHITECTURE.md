# SwarmZ architecture notes

Deep-dive companion to `AGENTS.md` — the full how-and-why per subsystem.
Update this file when behavior described here changes.

## Transport & backends

`src/lib/transport.ts` exports the native Tauri backend. Backend capabilities are declared in `backend-types.ts` and implemented in `backend-tauri.ts`. Native-only features may skip the backend interface and use `@tauri-apps/api` directly (e.g. `lib/dnd.ts`, `lib/openrouter.ts`, `lib/worktree.ts`).

## Store & persistence

`src/store.ts` holds all app state. Persisted: profiles (including agent runtime: `claude` / `codex` / `shell`; Codex standard profiles are backfilled when missing, led by `Codex · YOLO` with full-access permissions), usage history, workspace tabs (`Workspace`: name/order/defaultCwd + active tab, store key `workspaces`, debounced via `schedulePersistWorkspaces`), the grid snapshot for session restore (store key `grid`, see below) + app settings (`AppSettings`: last used folder, default runtime, default font size, default startup command, claude/codex/git path overrides, auto-update, restore-on-launch, voice-dictation prefs (engine cloud/local, hotkey mode, auto-submit, cleanup toggle/model/prompt, STT model, mic device id+label), worktree repo registry (`worktreeRepos`, written by the worktree feature, not the dialog), orchestrator prefs (provider, OpenRouter model, auto-submit, busy policy, scan roots — see the Phase-6 section) — edited in `SettingsDialog.tsx`, opened via the title-bar gear or `⌘,`) (Tauri store `swarmz.json`). Always mutate settings through `updateSettings` (merges + persists the whole object). In-memory only: floating terminals, focus mode, fleet/palette state. Window size/position persists natively via `tauri-plugin-window-state`.

New agents and splits go through the New Agent dialog and land in the active workspace; splits prefill it from the source pane (runtime, cwd, profile, startup), plain "new" prefills the configured default runtime + startup command (fresh installs default to Codex full-access mode; existing settings from pre-default-runtime builds are normalized to the previous Claude default; picking a profile overrides it) and cwd from workspace defaultCwd → profile default cwd → last used folder. The dialog resets its fields only on the opening edge (background store writes while it is open must never wipe typed input). Runtime path overrides are applied at PTY spawn (`applyRuntimePath` in `utils.ts` — replaces a leading `claude` or `codex` token only); binary-path overrides commit on blur and are stat'ed via `path_is_file` with inline feedback.

## Session restore

The live grid is snapshotted continuously (store key `grid`, debounced `schedulePersistGrid` in `store.ts`: each agent's restore-relevant fields incl. runtime and the latched `sessionId`, plus the per-workspace layout trees and active panes) and flushed on every quit path together with the workspaces + usage-history debounces (`flushAllPersists`, called in `lib/quit.ts` before the window actually closes). On launch `hydrate()` rebuilds panes from the snapshot only if `settings.restoreAgents === true` (default **off**) and no agent was already spawned; torn snapshots degrade gracefully **per entry** (snapshot carries `version: 1`; agents missing id/workspaceId are skipped, other fields get safe defaults, a corrupt tree only costs its own workspace, panes without their agent are pruned, agents without a pane are dropped, fully-pruned trees normalize to `null`).

The store file itself has a crash net: `storefile.rs` runs in Tauri `setup` before the webview loads — a healthy `swarmz.json` refreshes `swarmz.json.bak`, a corrupt one (torn write, disk-full) is moved aside as `swarmz.json.corrupt-<ts>` and the backup restored, so the plugin's silent treat-unreadable-as-empty behavior can no longer factory-reset the app.

Restored agents carry the transient `Agent.resume = sessionId` — applied only at PTY spawn (`Terminal.tsx` → `resumeRuntimeStartup` in `utils.ts`, using `claude --resume <id>` for Claude and `codex resume <id>` for Codex; compound commands and explicit resume forms pass through), so `agent.startup` stays clean for split-prefill and re-persisting. Claude resume keeps the same session id, so usage latches right back onto the same jsonl — and a latched session is parsed without the since-filter (`usage_for_session` in `usage.rs`), so stats and the context donut include the pre-restart turns despite the reset `createdAt`. Claude session latching is gated on real activity: an unlatched pane only runs session discovery once Claude first reported busy (`Agent.firstBusyAt`, stamped in `setActivity` from OSC 9;4), and discovery then first matches files born around that first activity (App.tsx usage refresh). Codex latching uses `~/.codex/sessions/**/*.jsonl` `session_meta.cwd` / `session_id` metadata and starts at pane creation because Codex does not rely on Claude's OSC 9;4 signal; already-latched sibling ids are excluded. Floating terminals are not captured.

## Workspaces

Workspaces are the top-level containers (tabs in the title bar, `WorkspaceTabs` in `TitleBar.tsx`): each has its own tiling tree (`layouts`/`activePaneIds` keyed by workspace id). Deliberately not bound to a project — `defaultCwd` only prefills the New Agent dialog; the first agent with a cwd auto-names the tab (`folderName`, until the user renames it) and sets `defaultCwd`. Tabs show an aggregated status dot (static muted = busy, faint = alive/idle — deliberately NOT green, which stays "just finished"/success per DESIGN.md; dimmer faint = empty) + `busy/total` count and an amber `⚑n` needs-you count, rename on double-click, reorder by drag, close via the X (revealed on hover AND keyboard focus) or middle-click (`CloseWorkspaceDialog.tsx` confirms when agents are inside; closing the last workspace spawns a fresh one — there is always ≥1). Structurally each tab is a `<div data-tauri-drag-region="false">` with two real button siblings (activate + close) — a close X nested inside the tab button was keyboard-unreachable and invalid ARIA. Shortcuts: ⌘1–9 switch, ⌘⇧[/⌘⇧] cycle workspaces, ⌘]/⌘[ cycle panes within the active workspace (layout order, wraps; in focus mode the zoomed pane moves along — `cycleActivePane` in store.ts, also in the ⌘K palette), ⌘⇧N new, ⌘⇧W close. Dragging a pane header onto a tab moves the agent there (`moveAgentToWorkspace`; tab highlight via `tabDropTarget`).

## Workspace presets

Store key `workspacePresets`, helpers in `lib/presets.ts`: named layout trees whose leaves are pane templates (`PresetPaneNode`: cwd unset = *inherit* the folder asked once at load via `LoadPresetDialog.tsx` — skipped when every pane is pinned; startup unset = default startup command, `""` = plain shell; plus name/profileId/color). Seeded with Solo/1×2/2×2/1+2 only when the key is **missing** — an emptied list stays empty. Loaded from the empty-workspace screen (`EmptyState` in `WorkspaceLayer.tsx`, cards with `PresetThumbnail.tsx` mini-grids; ⌘⇧N → empty tab → cards covers the new-workspace flow): `applyPreset` builds all agents + the live layout in one pass, fills the active workspace if still empty (else a new tab) and names the tab after the preset (`renamed: true`). Created via ⌘K → "Save workspace as preset" (`SavePresetDialog.tsx` → `presetLayoutFromGrid` snapshots the live grid; agent names only carry over when user-renamed). Managed in Settings → Presets (rename, per-pane folder/command, remove panes).

## Terminals & PTYs

**Terminals live outside React** in `src/lib/term-host.ts` (xterm + PTY per id; `TerminalView` only attaches/detaches the host element). Unmounting a pane detaches the DOM but keeps the terminal running — that is what lets panes remount across workspace grids without losing scrollback/PTY. PTYs die exactly on store removal: `removeAgent` / `removeFloatingTerminal` → `destroyTerm` (NOT on unmount). The WebGL addon disposes itself on context loss (browser caps live WebGL contexts; with every workspace mounted that's reachable) and xterm falls back to the DOM renderer. A failed `pty_spawn` (deleted cwd, removed worktree) writes an error banner into the terminal and flips the pane to `exited` instead of hanging on "starting".

PTY input goes through a per-session writer thread in `pty.rs` (channel send, single consumer): `pty_write` stays a fast sync command (keystroke ordering preserved) and can never block the main thread on a PTY whose foreground process stopped reading. `pty_kill` is async (portable-pty's kill sleeps up to ~250 ms in its SIGHUP grace loop) and sessions are killed outside the manager mutex + reaped on a detached thread (no `<defunct>` zombies).

All workspace grids stay mounted in `WorkspaceLayer.tsx`; inactive ones are `visibility:hidden` + `pointer-events-none` (never unmounted, never resized — hidden agents keep working). **Fleet overview / ops board** (⌘E / grid button): the same wrappers are scaled into a card grid with pure CSS transforms (no size change → no PTY resize, terminals stay live), with `FleetCard` chrome on top and click-through that maps the cursor onto the exact pane (`paneRects`) and jumps to it. Cards deliberately keep each workspace's **true spatial layout** — never a sorted/priority grid — so a card is exactly what zooming into it reveals (spatial truth beats triage sorting; the Deck's triage queue and the board's Tab-cycle own "what's next"). All chrome renders on the *unscaled* overlay sibling (only the live wrapper underneath is transformed), so it stays readable at any card size: the workspace name pill + static amber (`--attn`) card border + ⚑ on attention, plus **per-pane ops chrome** aligned via the same `paneRects` (inset-corrected by `8·scale` for the wrapper's `p-2`) — a bottom-left pill with the signal-triad status chip (shared `paneSignal` from `AgentPane.tsx`), a faint `project · branch` line (worktree panes: main-repo name + worktree branch), the pane's last fleet event + age (`lastEventForPane` over `lib/events.ts`, omitted when none) and a 3 px context bar + % from usage (`--warning` fill ≥ 75 %; omitted for shells/no-usage panes). Each pane pill degrades like the pane headers, via container queries on the pane's on-screen rect: the event line drops first (`@max-2xs`), then the project line (`@max-3xs`). A quiet `FleetHeader` pill floats top-center (`Fleet · N agents · N working · ⚑ N need you · N workspaces`; needs-you segment only when > 0) — counts come from `fleetCounts` in `orchestrator/snapshot.ts`, the same census `fleetSummaryLine` uses. **Keyboard**: arrows move a blue `--ring` selection ring geometrically (nearest pane center in that direction across all cards, score = ahead + 2·orthogonal drift, no wrap), Enter jumps into the selected pane (the exact click path: `setActiveWorkspace` + `focusAgent` + `focusTerm`), Tab/⇧Tab cycles needs-you panes only (oldest first, shared `lib/triage.ts` ordering with the Deck), Home/End selects the first/last triage target, Escape/⌘E exit (entering blurs the focused terminal so keys arrive at the window; an open `role="dialog"` or focused text field owns the keyboard instead). Initial selection on open = the oldest needs-you pane, else the active pane; the selection is purely visual chrome — no real focus, no PTY resize. Exit animates via a render-phase `fleetClosing` flag, mirroring the focus-mode pattern.

## Command palette & custom commands

**Command palette** (⌘K, `CommandPalette.tsx`, cmdk): fuzzy-jump to any agent/workspace + all global actions; custom commands surface as their own group **only while a search is typed** (folder commands of the target pane + globals) and run with the same paste/⌘↵ semantics as ⌘⇧K. **⌘⇧A** (`attentionJump`) cycles through agents that need input (bell attention or OSC "waiting") across all workspaces.

**Custom commands** (⌘⇧K, `InsertCommandPalette.tsx`): user-saved prompt snippets, global + per folder (store key `customCommands` → `CustomCommandsData`, folder lists keyed via `presetKey` — deliberately separate from the floating-terminal `commandPresets`, which are executed shell commands). Selecting **pastes** into the target pane via `term.paste()` without submitting; ⌘Enter submits via a **separate** `ptyWrite "\r"` after the paste (a `\r` inside the bracketed paste is only a literal newline in claude's input box) — the paste/submit core is shared with ⌘K in `lib/insert-command.ts` (`insertCommandText`); a `{{input}}` command picked in ⌘K detours through the picker's form via store `commandPickerPreselect` (in-memory, consumed on open). Target = focused agent, else active pane; agent panes only (floating terminals have no focus tracking in the store — extending later is just target resolution, `insert()` only needs a pty id). Placeholders live in `lib/command-vars.ts`: `{{folder}}/{{cwd}}/{{branch}}/{{agent}}` fill from the pane, `{{input:Label}}` asks inline before inserting (same label twice = same value), unknown `{{…}}` stay literal. Managed in Settings → Commands, quick-add directly in the picker. The ⌘⇧K keydown branch in `App.tsx` must stay **before** the plain ⌘K branch (which doesn't check shift). While any dialog is open, global shortcuts early-return (only ⌘K/⌘⇧K may still close their own palettes) — keydown bubbles through Radix's focus trap, and an unguarded ⌘W would kill the pane behind the dialog; ⌘W is also always `preventDefault`ed so an empty workspace can't fall through to the native Close Window menu item.

## Quick notes

**Quick notes** (⌘N, title-bar note button, `QuickNotesPanel.tsx`): a right-side slide-in drawer (same overlay/z-40/`animate-slide-in-right` pattern as the usage drawer) holding lightweight checklists — one **global** list plus one **per project folder** (store key `quickNotes` → `QuickNotesData`, folder lists keyed by repo root). The drawer opens in the scope of the active pane via `activeProjectRoot()` in the store — a worktree pane resolves to its **main repo**, never the `.worktrees/<slug>` path; fallback is the workspace `defaultCwd`, else Global — and shows a chip per scope (Global, the active project, plus every folder that already has notes). Items (`NoteItem`) check off, edit inline (click the text; committing empty deletes), reorder by dragging the grip (sibling-midpoint pattern like the workspace tab strip) and can toggle to plain text lines without a checkbox (`plain`); "Clear done" drops every checked item, emptied folder lists drop their key. Every mutation persists immediately (no debounce — nothing to flush at quit). The drawer is a `role="dialog"` so the global-shortcut guard kicks in; ⌘N closing it is special-cased in the dialog branch of the `App.tsx` keydown handler (like ⌘K/⌘⇧K), and the new-note input autofocuses on open so capture is type-Enter-done.

## Voice dictation

Hold mode = plain ⌘ push-to-talk, toggle mode = ⌘⇧M, per `dictationHotkeyMode`; or the mic button in pane/floating headers. Hold mode arms on Meta keydown and only opens the mic after ~250 ms held **alone** (`armHoldDictation` — ⌘-shortcuts never trigger it; any other keydown cancels silently via `cancelHoldDictation`; arming is skipped while a dialog is open or a real text field — not xterm's helper textarea — has focus), Meta keyup or window blur finishes (`finishHoldDictation`). A `holdGen` generation counter closes the getUserMedia race: if ⌘ was released while the mic was still opening, the recording is aborted instead of running hot. Recordings under ~1 s are discarded instead of transcribed (OpenRouter rejects near-empty audio). Recording happens in the webview (`lib/dictation.ts`, outside React like term-host — getUserMedia + MediaRecorder, rotated into ~50 s segments because the OpenRouter transcription API wants ≤60 s uploads, 5 min hard cap; `settings.dictationMicId` picks the input device as an **ideal** deviceId constraint, so an unplugged saved mic falls back to the system default instead of failing — chosen in Settings → Voice via `listMicrophones()`, which enumerates lazily on picker open only (briefly opening the mic if WebKit still hides labels) and persists id + label so the selection renders without re-probing; `stopDictation` awaits an in-flight rotation so the boundary segment is never lost; the AnalyserNode feeds the waveform pill in `Dictation.tsx`, rendered inside the target's terminal container). getUserMedia opens the mic with **all WebRTC processing disabled** (`echoCancellation`/`noiseSuppression`/`autoGainControl` all `false`): in WKWebView the APM ramps up for ~0.6–1 s after the track opens and delivers pure silence — push-to-talk was swallowing the first word; raw capture gets real samples within ~150 ms. The trade-off is a ~10–30× quieter signal (the APM is also what made it loud), so the waveform pill normalizes against a slowly-decaying running peak instead of a fixed gain — don't reintroduce processing or a constant multiplier. An in-flight transcription can be cancelled (`cancelTranscription` — mic button or the pill's Cancel).

Two engines (`settings.dictationEngine`, default `"openrouter"`): cloud transcription, optional LLM cleanup and the model catalog run natively in `src-tauri/src/openrouter.rs`; the **local engine** (`"local"`, `src-tauri/src/localstt.rs` + `lib/local-stt.ts`) runs NVIDIA Parakeet TDT 0.6b v3 int8 on-device via transcribe-rs/ONNX Runtime (statically linked at build time through ort's `download-binaries` — no Python or system deps ever) — the ~670 MB model (4 files from HF `istupakov/parakeet-tdt-0.6b-v3-onnx`, **pinned to a commit revision with per-file SHA-256s** in `MODEL_FILES` — size + hash are verified before the `.part`→final rename, and `is_installed` is size-aware so a corrupt install self-detects) is downloaded once from Settings → Voice into `~/Library/Application Support/SwarmZ/models/` (`.part` + rename, resumable via HTTP Range, 15 s connect/30 s read timeouts so a stalled connection errors instead of wedging, progress via `localstt://progress`), loaded lazily on first dictation (~1.5–2 GB RAM, cached until unload — switching the engine back to cloud unloads it; the loaded flag is an `AtomicBool` so `local_stt_status` never blocks on the inference mutex) and fed 16 kHz mono PCM that the webview decodes/resamples from the recording via OfflineAudioContext (`blobToPcm16k`).

Mic gating is engine-aware (`selectDictationReady`: local → store `localSttStatus.installed`, cloud → `openrouterStatus`); the cleanup pass always goes through OpenRouter, so with the local engine it only runs when a usable key is also stored. The API key lives in the macOS Keychain (service `SwarmZ-OpenRouter`, via the Security framework — never in a subprocess argv; only limits.rs still uses the `security` CLI because it reads *Claude's* item) and never reaches JS — `lib/openrouter.ts` is native-only direct `invoke` (skips the backend interface, like dnd.ts). The transcript is **pasted** like the insert picker (`term.paste`, separate `\r` only with `dictationAutoSubmit`). Cleanup uses structured output (`json_schema {cleaned}` + `require_parameters`) with `reasoning: {effort: "minimal"}` — NOT `enabled: false`, Gemini 400s on that ("Reasoning is mandatory … cannot be disabled"); models that reject "minimal" get one retry without the reasoning field (only on a reasoning/parameter rejection — never on timeouts/429/5xx) — and a user-editable prompt whose default forbids translating — dictation must stay language-agnostic. All mic UI is gated on store `openrouterStatus` (fetched at launch and on key save; `valid: null` = present-but-unverified counts as usable, only a 401/403 hides it). One dictation at a time (store `dictation`, in-memory). Mic permission comes from `src-tauri/Info.plist` (`NSMicrophoneUsageDescription`).

## Usage & limits

Claude usage is parsed from `~/.claude/projects/*.jsonl` by `src-tauri/src/usage.rs`. Parsing is **incremental**: a per-(file, since) offset cache only reads appended bytes (entries of deleted files are evicted during `usage_totals` walks); costs/by_model are recomputed from cached counters on every read (so pricing updates need no cache invalidation). Codex usage is parsed from `~/.codex/sessions/**/*.jsonl` by `src-tauri/src/codex_usage.rs`, using `session_meta`, `turn_context`, `event_msg/token_count`, `task_started` and `task_complete` rollout events for session id, cwd, model, tokens, context, limits and activity. The usage commands are async + `spawn_blocking` — the first call after launch parses the whole backlog and must not freeze the UI. Claude pricing is fetched live from the OpenRouter model catalog (daily refresh) with a hardcoded per-family fallback for offline; Codex ChatGPT-plan rollouts do not include reliable USD cost, so Codex cost stays zero unless a future parser can estimate it honestly.

**Context-window (donut) inference** (`context_limit` in `usage.rs`): the JSONL records only the **bare** model id (`claude-opus-4-8`) even on 1M sessions — the `[1m]` suffix is stripped from the response `message.model` — so the 200k-vs-1M window is inferred, in order: explicit `[1m]` in the id → `~/.claude/settings.json` `model` being this model's `[1m]` variant → the model having actually been run at 1M before (any `projects.*.lastModelUsage` key with a `[1m]` suffix in `~/.claude.json`, harvested by `onem_models()`, cached against that file's mtime) → observed context > 200k. The `~/.claude.json` lookup is what makes the donut show 1M from the **first** turn instead of only after a session crosses 200k — `settings.json` is usually empty because the model is picked via `/model`, which doesn't write it there.

**Subagents (Task tool).** Claude Code writes each spawned subagent to its own jsonl in a sibling dir of the parent session: `<project>/<session_id>/subagents/agent-<id>.jsonl` (same `sessionId`, every assistant line `isSidechain:true`, plus `attributionAgent` = the agent type, e.g. "Explore"). The aggregate `usage_totals` walk (`WalkDir`, recursive) already counts these — blended into the parent session. For **per-agent** visibility, `usage_for_session` additionally scans that subagents dir (`subagents_for_session_file`) and attaches a `SubagentUsage[]` to the returned `SessionUsage`. Each is parsed with `parse_file(.., force_context=true)` — the `isSidechain` context gate in `process_line` would otherwise leave `context_tokens` at 0 since every line is sidechain; forcing it open tracks the subagent's **own** context window. `running` is a heuristic (file modified within ~8s). The frontend shows them as a thin strip of mini context-bar chips under the pane header (`SubagentStrip`/`SubagentChip` in `AgentPane.tsx`) — kept separate from the parent's `ContextGauge`, since subagents have their own windows — and a Subagents section in the stats popover.

Claude subscription limits (5h/weekly windows, shown in the **Deck**) come from the Anthropic OAuth usage endpoint, authenticated with the local Claude Code login (macOS Keychain `Claude Code-credentials`, fallback `~/.claude/.credentials.json`) — implemented in `src-tauri/src/limits.rs`, polled once a minute by `src/lib/limits.ts` (paused while hidden). A `null` result means "no usable Claude login" (missing credentials, or an expired/revoked token — 401/403) and hides the Claude meters; transient fetch errors (network, 5xx) reject instead, and the store keeps the last known values — preserve this distinction. Codex limit windows are **account-level**: `codex_account_limits` (in `codex_usage.rs`) finds the newest `token_count.rate_limits` event across ALL of `~/.codex/sessions/**/*.jsonl` — files are iterated newest-mtime-first, each only via a bounded 256 KiB tail read, stopping at the first hit (worst case: one stat pass + N tail reads; steady state: one) — and is polled on the same 60s cadence into the limits store (`useLimits.codex`). Per-session `codex_limits` parsing stays intact for the pane stats popovers. Tokens/cost are tracked but only shown on demand (per-pane stats popover — also in the ⋯ menu's Stats submenu for narrow panes — and the usage drawer) — not as raw token totals in pane headers or the title bar.

## Deck & fleet events

The **Deck** (`components/Deck.tsx`) is the permanent ~30 px status bar under the workspace grid — mounted in `App.tsx` inside a flex-col workspace column (grid area `flex-1` + Deck below), so the fleet overlay and floating terminals keep covering exactly the grid and the orchestrator panel stays a full-height sibling to the right. Title bar = place/navigation, Deck = system status (see DESIGN.md). Anatomy left→right:

1. **Triage queue**: `⚑ N need you` in `--attn` — hidden entirely at N=0. Click (or focus + Enter) opens a Radix popover (`ui/popover.tsx`, `--popover` surface; Radix gives the content `role="dialog"`, so the App.tsx dialog-shortcut-guard applies) listing every needs-you pane app-wide, oldest waiting first — waiting-since comes from `Agent.lastBusyEndAt` (stamped exactly at busy→waiting), and bell-attention-only panes carry `Agent.waitingSince` (stamped in `setAttention(true)` when the activity isn't "waiting", cleared with the attention, in-memory only) so they sort by age too instead of always last. Arrow keys rove the rows, Enter jumps (`focusAgent` + `focusTerm`, the palette pattern); the footer shows the ⌘⇧A hint.
2. **Event ticker**: the last 3 fleet events inline (newest last, mono 10px, absolute HH:MM), each clickable to jump while its pane still exists; a small history icon opens a popover with the last ~30.
3. **Meters**: the Claude subscription meters (moved out of the title bar, same `LimitMeter` rendering) + the account-level Codex meters. The terse labels (`cl 5h`, `cx wk`, …) get full-word tooltips ("Claude · 5-hour window · 34% used · resets 14:00"); the trigger spans are focusable (`tabIndex`) so the tooltip has a keyboard path, and bar widths animate over 200 ms. Codex is never silently absent: no data ever → a `CX —` placeholder with a "updates with Codex activity" tooltip; data older than 1h gets an "as of HH:MM" tooltip suffix (`as_of_ms` from the source event, fallback file mtime).
4. **Orchestrator dot**: `orch` + dot — `--attn` when any chat has undelivered pings, `--ring` while a turn runs, faint otherwise; click toggles the panel.

**Fleet event feed** (`src/lib/events.ts`, a standalone zustand store like `lib/limits.ts`): `FleetEvent { id, at, kind, paneId, paneName, workspaceId }` with kinds `finished` / `waiting` / `orch_prompt` / `created` / `exited`, capped at 50, **in-memory only by design** (a glanceable trail, not an audit log). Producers call `pushFleetEvent` from the sites that already know: `setActivity`/`setUsage` in store.ts (busy→idle = `finished`; entering "waiting" = `waiting` — busy→waiting emits only `waiting`), `setAttention(true)` (bell attention = `waiting` too), `createAgent` and `applyPreset` (`created` — the preset path builds its agents in one pass, bypassing `createAgent`), `setStatus`→exited / `removeAgent` of a still-live pane (`exited`; closing a whole workspace is a deliberate bulk teardown and stays silent), and `notePromptDelivered` in `orchestrator/executors.ts` (`orch_prompt` for prompt_pane + create_panes startup prompts). Repeated `waiting` for the same pane within 3 s is deduped in the feed itself (aligned with the Phase-5 ping flap guard). Consumers: the Deck's ticker/history and the fleet ops board's per-pane event line (`lastEventForPane`, a pure last-match selector). The triage *ordering* (needs-you panes, oldest waiting first) lives in `src/lib/triage.ts` — shared by the Deck queue and the fleet board's initial selection / Tab cycle.

## Git status

Per-pane git status (repo, branch, ±lines, untracked, `origin` URL for "Open repo in browser") is strictly read-only and shells out to the git binary: `src-tauri/src/git.rs` (prefers the Settings `gitPath` override, else `/usr/bin/git` because GUI apps get a minimal PATH; every subprocess runs through `output_with_timeout` — a repo on a dead network volume must not wedge the poll loop forever). `src/lib/git.ts` polls every 7 s, one fetch per distinct cwd, paused while the window is hidden; `null` on an agent means "checked, not a repo".

## Git worktrees

`src-tauri/src/worktree.rs` + `src/lib/worktree.ts`, native-only direct `invoke` like openrouter.ts — deliberately *writing* git, unlike the read-only git.rs. The New Agent dialog shows a worktree toggle (default off) whenever the chosen cwd is in a git repo; enabling it reveals a branch field (random `repo/prefix-suffix-1234` name from `generateBranchName`, dice = reroll, manual input wins) and a "Copy environment" switch (default on). Submit is gated while the repo check is in flight (it must never silently downgrade to a plain pane), creates `<repo>/.worktrees/<slug>` via `git worktree add -b` off the current HEAD (main root resolved via `--git-common-dir`, so splitting from inside a worktree still lands in the main repo), excludes `.worktrees/` via `.git/info/exclude` (never the tracked `.gitignore`), copies every untracked file (`ls-files --others`, incl. gitignored `.env`s) minus the heavyweight `SKIP_DIRS` (node_modules, target, dist, …), then spawns the agent with cwd = worktree path and `Agent.worktree = {root, branch}` (persisted in the grid snapshot → restored panes keep their badge/lifecycle; tab auto-name/defaultCwd/lastCwd use the *root*, never the `.worktrees` path). Cancelling the dialog mid-create rolls the fresh worktree back instead of spawning an unwanted pane. Splits from a worktree pane prefill the toggle + a fresh branch on the root repo.

**Close lifecycle** (`requestRemoveAgent`): a pane with an agent actively working first raises `CloseBusyDialog.tsx` (Claude activity from OSC 9;4, Codex activity from rollout events; misclick guard mirrors the quit guard's rationale); then worktree panes check `worktree_status` — clean (no uncommitted changes, `ahead == 0` where ahead = commits no other local branch/remote can reach) → silently scheduled for deletion (folder `worktree remove --force` + `branch -D`, executed in `removeAgent` via the module-level `pendingWorktreeCleanup` map; a cancel **or detach** in the floating-terminal dialog aborts it — a detached shell keeps the worktree as its cwd); dirty/local-only → `CloseWorktreeDialog.tsx` (keep / delete / cancel; keep = pane closes, worktree stays). Cleanup entries carry a `confirmed` flag: a silent clean verdict is **re-checked in `removeAgent` right before execution** (the decision can be minutes old after dialogs sat open — on doubt the worktree is kept and shows as orphan), only an explicit dialog "delete" runs unconditionally. Gitignored files (e.g. the copied `.env`s) never count as dirty — by design. Closing a whole workspace keeps worktrees by default, with a separate "Close & clean safe" action that kills the workspace's panes and then removes only worktrees that still re-check clean/no-local-only-commits. Removing an agent clears any confirm dialog still pointing at it.

**Management panel** (`WorktreePanel.tsx`): a title-bar `FolderGit2` button, visible only while ≥1 worktree exists, lists all SwarmZ worktrees grouped per repo (open/orphaned/missing + dirty/ahead state, refreshed on open) with open-in-pane (or jump-to-pane), reveal-in-Finder (`opener:allow-reveal-item-in-dir` capability), per-row finish/delete and a "Clean up safe" bulk action for clean unattached worktrees. Risky rows (uncommitted changes or local-only commits) still require a second delete click; deletion is disabled while a pane is open on that worktree. Scanned repo roots persist in `settings.worktreeRepos` (registered on create, pruned automatically once a repo has no worktrees left — that's what makes the icon disappear again; `worktree_list` returns a `WorktreeScan {entries, scanned}` and only roots that actually **scanned** empty are pruned, against the current registry — an unmounted volume or broken git override never loses orphans). A hand-deleted worktree folder still reports its branch's unreachable commits as `ahead` (computed from the main repo), so the panel's missing rows stay behind the two-step confirm when deleting would lose work.

## Panes, focus mode, floating terminals

Pane headers are container-query responsive (`@container` on the header in `AgentPane.tsx`): below ~36rem the folder path, gauge readout and split buttons collapse, below ~32rem the model badge, below ~28rem also the git chip, stats, mic and focus buttons and the status badge's word (its glyph/dot stays — a status is visible at every tier) — only status glyph, truncated title (`min-w-0 truncate`, never `shrink-0` — it must shrink for the ellipsis to engage), context donut, ⋯ menu and close remain. Everything hidden stays reachable via the ⋯ menu (incl. the Stats submenu) or hotkeys (focus = header double-click, dictation = ⌘ hold/⌘⇧M).

Focus mode (`focusedAgentId` in the store, in-memory): the pane zooms (~150 ms) to an overlay above a dimmed backdrop in `TilingGrid.tsx` — nothing unmounts, all other PTYs keep running underneath. Enter via header button / header double-click / ⋯ menu; exit via the same toggle or a backdrop click (no Escape — the terminal eats it). Creating or removing agents, switching workspaces and opening the fleet all leave focus mode; pane drag-rearrange is disabled while focused. The exit animation relies on a render-phase `closingFocusId` in `TilingGrid.tsx` that keeps the transition class on the wrapper while the rect snaps back.

Floating terminals (`FloatingTerminals.tsx`): PiP-style plain shells above the grid, owned by an agent pane (same cwd), opened via header button / ⋯ menu / `⌘J`. In-memory, rendered in one layer inside `<main>`; windows stay mounted through minimize/detach and are only hidden (not unmounted) while their owner's workspace is inactive or the fleet is open — removal from the store is what kills the PTY (`destroyTerm`); detached windows (`agentId: null`) show in every workspace. Stacking via a per-window `z` (never reorder the DOM — xterm canvas). Windows name themselves after the last command run — preset clicks and typed lines alike (`onCommand` on `TerminalView`, a best-effort input-line buffer that skips history-recall/escape-edited lines). Their quick-command bar mixes **presets per project folder** (persisted, store key `commandPresets` → `FolderCommands { presets, hidden }`, keyed by cwd via `presetKey`; presets are editable in place, a preset matching a detected command's label or command **overrides** it, and detected commands can be hidden per folder + restored) with auto-detected commands: `project_commands` in `src-tauri/src/project.rs` (package.json scripts incl. lockfile→package-manager mapping, cargo, Makefile/justfile targets; script order is alphabetical; script names are sanitized to a safe character set — a name with control characters must never become a button that writes hidden shell lines). Closing a pane goes through `requestRemoveAgent`, which asks the backend whether a floating terminal still has child processes (`pty_has_children`, pgrep on the shell pid) and only then raises the kill-vs-detach dialog (`CloseAgentDialog.tsx`); detached terminals live on with `agentId: null`.

## Status reporting & quit guard

Pane title & live status come from runtime-specific signals. `Terminal.tsx` handles OSC 0/2 titles (Claude's auto-generated topic) and OSC 9;4 progress reporting into `Agent.activity` busy/idle (the header status badge, the live per-tab `busy/total` counts, the fleet cards, and the quit guard). Rendering follows the DESIGN.md signal triad (`PaneStatus` in `AgentPane.tsx`): busy = quiet `▸ working` + a sweeping hairline under the header; waiting/bell attention = `⚑ needs you` in `--attn` with header tint, static pane outline and a one-shot `arrive-flash` on entry (keyed by a transition counter); busy→idle stamps `Agent.lastBusyEndAt` (in `setActivity` for Claude's OSC path and in `setUsage` for Codex rollout activity, in-memory only) and shows an ephemeral `✓ finished · Nm ago` for ~5 min, decayed by a 30s ticker scoped to the badge component. Claude Code only emits OSC 9;4 if the terminal advertises support — the PTY backend sets `ConEmuANSI=ON` for that (deliberately, least invasive marker; don't remove). OSC 21337 (Claude's idle/busy/waiting tab-status pill) is pre-wired but dormant — current Claude Code builds never emit it. Codex pane titles come from `~/.codex/session_index.jsonl` `thread_name`, and Codex busy/idle comes from `task_started` / `task_complete` rollout events parsed by `codex_usage.rs`.

Quit guard (`src/lib/quit.ts`, started in `App.tsx`): closing the app raises `QuitConfirmDialog.tsx` when quitting would lose something (`quitBlockerIds` + async `floatBlockerIds`) — agents with `activity === "busy"` always count (the run gets interrupted, spinner in the list); idle-but-open panes count only while `restoreAgents` is off (with restore on they come back anyway, so they don't block); `exited` panes never count; floating terminals count when `pty_has_children` says a process still runs in them (floats are never restored). Native covers both paths — window close via `onCloseRequested` (every first close is preventDefault'ed, all debounced persists are flushed (`flushAllPersists`), then `close()` is re-issued past a module-level `confirmed` flag so the window-state plugin still saves), and ⌘Q/menu quit via Rust (`lib.rs` run-callback: `ExitRequested` with `code.is_some()` → `prevent_exit()` + emit `app://quit-requested`; `prevent_exit` is a tauri-built-in no-op for the updater's `RESTART_EXIT_CODE`, and code `None` = last window closed passes, which is how a confirmed quit actually exits).

## OS file drag & drop

Tauri's default `dragDropEnabled` intercepts file drags at the NSView level, so the DOM never sees HTML5 drop events — `src/lib/dnd.ts` (started in `App.tsx`) subscribes to the webview's `onDragDropEvent`, hit-tests the cursor via `elementFromPoint` against `[data-file-drop]` zones (the terminal containers in `AgentPane.tsx` and `FloatingTerminals.tsx`, attribute value = pty id) and on drop **pastes** the Terminal.app-style backslash-escaped path(s) (`escapeDropPath` in `utils.ts`) + a trailing space into that terminal via `term.paste()` (xterm instances are looked up in `lib/term-registry.ts`, registered by `TerminalView`), then focuses/raises the target. Paste, not a raw PTY write, is essential: `term.paste()` wraps the text in bracketed-paste markers when the app enabled mode 2004, which agent CLIs handle more reliably for file paths and multiline prompts than raw keystroke writes. While a drag is in flight, every zone shows `FileDropOverlay.tsx` (store key `fileDrag`, in-memory; `pointer-events-none` so the overlay stays invisible to the hit test); the hovered zone lights up `--ring` blue. Beware: the drag events' `position` is typed `PhysicalPosition`, but on macOS the wry → tauri chain never scales NSView points, so the values are effectively logical CSS px — use them unscaled.

## Orchestrator sensing (Phase 1)

Read-only sensing foundation for the planned AI orchestration layer — no UI, no LLM calls, no terminal writes yet. `src-tauri/src/transcript.rs`: `transcript_read` returns actual message content from the same session jsonl files the usage parsers locate (Claude: encoded project dir + session id; Codex: rollout file found by the id suffix in the filename, head-scan fallback). Claude lines yield user/assistant text plus `type:"summary"` compaction entries; tool_use/tool_result blocks become one-line summaries (`[tool: Bash — git status]`, kind `"tool"`), sidechain/meta/command-wrapper noise (`<command-name>`, `<local-command-*>`, `Caveat:`) is skipped. Codex prefers the `event_msg` `user_message`/`agent_message` chat stream (all phases) and falls back to `response_item` messages (instruction wrappers filtered) for rollouts without event_msgs. Options: `tail_messages` (last N, default 20), `max_bytes` (default 1 MiB — huge files are seek-tailed to the first complete line, never fully read) and `include_first_user_message` (the Ursprungsprompt, from a separate capped head read). Texts are char-truncated (umlaut-safe) with an `…` marker. `project_docs` reads README.md/AGENTS.md/CLAUDE.md of a root (worktree paths collapse to the main repo; ~8 KB/file, ~20 KB total; missing files omitted). `src-tauri/src/projects.rs`: `discover_projects` merges project folders from Claude history (real cwd head-read from the newest jsonl per project dir — the encoded dir name is lossy), Codex `session_meta` cwds, frontend-known folders and a shallow scan (depth ≤ 2) for `.git` (dir or file); worktree paths collapse onto the repo root, entries dedupe by canonical path and sort by last activity. Both commands are async + `spawn_blocking`, uncached (stat/head reads only).

Frontend: `src/lib/orchestrator/` — `snapshot.ts` (`fleetSnapshot`/`fleetCounts`/`fleetSummaryLine`, pure functions over the store state: per-workspace pane list with runtime, project, branch, activity, model, context %; `fleetCounts` is also the census behind the fleet overview's header line; plus the layout-sensing helpers `workspaceLayout`/`crowdingNote` — see "Orchestrator layout awareness"), `native.ts` (direct-`invoke` wrappers like lib/worktree.ts; `discoverProjects` collects the known folders — workspace defaultCwds, profile/preset cwds, quick-note folder keys, `worktreeRepos`, `lastCwd` — from the live store), `types.ts` (payload + tool-bus types), and `dev.ts`, a dev-only smoke-test hook exposing `window.__orch` (loaded from `App.tsx` behind `import.meta.env.DEV`, tree-shaken in production).

## Orchestrator tool bus (Phase 2)

The tool layer the orchestrator brain calls — no chat UI here. **The registry is the single source of truth**: `src-tauri/src/orchestrator/registry.rs` defines every tool as `{ name, description, parameters (JSON Schema), timeout_ms }`; `tool_definitions()` serializes the catalog (11 tools) for the providers (Phase 3 hands it to Codex app-server `dynamicTools`, Phase 6 to the OpenRouter tool loop). The `orchestrator_tools(persona)` command returns `{ instructions, tools }` — the tools plus the compiled system instructions (`persona::build_instructions`, the single source both provider paths share — see "Orchestrator persona & memory"), which the OpenRouter loop needs for its wire system message. The frontend mirrors the *names* only (`ORCHESTRATOR_TOOL_NAMES` in `types.ts`, executor keys) — schemas are never duplicated in TS. Argument validation is a deliberately hand-rolled subset (required props, basic `type` tags, `enum`, array `minItems`/`maxItems`/`items`, applied recursively) — no jsonschema dependency; unknown schema keywords are accepted. The one write-to-disk tool is **`remember`** (executor → `orchestrator_memory_append`); it appears as an ordinary tool chip in the chat like every other call.

**Roundtrip bus** (`orchestrator/bus.rs` + `src/lib/orchestrator/bus.ts` + `executors.ts`): tools are defined in Rust but *execute in the webview*, where the Zustand store lives. `run_tool` validates against the registry, parks a tokio oneshot in a pending map under a generated request id, emits `orchestrator://tool-request` `{ id, tool, args, chat_id }` (`chat_id` = the triggering backend chat, null for dev/internal calls — Phase 5), and awaits the response with the tool's timeout (15 s default; `create_panes` 120 s — worktree creation copies env files and startup prompts wait for CLI boot). The webview bus (`startOrchestratorBus()`, registered once from `App.tsx` like dnd/quit, double-start-guarded) looks up the executor, runs it and **always** answers via the sync `orchestrator_tool_response(id, ok, payload)` command (errors → `ok:false` with the message string); unknown/expired ids there are a logged no-op — late responses after a timeout are normal. On timeout the pending entry is removed and a clear error returned. The bus core (`PendingMap`, `run_tool_via` with an injected emit fn) is tauri-free and unit-tested. `orchestrator_run_tool(tool, args)` (async command) is the Phase-2 acceptance surface — the dev hook drives the full loop via `await __orch.tool("fleet_snapshot", {})`, `__orch.tools()` returns the catalog.

**The 10 V1 tools** (executors in `src/lib/orchestrator/executors.ts`; `pane_id`s resolve against `useSwarm.getState().agents`, unknown ids error with the list of valid ones): `fleet_snapshot` (Phase-1 snapshot + summary line) · `read_transcript` (pane meta + Phase-1 transcript tail; shell panes / undiscovered sessions error with the reason) · `read_project_docs` (exactly one of `pane_id` — a pane's worktree root ?? cwd, OR a native Vibe **session** id whose `projectDir` is read, resolved via `requireTarget` like `prompt_pane`/`read_transcript` — or `path`) · `read_notes` (quick notes as `{text, done}`, ids dropped) · `git_status` (the pane's cached `agent.git` + derived `dirty`, or `git:null` with a note) · `list_projects` (Phase-1 discovery; `scan_roots` omitted = the user's `orchestratorScanRoots` default folders, Phase 6) · `list_blueprints` (compact profiles + workspace presets with per-pane templates, plus per-runtime default startup + recently used model ids from open panes and the usage history — the ids `create_panes` accepts as `model`) · `prompt_pane` (write) · `create_panes` (write) · `create_workspace` (write, store's `createWorkspace`).

Write-tool invariants: **`prompt_pane` delivers via `insertCommandText`** (`term.paste()` bracketed paste + a *separate* `\r` on submit — never raw `ptyWrite` for the text, same invariant as drops/dictation); exited panes are refused, busy panes still receive the text but the response carries a `warning` (queued in the CLI's input) — unless the Phase-6 busy policy refuses (see "Orchestrator brain: OpenRouter loop + settings"); Phase 5 adds a double-prompt guard (see "Orchestrator status pings"), Phase 6 the review-mode submit gate. **`create_panes` never silently downgrades**: `worktree:true` on a folder that isn't a git repo is a per-pane *error* (repo check via `fetchGitInfo`, branch generated with `generateBranchName` when omitted, then the same native `addWorktree` flow as `NewAgentDialog` — `.worktrees/<slug>`, env copy — and only then `createAgent` with the worktree meta). An optional per-pane `model` (+ codex-only `reasoning`) appends `--model`/`-m`/`model_reasoning_effort` to the effective startup — strictly character-validated (the startup runs in a shell) and refused for shell panes; omitted = the user's default configuration. Panes are created sequentially (parallel `git worktree add` races on git's lock files); per-pane errors don't abort the batch. Placement (which workspace, arrangement, overflow, `beside`) is planned and balanced — see "Orchestrator layout awareness"; the target workspace is activated first (`createAgent` inserts into the active workspace, then the batch's layout is rebuilt equal-sized). Startup `prompt`s are delivered after spawn by `deliverStartupPrompt`: poll for `status === "running"` (≤ 20 s), then a fixed settle delay (300 ms shell, 2 s agent CLIs — there is no readiness signal from the CLIs today, and pasting too early would feed the login shell instead), then `insertCommandText(…, submit=true)`; failures surface as per-pane `warning`s.

## Codex app-server host (`src-tauri/src/codex/`)

The generic layer under every `codex app-server` consumer — the orchestrator brain today, the Vibe-Mode native Codex sessions next. Spawn, framing and respawn code exists exactly ONCE, here; consumers never touch the process directly.

**Wire protocol** (`codex/protocol.rs`, pure + unit-tested): newline-delimited JSON **without** the `"jsonrpc":"2.0"` header (verified against codex 0.142.5 — the server neither sends nor expects it). Our requests use an own ascending numeric id; server-initiated request ids are kept as raw JSON and echoed back exactly (`RequestId` is `number | string` on the wire). `parse_line` classifies each stdout line into response / server request / notification, defensively (garbage → None). The unit tests freeze the live-captured 0.142.5 shapes the session layer consumes: `item/started`/`item/completed` with `commandExecution` (real shell line, `commandActions`, `aggregatedOutput`+`exitCode` only on completion) and `fileChange` items (`changes[].kind` is a tagged object; for `add` the `diff` is the raw new content — the unified diff arrives separately as `turn/diff/updated`), `item/agentMessage/delta`, `turn/diff/updated`, `turn/plan/updated` (source-derived — never observed live on 0.142.5), `thread/tokenUsage/updated` (total = thread-cumulative, last = last turn, plus `modelContextWindow`), `account/rateLimits/updated` (account-scoped — carries NO threadId) and `item/commandExecution/requestApproval` incl. the live-only `availableDecisions` extra (absent from the generated schema — never rely on it), plus the four plain response decisions `accept`/`acceptForSession`/`decline`/`cancel`.

**Client & binary resolution** (`codex/host.rs`): `Client::spawn` starts `<program> app-server` via `tokio::process` with one writer task (ordered stdin), a stderr logger and a reader task that resolves responses via a numeric-id pending map and forwards server requests/notifications. Reader EOF = process death: the `alive` flag drops and all pending RPCs fail with a clear error. The binary is resolved to an ABSOLUTE path (`resolve_codex_program`: Settings override → current $PATH → well-known install dirs (~/.local/bin, /opt/homebrew/bin, …) → `zsh -lc 'command -v codex'` login-shell probe) because macOS gives the built GUI app a minimal PATH — a bare `codex` works in dev but fails packaged; the child's PATH is additionally enriched with the binary's dir so codex-spawned helpers resolve too. Resolution is cached process-wide (one cache for all consumers — `set_codex_override` installs the Settings value and drops the cache, as does a spawn failure). The ignored spike tests run the resolution path — re-run with `PATH=/usr/bin:/bin` to regression-test the packaged-app environment.

**Thread registry & event routing**: a `Connection` (Client + `initialize` handshake + router task) holds a route table `threadId → EventSink` (an mpsc sender of `ThreadEvent`s; several threads may share one sink — the orchestrator's single dispatcher does). The router forwards each server event to the sink registered for its `params.threadId`; routed server requests carry a `Responder` (owning client + echoed id), so consumers answer without tracking which process generation a request came from. No sink responsible → notifications are ignored (account-level notifications, foreign threads) and server requests are refused with -32601 (the server treats that as a denial). Process death delivers `ThreadEvent::Exited` once per DISTINCT sink (not once per registered thread) and drains the routes — routes die with the process and must be re-registered after a respawn.

**Process strategies & respawn** (t3code-inspired): a `ProcessHost` is one lazily (re)spawned connection slot with a generation counter bumped per successful spawn+handshake. The two strategies are just slot allocation: **(a) shared process** — the orchestrator keeps ONE static slot and multiplexes all chat threads over it; **(b) dedicated process per thread** — each Vibe session allocates a private slot, so a crash kills only that session (live-verified: two dedicated processes ran turns genuinely in parallel). A consumer that sees a different generation than it stored for its thread transparently `thread/resume`s (and re-registers its route) before the next turn. `resume_thread` classifies failures typed: `ResumeError::ThreadNotFound` — on 0.142.5 the live answer is `no rollout found for thread id … (code -32600)`, matched together with t3code's not-found variants — is the consumer's cue to fall back to a fresh `thread/start` (rollout deleted / never persisted, e.g. ephemeral threads); everything else is `Other`.

**Tests**: framing + frozen fixtures in `codex/protocol.rs`; pending map, resolution, routing/exit-dedupe and resume classification in `codex/host.rs`. The `#[ignore]`d `session_spike` is the live proof against the real codex CLI (two dedicated processes in parallel under `sandbox:"workspace-write"` + `approvalPolicy:"untrusted"`, real fileChange **and** commandExecution approval accept-roundtrips verified by item completion + on-disk side effects, `turn/diff/updated` and token usage observed, bogus-resume classification checked live): `cd src-tauri && cargo test session_spike -- --ignored --nocapture`. Live findings from that spike: under `untrusted`, apply_patch triggers `item/fileChange/requestApproval` (reason-less, no `availableDecisions`) even under workspace-write, and read-only commands like `cat` count as trusted (run approval-free) while `touch` forces the command approval.

## Vibe-Mode native Codex sessions (`src-tauri/src/codex/sessions.rs`, Phase 2)

The SECOND consumer of the codex host (the orchestrator brain is the first): plain agentic Codex sessions driven natively and mirrored into the SwarmZ UI. Process strategy **(b)** — each session owns a PRIVATE `ProcessHost` slot, so a crash isolates to that one session. Unlike the orchestrator, a Vibe session keeps codex' STANDARD harness intact — **NO `dynamicTools`, NO `developerInstructions`** (a normal exec + apply_patch session, just wire-driven). The frontend data layer lives in `src/lib/vibe/` (see "Vibe-Mode data layer").

**Registry & profile**: a static `SESSIONS` map (`session_id → SessionState`) holds each session's private `ProcessHost`, `thread_id`, spawn `generation` (respawn detection → transparent resume, same mechanism as the orchestrator), profile (`cwd`, optional `model`/`effort`, `access`), `current_turn_id`, a `busy` flag (one turn per session — claimed synchronously in `send`), the unanswered approval `Responder`s, and the session's `EventSink`. `session_id` is assigned by the frontend (it keys the store's `VibeSession`); it is passed to `vibe_session_start`/`_resume` — not minted in Rust.

**Access → sandbox** (exact wire strings verified against the 0.142.5 reference): `workspace` = sandbox `workspace-write` + approvalPolicy `on-request`; `full` = sandbox `danger-full-access` + approvalPolicy `never` (the vibe default — get out of the agent's way). **`model` and `effort` are per-turn overrides** (both ride on every `turn/start`), not thread/start-only fields — that is what lets the user change model/effort mid-session (`vibe_session_set_model_effort(session_id, model?, effort?)` mutates the profile; the change applies on the next turn, no fresh thread). An **access change** takes effect on the NEXT turn via a per-turn override (`sandboxPolicy` **object** form + `approvalPolicy`) — attached only when access changed since the last turn (a `access_override_pending` flag), so the object-form sandbox stays off ordinary turns.

**Commands** (all async, in lib.rs): `vibe_session_start(session_id, cwd, model?, effort?, access, codex_path?)` → `{thread_id}`; `vibe_session_resume(session_id, thread_id, …)` → `{thread_id, resumed}` (a `ThreadNotFound` rollout-gone falls back to a fresh `thread/start`, `resumed:false` tells the UI its prior context is gone); `vibe_session_send(session_id, text)` → `{turn_id}` **non-blocking** (returns after the `turn/start` ack — the transcript + completion arrive as events; a busy session errors); `vibe_session_interrupt`; `vibe_session_respond_approval(session_id, approval_id, decision)` with decision ∈ `accept|acceptForSession|decline|cancel` (resolves the remembered `Responder`); `vibe_session_set_access`; `vibe_session_set_model_effort` (updates the profile — applies next turn); `vibe_session_close` (best-effort interrupt, answer every pending approval with `cancel`, unregister, drop the entry → the private process ends via stdin-close/`kill_on_drop` and its dispatcher task ends once the sink is gone).

**Event mapping → `vibe://session-event` `{session_id, kind, data}`** (a per-session dispatcher consumes the routed `ThreadEvent`s; the pure `map_notification` is unit-tested against fixture lines): `turn_started` · `delta` `{item_id, text}` (agentMessage) · `message` `{item_id, text, phase}` (agentMessage item/completed — routed here, NOT to `item_completed`) · `item_started`/`item_updated`/`item_completed` `{item}` with the item normalized (commandExecution `aggregatedOutput` capped to 64 KB on the tail; fileChange/webSearch/plan pass through; unknown item types pass through; **userMessage + reasoning are dropped** — the user bubble is added optimistically frontend-side, reasoning text never surfaces over ChatGPT auth) · `item_output_delta` `{item_id, delta}` (commandExecution output streaming — **live-confirmed** in the spike) · `turn_diff` `{diff}` · `plan` `{explanation, plan}` · `token_usage` `{total, last, modelContextWindow}` · `approval_request` `{approval_id, kind, request}` (the request params — itemId, reason, command, … — pass through verbatim; the UI reads the diff via the fileChange item's itemId) · `turn_completed` `{status}` · `turn_failed` `{error}` · `warning` · `process_exited`. Any non-approval server-initiated request is refused with -32601 (treated as a denial).

**Tests**: pure mapping + access strings + output cap in `sessions.rs`. The `#[ignore]`d `sessions_spike` is the live proof against the real codex CLI, answering the three open Phase-1 questions: `cd src-tauri && cargo test sessions_spike -- --ignored --nocapture`. **Live findings**: (a) `turn/interrupt` on a running command → `turn/completed` status **`interrupted`** (confirmed); (b) an `on-request` command approval fires only when a command **escalates past the sandbox** (an in-workspace `touch` runs approval-free — unlike Phase-1's `untrusted` probe; writing outside the workspace forces it), and a **`decline`** marks the command item `status:"declined"` (exec fails "rejected by user") while the **turn continues to `completed`** — decline does not abort the turn; (c) `item/commandExecution/outputDelta` **does** stream while a command runs (`item/updated` never fired) — hence the `item_output_delta` event.

## Orchestrator brain: Codex app-server (Phase 3)

The LLM behind the orchestrator — still no chat UI (Phase 4); acceptance runs through the dev hook (`__orch.chatStart()` → `__orch.chatSend(chatId, "Welche Panes sind offen?")` streams `[orch chat-1] tool_call {tool:"fleet_snapshot"…}` to the console and resolves with the final answer).

**Process lifecycle** (`src-tauri/src/orchestrator/appserver.rs`, a thin consumer of `src-tauri/src/codex/` — see "Codex app-server host"): ONE shared `ProcessHost` slot multiplexes all orchestrator chats over a single long-lived `codex app-server` child, spawned lazily on first use — the Settings `codexPath` override is passed by the frontend on every `chat_start`/`chat_status` call (empty string clears it) and installed via `host::set_codex_override`. Every chat thread registers the dispatcher's one event sink with the host, which routes that thread's server requests/notifications back per `threadId`. On process death (host delivers `Exited`), running turns resolve as failed and every chat gets a `warning` event — the next call respawns. After a respawn, chats transparently `thread/resume` their thread (and re-register their route) before the next turn — a per-chat `generation` counter compared against the host's spawn generation detects the restart; rollouts persist on disk and codex restores the declared dynamic tools from them (verified). A `ThreadNotFound` resume failure is NOT retried as a fresh thread here — it surfaces as an error and the frontend controller (Phase 4) handles it by starting a new thread.

**Chat ↔ thread mapping**: `orchestrator_chat_start` runs the `initialize` handshake (`capabilities.experimentalApi: true` — required for dynamic tools; the response's `userAgent` carries the codex version) and `thread/start` with `cwd` = home (neutral), `sandbox: "read-only"`, `approvalPolicy: "never"`, the whole registry adapted to the experimental `dynamicTools` field (`{type:"function", name, description, inputSchema}` — our `parameters` schema passes through as `inputSchema`), and the compiled orchestrator system instructions (persona + memory + operative core, see "Orchestrator persona & memory") as `developerInstructions` (keeps codex's base prompt — and with it the tool harness — intact, unlike `baseInstructions`). The persona (voice, from `settings.orchestratorPersona`) is passed by the frontend on `orchestrator_chat_start`/`orchestrator_chat_resume` (like `codexPath`) and captured on the chat; memory is read fresh from disk each start/resume. Returns `{chat_id, thread_id}`; `orchestrator_chat_resume(thread_id, persona)` reopens a persisted thread (dynamicTools are **not** re-declarable on resume — codex restores them from the rollout). `orchestrator_chat_send(chat_id, text, model?, effort?)` prepends a fresh fleet-summary line (`[fleet status: …]`, best-effort via `run_tool("fleet_snapshot")`) to the user text, starts the turn and resolves with `{status, text}` when `turn/completed` arrives (30-min cap; status "failed" rejects with the turn error). The optional `model`/`effort` are per-turn `turn/start` overrides (the per-chat picker) — omitted = the user's default. `thread/tokenUsage/updated` is re-emitted as a `token_usage` chat event (`{total, last, modelContextWindow}`) for the chat's context gauge (in-memory in the chat-store, never persisted). `orchestrator_chat_interrupt` needs the current turn id — tracked from `turn/started`. `orchestrator_chat_status` reports `{running, version, account}` (`account/read` summary: logged in, plan, email) and never rejects for a dead process (`running:false` + error).

**Dynamic-tools bridge** (`orchestrator/adapter.rs` holds the shape adapters): `item/tool/call` server requests are answered from the Phase-2 bus — `orchestrator::run_tool(&app, tool, args)` (webview roundtrip) with args normalized (null → `{}`, stringified JSON unwrapped), the result adapted to `{success, contentItems:[{type:"inputText", text}]}` (plain-string results pass through unquoted, other payloads as compact JSON; errors come back `success:false` with an `ERROR:` text so the model can react). Each call runs in its own task so deltas keep streaming during tool roundtrips.

**Event stream**: everything observable emits `orchestrator://chat-event` `{chat_id, kind, data}` with kinds `turn_started`, `delta` (agentMessage delta text), `message` (completed assistant message), `tool_call` `{tool, args_summary}` (args capped to one line), `tool_done` `{tool, ok}`, `turn_completed` `{status}`, `turn_failed` `{error}`, `warning`. Typed wrappers + `onChatEvent` live in `src/lib/orchestrator/chat.ts`; unknown notifications are ignored silently (defensive parser).

**Approvals & safety**: under read-only sandbox + `approvalPolicy: "never"` + the instructions, approval requests must never occur — if one arrives anyway, `item/commandExecution/requestApproval` / `item/fileChange/requestApproval` are auto-DECLINED (`{decision:"decline"}`) and any other server-initiated request (user-input prompts, elicitations, permission grants, legacy v1 approvals) is refused with a JSON-RPC error; both emit a `warning` event. Known gap: MCP servers from the user's codex config (e.g. the Codex.app `computer-use`/`node_repl` bundles) still attach to orchestrator threads — config overrides (`-c mcp_servers={}`, feature/plugin disables) proved ineffective in the spike; the system instructions forbid using non-SwarmZ tools, but that is instruction-level, not enforced.

**Version guard**: dynamic tools are experimental protocol surface — `thread/start` failures that smell like a missing `dynamicTools` field are wrapped in an actionable error naming the installed version (from `initialize`) and the verified-good codex 0.142.5.

**Tests**: `codex/protocol.rs` covers framing (real fixture lines) and `codex/host.rs` the id map, resolution and routing (see the host section); `orchestrator/adapter.rs` covers the dynamic-tool/tool-response adapters and arg normalization; `appserver.rs` covers the version guard and thread params. The full ping-tool loop against the real installed codex is the `#[ignore]`d `appserver_spike` test (needs codex + login + network; CI stays green): `cd src-tauri && cargo test appserver_spike -- --ignored --nocapture`.

The user-facing chat over the Phase-3 brain: `OrchestratorPanel.tsx`, toggled via ⌘⇧O or the title-bar bot button. Grid-mode only — in Vibe Mode the same entry points route to the Conductor stage instead (see "Conductor in the stage" below). **Layout**: the panel is a flex *sibling* of the workspace column — `App.tsx` wraps `WorkspaceLayer` + `FloatingTerminals` in a `relative flex-1 min-w-0` grid area (stacked above the Deck inside a flex-col column) with the panel to the column's right, so opening it genuinely squeezes the grid (PTYs resize through the normal ResizeObserver path; that's expected here, unlike the fleet view) while the fleet overlay and floating terminals keep covering exactly the grid area, never the panel or the Deck. Its textarea is a real text field, so plain-⌘ hold-dictation never arms while typing (the existing `editing` guard covers it); Escape in the textarea only blurs, never closes the panel; opening the panel focuses the textarea. Assistant messages render through `OrchestratorMarkdown.tsx` — a tiny dependency-free markdown subset (headings, lists, fenced/inline code, bold/italic, links via `openUrl`) built as React nodes, never innerHTML, so model output can't inject markup; unclosed markers in a still-streaming message render literally until the closing half arrives. User bubbles stay raw text on purpose.

**Chat presentation** (redesign, `components/orchestrator/ChatView.tsx` — purely a view over the chat-store, the persisted `role:"tool"` message format is untouched): a t3code-style human-readable chat. The message column is capped to a reading width (`CHAT_MAX_W ≈ 46rem`, centered) with the composer flush beneath it (shared by the ConductorStage, the panel and — same cap — the Vibe `ItemFeed`); body text is ~13.5px sans with generous leading, tool/meta lines stay small mono and recede (faint). Consecutive tool calls fold into one **activity block** (`groupChatMessages` + `toolActivityLabel` in `lib/orchestrator/tool-labels.ts`, pure + unit-tested). **One quiet-line system for everything non-prose.** A `QuietLine` primitive (status glyph + human text + optional tooltip + trailing chips) renders EVERY tool step, status ping and warning identically, so the feed reads as one homogeneous stream regardless of what the tool/ping is now or in the future. Its `StatusIcon` follows the signal triad — `✓` `--success` (done / a pane finished), `⚠` `--warning` (a failed step / warning), `⚑` `--attn` (a pane waiting for input), a neutral `·`/`…` while running or for an unclassifiable ping. Rendering split (both pure + unit-tested): a **lone** tool step (`isSingleStepActivity`) renders as one `QuietLine` (status glyph + human verb, raw tool name + args in the tooltip) — never the bulky "Worked · 1 step" disclosure; **multiple** steps still fold into the collapsible group ("Working · N steps" header, live-growing ✓/⚠, expands to a per-step list with human verbs — never raw tool names, those live in the step tooltip; pane/session **jump chips stay first-class** as a union row even while collapsed, and failed steps stay visible collapsed). Phase-5 **status pings** (`role:"system"`) render through the same `QuietLine`: `systemPingKind(text)` reads finished/waiting/info purely from the controller-owned text (the message format is never rewritten) → `✓`/`⚑`/`·` plus the jump chip and "Review" button. **Guarantee:** an unknown/future tool name still yields a clean verb via `toolActivityLabel`'s `"Used a tool"` fallback (raw name in the tooltip), and an unrecognised ping falls back to the neutral `info` marker — no raw tool name or iconless line ever reaches the feed. Grouping still breaks at every non-tool message (chronologically honest — pings/warnings are not reordered into tool runs); the shared `QuietLine` aesthetic is what makes the flow homogeneous, not any timeline reshuffle. **File-path pills**: `lib/paths.ts` (`splitTextWithPaths`, pure + unit-tested) detects `~/…`, absolute AND bare relative paths (strict shape: dot-free dir segments + a file extension on the last one — rejects domains and `@scope/pkg`) in prose text nodes **after** markdown parsing (inside `OrchestratorMarkdown`, so it covers both the orchestrator chat and the Vibe feed) and renders them as a compact pill showing ONLY the filename plus an optional `:line` — the full path lives exclusively in the tooltip; markdown links whose href is a path (`isPathHref`) render as the same pill instead of a link. Code spans and fenced blocks are never touched. The shared `ChatMeta` (model/effort chip + context gauge) sits in both the ConductorStage and panel headers; the chip opens the shared `ModelEffortPicker` (per-chat model/effort override, "applies from the next turn"). The picker's "Available" section is the authoritative catalog from the app-server's `model/list` (`codex_list_models` command → `list_models` in appserver.rs, hidden entries dropped, server order = default first; live-verified on 0.142.5), fetched once per app run (`useCodexModels`) with recently-used + free text as the always-present fallback.

**Store slice** (`src/lib/orchestrator/chat-store.ts`, a standalone zustand store like `lib/updates.ts`/`lib/limits.ts` — kept out of the already-large store.ts): `chats: OrchestratorChat[]` (`{id, provider, threadId, model, wire, title, createdAt, messages}`, newest first — `provider`/`model`/`wire` are the Phase-6 fields), `activeChatId`, `panelOpen`, `panelWidth`, plus in-memory `busy` (per chat) and the codex `status`. Message roles: `user` / `assistant` (transient `streaming` flag while deltas arrive) / `tool` (`{tool, argsSummary, ok?, paneRefs?}`) / `warning` / `system` (the Phase-5 status pings, with optional `paneRefs`). Persisted under store key `orchestratorChats` (chats + active id + panel open/width in one blob), debounced ~800 ms like the grid snapshot, **flushed by `flushAllPersists` at quit**, hydrated from store.ts `hydrate()` with per-entry graceful degradation (a malformed chat or message only costs itself; `streaming` flags are never restored). **Caps**: ~200 messages per chat (oldest drop first) and ~30 chats total (oldest chats deleted first). `deleteChat` only removes the chat from SwarmZ — the codex thread rollout stays on disk by design (it simply never gets resumed again).

**Controller** (`src/lib/orchestrator/controller.ts`, outside React like term-host/dictation): owns the store-chat ↔ backend-`chat_id` mapping and routes `onChatEvent`. `sendMessage` appends the user message, auto-titles the chat from the first user message (~40 chars, once), lazily `chatStart`s the backend chat — or `chatResume`s the persisted `threadId` after an app restart; a failed resume (rollout deleted) surfaces a warning and transparently starts a fresh thread, keeping the displayed history (the model just can't see it) — then `chatSend`s. **Delta batching**: word-level `delta` events accumulate in a buffer and flush to the store at most every ~80 ms (≤ ~12 writes/s; the `message` event replaces the accumulation and clears `streaming`) — never a store write per delta; message rows are memoized so only the streaming row re-renders. `tool_call`/`tool_done` append/patch mono tool chips (FIFO-matched per tool name); `turn_failed` and `warning` become warning messages (a `chatSend` rejection adds one only when no `turn_failed` event already did); turn end clears the per-chat busy flag, which also swaps the send button back from its interrupt (stop) form. **Jump chips**: tool args containing a live pane id (scanned from `args_summary`) and panes born during a `create_panes` call (order diff around the call — the tool *result* flows Rust → codex and never reaches the frontend) attach `paneRefs` to the tool message; the UI renders "→ name" chips that jump via `focusAgent` + `focusTerm` (the command-palette pattern) and hide themselves once the pane is gone. **Availability**: on first panel open per app run, `chatStatus` is checked once — not running / not logged in replaces the input with a quiet inline notice (+ recheck button) instead of letting the first send error.

## Orchestrator status pings (Phase 5)

When the orchestrator prompted a pane and that pane finishes, the owning chat gets a visible, persisted system ping — and pending pings are fed back into the model's context on the next turn, so "mach jetzt X" works without re-explaining the fleet.

**Chat context through the bus**: `bus.rs`'s `ToolRequest` carries an optional `chat_id` — the *backend* chat id of the chat whose turn triggered the call, threaded from `appserver.rs`'s `handle_server_request` through `orchestrator::run_tool` into the emitted `orchestrator://tool-request` event (`None` for the `orchestrator_run_tool` dev command and internal calls like `chat_send`'s fleet-summary snapshot — those never track anything). The webview bus resolves it to the *store* chat id (`chatIdForBackend` in controller.ts) and hands every executor a `ToolCallContext` `{chatId: string | null}` as second argument.

**Touched-pane tracking** (chat-store.ts, persisted with the chat): `OrchestratorChat.touchedPanes` (`Record<paneId, {name, lastPromptAt}>`) records every pane the chat prompted — `prompt_pane` (submit true *or* false) and `create_panes` startup prompts that actually delivered, both via `notePromptDelivered` in executors.ts. `pendingPings` (`{paneId, paneName, activity, at, delivered}`, oldest first, capped at 20 — `MAX_PENDING_PINGS`, oldest dropped, delivered *and* undelivered count against the cap) is the injection queue; both survive restarts (a pane finishing just before a quit still reaches the model after relaunch). Hydration sanitizes both per entry, like messages.

**Activity watcher** (controller.ts, `startOrchestratorActivityWatcher()`, started once from App.tsx next to `startOrchestratorBus`): a `useSwarm.subscribe` OUTSIDE React (early-outs on unchanged `state.agents` reference) watches `Agent.activity` transitions. On busy → idle/waiting of a pane in some chat's `touchedPanes`, every owning chat gets (a) a persisted `system` message — compact English (one product language): `«name» finished` / `«name» waiting for input`, with a `paneRefs` entry so the existing jump chip renders — and (b) an undelivered `pendingPings` record. **Debounce**: a pane that pinged < 3 s ago is skipped (flap guard, `PING_FLAP_MS`, across chats), and transitions within 2 s of the chat's own prompt delivery are startup noise (`PROMPT_SETTLE_MS`, per chat). Pings fire even while the chat's turn is running — they simply ride along with the *next* send; nothing auto-starts a turn (that's a later feature).

**Context re-injection** (`sendMessage`): right before the wire write, the chat's undelivered pings are marked delivered (`takePendingPings`) and a compact status block is prepended to the outgoing text — `[Status update] «api» finished at 14:32.` / `«api» waiting for input since 14:32.` (one English sentence per ping, newline-separated, absolute HH:MM) + blank line + the user text. **Wire text only**: the stored user bubble stays the raw text (the pings are already visible as system messages). Rust's `chat_send` still prepends its `[fleet status: …]` line — both stay; the fleet line is the *current* snapshot, the status block is *what happened since the last turn*. Pings arriving after the take stay undelivered for the following send; a failed send does not restore taken pings (the turn error is visible in the chat).

**"Review"**: system pings with a `paneRefs` entry render a quiet mono chip button that sends a normal, visible user turn ("Read the transcript tail of pane «name» (id) and summarize briefly …") — disabled while the chat is busy. English like all UI copy; the orchestrator answers in whatever language the user writes (its instructions say so).

## Orchestrator brain: OpenRouter loop + settings (Phase 6)

A second brain behind the same chat UI: an OpenRouter tool-calling loop over the SAME registry, plus the Settings → Orchestrator section with the operational knobs.

**Provider per chat**: every `OrchestratorChat` carries a persisted `provider: "codex" | "openrouter"` (missing = codex, for pre-Phase-6 chats), stamped at creation by `createChat()` in controller.ts from `settings.orchestratorProvider` — a chat keeps its brain for life; switching the setting only affects NEW chats (re-using an *empty* chat re-stamps it, since it has no history yet). OpenRouter chats also capture `model` at creation (`settings.orchestratorModel`, default `DEFAULT_ORCHESTRATOR_MODEL` in openrouter-loop.ts). The panel shows a faint mono provider/model line at the top of every chat ("codex" / "openrouter · google/gemini-3.5-flash"); the input-area availability notice is provider-aware — codex chats gate on the app-server check (which is skipped entirely while an openrouter chat is active, so a pure OpenRouter setup never spawns codex), openrouter chats on the stored key (`openrouterStatus`).

**Loop** (`src/lib/orchestrator/openrouter-loop.ts`, `runOpenRouterTurn`): `sendMessage` routes by provider *after* the provider-agnostic ping injection. The loop appends the user wire message, then iterates completion → tool calls → tool results until the model answers without tool calls, hard-capped at 15 iterations (then a warning + normal turn end). Tool calls execute **directly against the local executors** (`executors.ts`, with `{chatId}` context — touched-pane tracking and all Phase-5/6 guards apply; no Rust roundtrip, which also means the registry's arg validation is skipped — the executors' own semantic errors cover it). Results are JSON-stringified, truncated to ~8 k chars, and appended as `tool` wire messages; executor errors become `ERROR: …` result strings (same convention as the codex bridge). All UI comes from the SAME controller plumbing: the loop synthesizes chat events (`turn_started`, `tool_call`/`tool_done`, `message`, `warning`, `turn_completed`/`turn_failed`) through the controller's `handleEvent` (passed in as `opts.emit` — no import cycle), so delta batching, tool chips and create_panes pane-ref diffs are shared, not duplicated.

**Rust side** (`src-tauri/src/orchestrator/openrouter.rs`, command `openrouter_chat_completion(chat_id, model, messages, tools)`): one streamed POST to `/chat/completions` (`stream:true`, `tools`, `tool_choice:"auto"`) per loop iteration, keyed with the keychain key (never exposed to JS, like all of openrouter.rs). Content tokens emit as the SAME `orchestrator://chat-event` `delta` payloads the app-server client emits — under the **store** chat id, which the controller self-links into its backend↔chat map. The SSE / index-keyed tool-call-fragment assembly (`SseAssembler`) is tauri-free and unit-tested against fixture chunks (chunk boundaries may split lines and UTF-8 sequences; `data: {"error":…}` events surface as errors); the call resolves with the assembled `{ content, tool_calls: [{id,name,arguments_json}], finish_reason }`. Timeouts: 15 s connect, 120 s max silence *between* chunks; 401/403, 402 and 429 map to messages naming the cause (invalid key / out of credits / rate limit).

**Wire history**: OpenAI-format messages persisted per chat (`chat.wire`), appended progressively during the turn and capped at `MAX_WIRE_MESSAGES` (60) — a capped history simply loses old context like any long chat (after capping, leading orphaned `tool` results are trimmed so the history stays valid). The system message is **not** persisted: it is rebuilt fresh each turn as instructions + `[fleet status: …]` line, mirroring what Rust `chat_send` prepends for codex. Hydration sanitizes wire entries like messages (per-entry degradation).

**Interrupt**: `interruptOpenRouterTurn` sets a per-chat abort flag (checked between loop iterations and between tool calls) *and* calls `openrouter_chat_cancel`, which flips an atomic checked between stream chunks — the completion then resolves with `finish_reason:"cancelled"`, keeping the partial streamed text; the turn ends as "interrupted". An interrupt that lands mid-tool-batch still appends placeholder `ERROR: interrupted` results for the remaining calls so the wire history stays valid.

**Settings** (Settings → Orchestrator, all via `updateSettings`; unset = default): `orchestratorProvider` (default codex) and `orchestratorModel` (the OpenRouter default; input + live-catalog datalist) apply to new chats. For the **codex** provider, `orchestratorCodexModel` + `orchestratorCodexEffort` are the defaults NEW codex chats are stamped with, edited through the **same shared `ModelEffortPicker`** the chat headers use (one chip → the "Available" catalog from `model/list` + Recent + Custom + the effort enum) — Settings just passes a `footer="Default for new chats."` and a chip trigger showing the current value or "Default". Semantics: **Settings = default for new chats, the per-chat picker = this chat from the next turn.** The per-chat `model`/`effort` live on `OrchestratorChat` (persisted), ride on `chat_send` as turn/start overrides (codex) or feed the OpenRouter model (openrouter, effort ignored there), and are editable mid-chat via the same picker. The section also SURFACES the shared OpenRouter key status (valid / rejected / missing, from store `openrouterStatus`) with a "Manage in Voice" jump — the key itself is managed once, in the Voice section, so there is exactly one key field app-wide; the chat panel's availability notices point to Settings → Orchestrator. Two guard knobs are read live by the executors: `orchestratorAutoSubmit` (default on; **off = review mode** — `prompt_pane` and `create_panes` startup prompts paste but never submit, and the tool response carries `submitted:false` + a note so the model can tell the user) and `orchestratorBusyPolicy` (`"deliver"` default = current queue-with-warning behavior; `"refuse"` = `assertPromptAllowed` errors on busy panes, telling the model to wait). `orchestratorScanRoots` (folder-list editor) is the `list_projects` default when the model passes no `scan_roots`. The `PersonaControls` / `MemoryControls` blocks (persona editor + memory list) live at the bottom of the same section — see the next section.

## Orchestrator persona & memory

The orchestrator has a configurable **persona** (voice/self-image) and a small curated **memory**, both compiled into its system instructions by ONE function so both provider paths (codex + OpenRouter) share the exact same prompt.

**`build_instructions(persona, memory)`** (`src-tauri/src/orchestrator/persona.rs`) assembles, in fixed order: (1) a **persona header** compiled from the editable fields (name/role/tone/principles — voice only); (2) `OPERATIVE_CORE` — the WORD-FOR-WORD former `ORCHESTRATOR_INSTRUCTIONS` (tools, context discipline, prompting, worktrees, layout, native sessions, delivery contract, style). **This block is the hard-wired guardrail and is NOT persona-overridable** — every safety rule (never edit files, never push unprompted, human owns approvals) lives here, outside the editable persona. (3) A **memory snapshot** (`## Your memory …`, only when non-empty) + the always-present memory-behaviour rule. Taking persona + memory as PARAMETERS (not globals) is the seam for future per-agent personas — no rework, just other arguments. Both provider paths call it: codex via `thread_start_params`/`thread_resume_params` (`developerInstructions`, re-sent on resume so memory changes land next resume), OpenRouter via the `orchestrator_tools(persona)` command (fetched fresh per turn, not cached, since instructions now depend on live persona + memory).

**Persona** is a `settings.orchestratorPersona` slice (`OrchestratorPersona { name, role, tone, principles[], emoji?, accent? }`, `updateSettings`, unset = the **Maestro** seed — kept in sync between Rust `persona::maestro()` and TS `PERSONA_PRESETS[0]`). Three presets in `src/lib/orchestrator/persona.ts`: **Maestro** 🎼 (default), **Hive** 🐝, **Orchestrator** ⌘ (neutral = the old pre-persona voice, regression-safe retreat). The frontend sends only `name/role/tone/principles` to Rust (`personaWire`); `emoji`/`accent` are UI-only. Persona is captured per codex chat at start (stored on `ChatState`, reused for a transparent respawn-resume) so a running thread keeps its voice; OpenRouter reads it live per turn. The persona surfaces (name + emoji) in the SessionRail Conductor card, the `OrchestratorPanel` header, the ConductorStage header, and both composers' placeholders (`Message {name}…`).

**Memory** is a small Markdown file `orchestrator-memory.md` in the app data dir (next to `swarmz.json`), `src-tauri/src/orchestrator/memory.rs`: one entry per line, ISO-date-prefixed, robustly parsed (bullet lines only; the header + hand-edits tolerated). Written ONLY through the explicit **`remember`** tool (no auto-extraction) — durable user-relevant facts only (the tool description + the memory-behaviour rule spell out what qualifies and tell the model to *propose* uncertain facts and only store after the user confirms; no silent auto-write). **Hard caps on write**: `MAX_ENTRIES` 20 / `MAX_TOTAL_CHARS` 2000 — overflow drops the OLDEST entries (FIFO) and the append result says so honestly ("stored; dropped the N oldest entr… to stay within the cap"). Read fresh into every new session (frozen per session, like Hermes' MEMORY.md). Commands (all `spawn_blocking`, the sync-command invariant): `orchestrator_memory_read` / `orchestrator_memory_append(text)` (the `remember` executor) / `orchestrator_memory_remove(index)` (Settings). The Settings `MemoryControls` block lists entries with a `N/20` count, per-line delete, and the file path (via `@tauri-apps/api/path`); adding is only ever the orchestrator's own `remember` call. **Anti-scope** (V1): no vector store / embeddings, no auto-extraction, no per-project/per-chat scoping, no LLM consolidation — just FIFO. Tests: `persona.rs` asserts the persona compiles ahead of the core, guardrails are present verbatim, the memory block appears only when non-empty; `memory.rs` covers the FIFO caps (count + chars), robust parsing and remove.

**Write guards** (executors.ts): `prompt_pane` errors when the same pane received *any* orchestrator prompt (incl. startup prompts, from any chat) less than 2 s ago (`DOUBLE_PROMPT_WINDOW_MS`, in-memory) — the message says it looks like a duplicate. `assertPromptAllowed` is the single choke point for every orchestrator write into a pane; Phase 6 wired the configurable busy policy into it and the global auto-submit gate next to it (see the next section) — the default behavior (deliver + submit, busy-pane warning) is unchanged.

## Vibe-Mode data layer (`src/lib/vibe/`, Phase 2)

The frontend half of the native Codex sessions (Rust side: "Vibe-Mode native Codex sessions") — store + controller; the UI shell is "Vibe-Mode UI shell (Phase 3)" below.

**Store** (`session-store.ts`, a standalone zustand store like the orchestrator chat-store): `sessions: Record<id, VibeSessionEntry>` + `order` (newest first) + `activeId` + in-memory `busy`. Each entry holds the `VibeSession` meta and the transcript **normalized by id** (`items: Record<id, VibeItem>` + `order`) — a streaming delta replaces ONLY its own item object, so every other item keeps its reference (the t3code identity lesson; one row re-renders per delta batch). `VibeItem` is a discriminated union (`user` / `assistant {text, streaming?} `/ `command {command, cwd?, status, exitCode?, output}` / `fileChange {status, changes}` / `plan` / `webSearch` / `approval {approvalKind, status, payload}` / `warning`); per-turn `turnId`/`diff`/`plan`/`tokenUsage` are transient entry fields (not persisted). **Caps**: 500 items/session (oldest drop from map + order), 30 sessions (oldest dropped); live command output capped 64 KB, persisted output 8 KB. **Persistence** under store key `vibeSessions` (session meta + items), debounced ~800 ms, **flushed by `flushAllPersists`** at quit, hydrated from store.ts `hydrate()` with per-entry graceful degradation — `streaming` is never restored and a persisted pending approval hydrates as `cancelled` (its process is gone).

**Controller** (`controller.ts`, outside React): typed `invoke` wrappers for the eight `vibe_session_*` commands + the single `vibe://session-event` listener. Maps events into normalized store items, batching word-level `delta` writes ~80 ms (only the streaming assistant item's store write; the persist debounce is separate). Public surface: `startSession` (assigns the id, creates the store entry, spawns the process — drops the entry on failure), `resumeSession` (lazy — after a restart the Rust registry is empty, so the first `sendMessage` resumes the persisted thread; a lost rollout warns and continues fresh), `sendMessage` (appends the user item, ensures the backend, fires the non-blocking turn — busy is cleared by the completion event), `interrupt`, `respondApproval` (optimistic item status → Rust), `setAccess`, `closeSession`, `focusSession` (switch to Vibe Mode + activate a session — the Deck/ticker jump target). Turn lifecycle also mirrors into the shared Deck ticker (`pushFleetEvent` with `source:"vibe"`): `approval_request`→waiting, `turn_completed`→finished, `turn_failed`→exited. Busy Vibe sessions count as quit blockers (`vibeBusyIds` → `lib/quit.ts`; the QuitConfirmDialog resolves their names). Dev hook `window.__vibe` (DEV-only, like `window.__orch`; exposes `store` plus rendering-only `seed.*` — `session`/`bigDiff`/`approval`/`queue`/`crossSession` and the perf seeds `longThread`/`manySessions`/`streaming` — and `bench.snapshot()`, all backend-free so the Vibe UI, diff stack and performance can be exercised in the plain Vite dev server). `window.__orch.summary()` is session-aware (passes `fleetSessions()`).

**Pure helpers** (`ui.ts`): `diffStats` (unified-diff +/− line counter, reused for the rail counters and per-file fileChange counts), `vibeSignal`/`hasPendingApproval`/`oldestPendingApprovalAt` (signal-triad derivation — needs-you wins over busy), `totalTokens`, `commandExit`, `approvalCommand`, `shortAge`. `triage.ts` — `vibeTriageEntries` (sessions with a pending approval, oldest first) merged into the Deck queue. `ui-store.ts` — transient dialog state (new-session + close-confirm). `session-store.ts` also stamps a transient `lastBusyEndAt` on busy→idle so the rail can show the ephemeral "✓ finished · Xm" moment.

## Vibe-Mode diff stack & approvals (`src/components/vibe/`, Phase 4)

**Diff library — `@git-diff-view/react` (the documented fallback, not `@pierre/diffs`).** t3code drives `@pierre/diffs` in production, but it breaks our hard criteria: its published `exports` map hides `parsePatchFiles`/`types` (t3code patches package.json to reach them, plus runs a beta + a behavior patch), and it renders inside a Shadow-DOM web component (`:host`) that fights our token theming. `@git-diff-view/react` imports cleanly, ingests **both** a raw unified diff (`data.hunks: string[]`) and raw old/new content — matching our two shapes exactly (`entry.diff` unified turn diff; a fileChange `add` whose `diff` is raw file content, synthesized into a unified add-diff by `diff.ts`) — themes via `--diff-*--` **CSS variables on plain DOM** (`diff-view-pure.css`, re-coloured to the SwarmZ tokens in `styles.css`), and toggles highlighting with a boolean. Pinned `^0.1.6` (`react` + `core` + `lowlight`).

**Off-thread parse + highlight** (`diff-worker.ts` + `diff-highlight.ts`): a small worker pool (2) runs the whole git-diff-view pipeline — parse → raw → syntax → build lines — and posts back `DiffFile._getFullBundle()`; the main thread `_mergeFullBundle`s it into a fresh `DiffFile` handed to `<DiffView diffFile>`. Vite bundles the worker via `new Worker(new URL("./diff-worker.ts", import.meta.url), { type: "module" })`, so it is fully offline (its own ~1 MB chunk, loaded lazily on first highlight). **Deviation from t3code's Shiki-in-worker:** we highlight with the bundled **lowlight** (class-based `hljs-*` tokens) instead of Shiki — class tokens theme via our CSS (Shiki bakes inline colours from a fixed theme that fights the monochrome system), it is synchronous, no WASM, and keeps the worker light; every performance property the Shiki recipe exists for (off-thread, lazy, LRU, `setMaxLineToIgnoreSyntax(1000)` line cap) is preserved. Bundles are LRU-cached in the worker (60) and briefly on the main thread (40); requests dedupe in-flight by the FNV diff hash (`diff.ts` `diffHash`).

**`DiffCard.tsx`** — the shared renderer. `FileChangeCard` (fileChange items) and `TurnDiffFiles` (the split turn diff) both render a header aggregate + one collapsible `FileRow` per file. **Highlighting is lazy**: a row renders plain first (`diffViewHighlight={false}`, the fully-functional path) and only an OPEN row asks the worker, swapping to the highlighted `DiffFile` async — same line count, so no reflow. Large sets mount **collapsed** (`> 3 files` or `> 160 changed lines`); small single edits auto-open. A `> 512 KB` file is byte-capped (`capDiff`) with a `[truncated]` banner. `diff.ts` is the single source for +/- counting (rail counter, cards, the turn-diff chip) and for shaping/splitting diffs; the "new file" (raw content) path is synthesized into a unified add-diff so it renders/counts as additions.

**Turn-diff access**: the stage header carries a `Δ N files +X −Y` chip (`TurnDiffChip`) that toggles a `TurnDiffPanel` above the composer — `splitUnifiedDiff(entry.diff)` → `TurnDiffFiles`, same caps.

**Approval composer takeover** (`Composer.tsx`): when the ACTIVE session has a pending approval the input is replaced by a `PENDING APPROVAL` panel (t3code pattern) — request preview (command: command mono + cwd + reason; fileChange: file list + a plain, height-capped diff preview via `payload.itemId` → fileChange item, fallback payload `changes`, fallback `entry.diff` first file, fallback "no preview available") + an action row destructive-left → primary-right: **Decline · Cancel turn · Allow for session · Allow** (all four decisions through `respondApproval`; the store's optimistic status update makes the next queued approval rise, and `1 of N` counts the queue). Keyboard: **⏎ = Allow, ⎋ = Decline**, a LOCAL handler on the auto-focused panel — no new global shortcut, dialog guards untouched. The amber feed card stays as the resolved-status reference (✓/✗). **Cross-session banner**: a pending approval in a NON-active session shows a thin amber banner atop the FocusStage (`«name» waits for approval` / `N sessions wait for approval`) that `focusSession`s it. Both the takeover and the banner select **primitive signatures only** (first-pending id, count) — never a fresh array — per the `useSyncExternalStore` invariant.

**Diff colour tokens** (`--diff-add` / `--diff-del`, DESIGN.md): semantic diff green/red, distinct from `--success`/`--destructive` (process status), dark-tuned; used for the +/- washes (via `color-mix`) and every +N/−M counter. The re-theme block in `styles.css` is scoped to the lib's `.diff-tailwindcss-wrapper` — no global bleed; the lib's bundled hljs syntax-token colours are kept.

## Orchestrator × Vibe integration (Phase 5)

The orchestrator learns native Vibe sessions as a **second agent species** and becomes the home of Vibe Mode (Orchestrator-first). Nothing new in the process/host layers — this is tool surface + UX over the existing bus, chat store and vibe session store.

**Tools know sessions** (`registry.rs` descriptions + `src/lib/orchestrator/executors.ts`; the schemas stay single-source in Rust):
- **`fleet_snapshot`** now returns `{ summary, ui_mode, workspaces, sessions }`. `sessions` is a pure `sessionSnapshot(...)` (`snapshot.ts`) over the vibe store's ordered entries + busy map — `{id, name, cwd, projectName, model, access, status, turnActive, contextPct, createdAt}` with an EXACT status (`working` / `idle` / `pending-approval`, a pending approval winning over busy). `ui_mode` is `settings.uiMode` ("grid"|"vibe") so the model knows where the user lives. Each workspace also carries a `layout` section (grid px + effective pane px) and the summary gains a crowding hint — see "Orchestrator layout awareness". `fleetSummaryLine(state, sessions)` gained an optional session segment ("… · 2 native sessions, 1 working, 1 waits approval"); the summary Rust prepends to every user turn (via the `fleet_snapshot` executor's `summary`) and the OpenRouter loop's system line both pass `fleetSessions()`.
- **`prompt_pane`** / **`read_transcript`** resolve the id against panes FIRST, then the vibe store (`requireTarget`). A session `prompt_pane` submits one turn via the vibe controller's `sendMessage`; a busy session is REFUSED (a native session runs one turn at a time — unlike a pane, which queues text) via `assertSessionPromptAllowed` (same double-prompt guard, no review-mode paste). A session `read_transcript` renders the normalized items as compact text via the pure `renderSessionTranscript` (`lib/vibe/transcript.ts`): user/assistant texts, `$ command → exit N` lines (output only for a failed exit, last ~5 lines), `files (status): path +N −M` (via `changeStats`), `plan: … — done/total`, `approval (kind): status`. Respects `tail`.
- **`create_panes`** gained a per-pane `native:true` (registry schema + a Rust test): creates a Vibe session (`startSession`, access defaulting to `workspace`) instead of a terminal pane; runtime/profile/worktree/branch don't apply (V1), cwd/model/reasoning→effort/name/prompt do. The batch/error semantics are unchanged.
- **`ORCHESTRATOR_INSTRUCTIONS`** (appserver.rs) gained a "Native Vibe sessions" section: the species (structured transcripts, exact status), when native is right (Vibe Mode / user asks / `ui_mode` "vibe" default signal), and the hard rule **session approvals are decided by the human — never instruct a session to bypass approvals, never promise to approve**.

**Conductor in the stage** (`components/vibe/`): the `FocusStage` is Orchestrator-first. A transient `stageMode` ("conductor"|"session", default conductor) lives in `ui-store.ts`; the pinned Conductor card activates it, a session card / `focusSession` / a freshly created session flip it to "session". ## Orchestrator layout awareness (grid mode)

So the orchestrator places terminal panes with knowledge of SPACE, not just pane counts — the MODEL supplies intent, the CODE owns the geometry and guardrails. Applies to terminal panes only; native Vibe sessions ignore all layout params (they're not in the grid).

**Sensing** (`snapshot.ts`, pure): `fleet_snapshot` attaches a per-workspace `layout` `{ grid, paneCount, tree, smallest }`. `workspaceLayout(layout, agents, dims)` walks the tiling tree with the *measured* container px and renders a compact effective-pane-px string — `cols[ "api" (repo) 800×1000 | rows[ "build" (repo) 800×500 / "review" (repo) 800×500 ] ]` (`cols[…|…]` side by side, `rows[…/…]` stacked; fractions × container, same math as `computeLayout`) plus the smallest leaf by area. `crowdingNote(workspaces)` appends a terse hint to the summary line when a workspace's smallest pane dips below a comfort floor (`ws «1» is tight: 5 panes, smallest 320×1000`). The container is measured in the `fleet_snapshot` executor via `document.querySelector('[data-ws-grid="<id>"]').clientWidth/Height` — **`clientWidth`/`clientHeight` are LAYOUT boxes, unaffected by the fleet view's CSS scale transform, and every grid stays mounted (inactive ones `visibility:hidden`)**, so all workspaces report their true render size (`TilingGrid` carries the `data-ws-grid` attribute; empty workspaces have no grid container → `null` dims → the pane list without a `tree`).

**Planning** (`placement.ts`, pure — the "math"): `planPlacement(input)` resolves the target workspace (`workspace: "current" | "new" | a case-insensitive name` → error on an unknown name; legacy `workspace_id` still honored when `workspace` is absent), separates `beside` specs (echoed straight through as targeted splits), then packs the rest into buckets by capacity. `capacityFor(dims, arrangement, total)` = how many panes fit above `MIN_PANE` (380×240 — the readability floor: `floor(w/380)·floor(h/240)` for a grid, one axis for rows/columns; **unmeasured dims → Infinity, i.e. the old no-overflow behavior**). Panes beyond the target's free capacity OVERFLOW into fresh workspace bucket(s), and same-project panes are kept together whenever a whole group fits in one bucket (grouping preserved across the boundary). New workspaces are named after their single project when there is one. The `arrangement` param ("auto" tiles by count+aspect, else rows/columns/grid) is resolved per bucket. Fully unit-tested via the scratchpad harness (overflow, grouping, keep-whole, `workspace` new/name/unknown, `beside`, capacity/arrangement resolution).

**Execution** (`executors.ts` `create_panes` — the "guardrails"): it gathers grid dims, builds the plan, and only carries it out. Per bucket it activates/creates the workspace, creates the panes sequentially (the git-lock reason is unchanged), then REPLACES the workspace layout with a balanced, equal-size tree — `buildArrangement(agentIds, arrangement, aspect)` (columns = row split, rows = column split, grid = ⌈√n⌉-wide rows; all equal sizes — this is the fix for the "each new pane smaller than the last" split cascade) grafted beside the untouched existing layout via `combineLayouts` (proportioned by pane count). `beside` panes are created then re-inserted next to their target with the existing `splitPane`. The rebuild goes through the new store action `setWorkspaceLayout(wsId, layout, activeAgentId?)` — agents keep their PTYs (the load-bearing "PTYs die only on store removal" invariant holds: pane nodes are keyed by `agentId`, so only placement changes). Hand-split/drag paths (`splitPane`/`movePane`/`setSplitSizes`) and existing panes' layout are untouched. The tool result carries an honest `summary` (`created 4 in «1»; 2 in «api» — overflowed …`) and each pane result its `workspace_id`/`workspace_name`.

## Performance invariants

PTY output is coalesced (≤12 ms / 128 KiB) and addressed per agent (`pty://data/<id>`); the usage file-watcher emits the changed project-dir names and the frontend skips refreshes for dirs it doesn't display, throttles to ≥2 s, and pauses while the window is hidden. Heavy Tauri commands (usage parsing, git subprocesses, keychain access, PTY kill, model load/inference) run async + `spawn_blocking` — sync commands execute on the main thread and may only do fast, non-blocking work.

**Vibe Mode** rests on three properties, all measured in the Vite dev server with the rendering seeds: (a) the item feed is virtualized (`virtua` `VList`) — a 500-item session (the per-session cap) renders only ~10–13 rows and scrolls a 42k-px feed at avg ~8 ms / p95 ~14 ms per frame, zero long frames; (b) per-id primitive selectors keep re-renders scoped — six sessions streaming deltas concurrently held frame time flat (avg ~8 ms, p95 ~9 ms, 0 long frames) with the feed still at ~10 rows, because a delta re-renders exactly its own row, never the rail; (c) the persist snapshot is cheap and debounced — 200 rapid store writes coalesce into a single ~800 ms-debounced disk write, and building the full persist snapshot at the store cap (30 sessions × 500 items = 15 000 items) takes ~0.4 ms. Never regress these: a store selector that returns a fresh array, an un-memoized feed row, or a per-delta persist would each break one of them.

## Security

CSP is set in `tauri.conf.json` (no remote script/connect sources — the webview talks IPC only). The OpenRouter key is keychain-ACLed to the SwarmZ binary (Security framework). The updater verifies minisign signatures against the pinned pubkey.
