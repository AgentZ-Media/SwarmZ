## Transport & backends

`src/lib/transport.ts` exports the native Tauri backend. Backend capabilities are declared in `backend-types.ts` and implemented in `backend-tauri.ts`. Native-only features may skip the backend interface and use `@tauri-apps/api` directly (e.g. `lib/worktree.ts`, `lib/orchestrator/native.ts`, `lib/vibe/controller.ts`, `lib/projects/store.ts` — the `canonicalize_path` dedupe resolver).

## Store & persistence

`src/store.ts` holds the shared app state. Persisted: usage history, quick notes, and app settings (`AppSettings`: last used folder, codex/git path overrides, auto-update, worktree repo registry `worktreeRepos`, orchestrator prefs — codex model/effort defaults, scan roots, persona) — edited in `SettingsDialog.tsx`, opened via the title-bar gear or `⌘,` (Tauri store `swarmz.json`). Always mutate settings through `updateSettings` (merges + persists the whole object). In-memory only: palette/notes/dashboard flags, the worktree panel scan, `quitConfirm`. Window size/position persists natively via `tauri-plugin-window-state`. Sessions live in their own store (`lib/vibe/session-store.ts`, key `vibeSessions`), orchestrator chats in `lib/orchestrator/chat-store.ts` (key `orchestratorChats`), project tabs in `lib/projects/store.ts` (key `projects`); all three are flushed by `flushAllPersists` at quit and hydrated from `hydrate()` — projects deliberately BEFORE the vibe sessions (the session migration assigns projectIds into the project store).

The old pane-era keys (`grid`, `workspaces`, `workspacePresets`, `commandPresets`, `customCommands`, `profiles`) are DELETED once by the schema-v2 migration (`DEAD_STORE_KEYS` in `lib/schema-version.ts` → `deleteStoreKeys` in the backend).

The store file itself has a crash net: `storefile.rs` runs in Tauri `setup` before the webview loads — a healthy `swarmz.json` refreshes `swarmz.json.bak`, a corrupt one (torn write, disk-full) is moved aside as `swarmz.json.corrupt-<ts>` and the backup restored, so the plugin's silent treat-unreadable-as-empty behavior can no longer factory-reset the app.

`swarmz.json` also carries a top-level **`schemaVersion`** key (a plain number, currently `2`) — the migration anchor, deliberately WITHOUT a migration framework. The pure hooks live in `src/lib/schema-version.ts` (unit-tested): `normalizeSchemaVersion` (valid stored versions — even NEWER ones — are left untouched, so a downgraded build never clobbers a newer store's stamp) and `planSchemaMigration`, which `hydrate()` runs first thing: a raw version `>= 2` plans nothing; `1` or missing/invalid (a possibly pre-rebuild store) plans the one-time dead-key cleanup + the `2` stamp. The plan is idempotent. The actual VALUE migration (sessions → projects) does not run here — it lives tolerantly in the vibe hydrator (see *Projects*). The `storefile.rs` rescue path is version-agnostic and stays untouched.

## Projects & swarm data model (Phase 2)

**Entity** (`Project` in `types.ts`, persisted key `projects` via `lib/projects/store.ts`): `{ id, dir, name, order, lastActiveAt, createdAt, closedAt? }`. `dir` is the CANONICAL folder path — `openProject` resolves it through the Rust `canonicalize_path` command (symlinks, `/var` aliasing; falls back to the normalized input when the folder is gone) and dedupes on it, so the same folder always maps to the same project. `name` is the folder basename with a lowest-free numeric suffix on collision ("api", "api 2"). `order` is the tab position among OPEN projects; `activeProjectId` persists. Pure logic (dedupe key, naming, reorder patches, migration) lives in `lib/projects/core.ts` (unit-tested); the zustand store around it debounces persists (~500 ms, flushed by `flushAllPersists`).

**Open vs closed**: closing a project TAB only stamps `closedAt` — the entity stays persisted and its sessions stay in the session store; reopening the same folder (picker, recents, a triage jump) flips the SAME project open again and the sessions reappear. `setActiveProject` auto-reopens.

**Sessions belong to projects** (schema v2): `VibeSession` carries `projectId`, `agentName` (the generated identity — immutable, branch names derive from it in Phase 4), `spawnedBy: "user" | "conductor"` and `worktree: { root, branch, shared } | null` (carried now, filled by the WorktreePanel flow and Phase 4's tools). **Hydrate migration**: every hydrated session runs through `useProjects.adoptSessions` (pure core: `assignSessionsToProjects`) — a valid `projectId` passes through; pre-v2 sessions and sessions whose project record was lost get projects derived from their `projectDir`s (deduped, matched against open AND closed projects, created OPEN otherwise); `agentName` defaults to the old display name, `spawnedBy` to "user". The stamped result persists immediately (`vibeSessions` writes `version: 2`). The session cap is **30 per project** (`MAX_SESSIONS_PER_PROJECT`, enforced on create and on hydrate).

**Agent names** (`lib/vibe/names.ts`, pure + unit-tested): a 130+ pool of short international first names. `pickAgentName(taken, rng?)` picks collision-free (case-insensitive; per project in the controller, globally strict in the New-Session dialog), suffixing "Maya 2" once the pool is exhausted; `branchSlugForAgent(name, task?)` → `swarm/maya-checkout` (NFKD diacritics-stripped ASCII slugs, task fragment capped at a word boundary). The dialog prefills a generated name with a 🎲 reroll; `create_panes` sessions without a `name` get a pool name too.

**Scoping**: the SessionRail shows only the ACTIVE project's sessions (primitive joined-ids signature; no active project = show all, a degenerate pre-migration state). The Deck triage/counters stay GLOBAL across projects — `focusSession` switches to the session's project tab first (reopening a closed one), then selects it; `activateProject` (tab click, ⌘1–⌘9, palette) aligns the stage: selection moves to the project's newest session, a session-less project lands on the Conductor. The orchestrator's `create_panes` resolves each session's project from its `cwd` (NOT the active tab) and never steals the active tab (`activate: false` for conductor spawns).

**TitleBar tabs** (interim look — Phase 6 restyles): one tab per open project with a live dot (busy = muted, idle = faint), `busy/total` count, `⚑n` needs-you badge, close X (middle-click too), double-click rename, drag reorder; the + button opens a folder picker / discovery-recents dropdown ("Open project" also lives in the ⌘K palette, along with tab switching). Tabs are plain `<div>`s and opt out of the header drag region via `data-tauri-drag-region="false"`.

## Quick notes

**Quick notes** (⌘N, title-bar note button, `QuickNotesPanel.tsx`): a right-side slide-in drawer holding lightweight checklists — one **global** list plus one **per project folder** (store key `quickNotes` → `QuickNotesData`, folder lists keyed by repo root). The drawer opens in the scope of the active session's `projectDir` (fallback Global) and shows a chip per scope. Items (`NoteItem`) check off, edit inline, reorder by dragging the grip, and can toggle to plain text lines (`plain`); "Clear done" drops every checked item, emptied folder lists drop their key. Every mutation persists immediately. The drawer is a `role="dialog"` so the global-shortcut guard kicks in; ⌘N closing it is special-cased in the dialog branch of the `App.tsx` keydown handler.

## Usage & limits

Codex usage is parsed from `~/.codex/sessions/**/*.jsonl` by `src-tauri/src/codex_usage.rs` (which also owns the shared usage types since the Claude parser was removed), using `session_meta`, `turn_context`, `event_msg/token_count`, `task_started` and `task_complete` rollout events for session id, cwd, model, tokens, context, limits and activity. The usage commands (`usage_for_dir`, `usage_for_session`, `usage_totals`) are async + `spawn_blocking` and codex-only; they currently have no frontend caller — live session accounting runs frontend-side instead (see below). Codex ChatGPT-plan rollouts do not include reliable USD cost, so cost stays zero (there is no pricing table anymore).

Codex limit windows are **account-level**: `codex_account_limits` finds the newest `token_count.rate_limits` event across ALL of `~/.codex/sessions/**/*.jsonl` — files are iterated newest-mtime-first, each only via a bounded 256 KiB tail read, stopping at the first hit — and is polled every 60 s by `src/lib/limits.ts` (paused while hidden) into the Deck meters. `limits: null` = no data ever seen (the Deck shows a quiet `CX —` placeholder); data older than 1 h gets an "as of HH:MM" tooltip suffix.

The **usage dashboard** (`UsageDashboard.tsx`, title-bar chart button) shows the persisted all-time history (store key `usageHistory`, Codex entries; the pre-rebuild Claude/shell entries are filtered from view). **Session accounting** feeds it live: the vibe controller mirrors each session's `token_usage` events into `recordUsageHistory` (keyed `codex:<session id>`, `runtime:"codex"`, `by_model` from the session's model override or "codex default") — debounced ~2 s per session, flushed once more on close. The thread's `total` bucket is cumulative, so each write REPLACES the entry; cached input counts as cache reads and is subtracted from fresh input; `message_count` = user turns in the transcript; cost stays 0 (no pricing).

## Deck & fleet events

The **Deck** (`components/Deck.tsx`) is the permanent ~30 px status bar under the session stage. Title bar = place/navigation, Deck = system status (see DESIGN.md). Anatomy left→right:

1. **Triage queue**: `⚑ N need you` in `--attn` — hidden entirely at N=0, and GLOBAL across all projects (the Deck aggregates the whole swarm, unlike the project-scoped rail). Click opens a popover listing every session with a pending approval, oldest waiting first (`vibeTriageEntries` in `lib/vibe/triage.ts`); arrow keys rove, Enter jumps (`focusSession` — which switches to the session's project tab first). ⌘⇧A jumps to the oldest directly.
2. **Event ticker**: the last 3 fleet events inline (newest last), each clickable to jump while its session still exists; a history icon opens a popover with the last ~30.
3. **Meters**: the account-level Codex meters (`cx 5h`, `cx wk`) with full-word tooltips.
4. **Orchestrator dot**: `orch` + dot — `--attn` when any chat has undelivered pings, `--ring` while a turn runs, faint otherwise; click activates the Conductor stage.

**Fleet event feed** (`src/lib/events.ts`, a standalone zustand store): `FleetEvent { id, at, kind, sessionId, sessionName }` with kinds `finished` / `waiting` / `orch_prompt` / `created` / `exited`, capped at 50, **in-memory only by design**. Producers: the vibe controller (`approval_request`→waiting, `turn_completed`→finished, `turn_failed`→exited) and `notePromptDelivered` in `orchestrator/executors.ts` (`orch_prompt`). Repeated `waiting` for the same session within 3 s is deduped in the feed itself (aligned with the ping flap guard).

## Git status

`git_info` (`src-tauri/src/git.rs`) is a strictly read-only snapshot (repo, branch, ±lines, untracked, `origin` URL) that shells out to the git binary (prefers the Settings `gitPath` override, else `/usr/bin/git`; every subprocess runs through `output_with_timeout`). There is no polling anymore — the orchestrator's `git_status` tool and the worktree flows call it on demand.

## Git worktrees

`src-tauri/src/worktree.rs` + `src/lib/worktree.ts`, native-only direct `invoke` — deliberately *writing* git, unlike the read-only git.rs. `worktree_add` creates `<repo>/.worktrees/<slug>` via `git worktree add -b` off the current HEAD (main root resolved via `--git-common-dir`), excludes `.worktrees/` via `.git/info/exclude` (never the tracked `.gitignore`), and copies every untracked file (`ls-files --others`, incl. gitignored `.env`s) minus the heavyweight `SKIP_DIRS`. In the Phase-1 interim state nothing creates worktrees from the UI — the V1 way to run a session inside a worktree is picking the worktree folder in the New-Session dialog; Phase 4 gives the orchestrator worktree tools.

**Management panel** (`WorktreePanel.tsx`): a title-bar `FolderGit2` button, visible only while ≥1 worktree exists, lists all SwarmZ worktrees grouped per repo (dirty/ahead state, refreshed on open) with open-in-session (the session runs in the worktree but belongs to the MAIN repo's project tab and carries the `worktree` meta) or jump-to-session, reveal-in-Finder, per-row finish/delete and a "Clean up safe" bulk action for clean unattached worktrees (a live session's `projectDir` counts as attached). Risky rows (uncommitted changes or local-only commits) require a second delete click; deletion re-checks status at execution time — on doubt the worktree is kept. Scanned repo roots persist in `settings.worktreeRepos` (pruned automatically once a repo has no worktrees left; `worktree_list` returns a `WorktreeScan {entries, scanned}` and only roots that actually **scanned** empty are pruned — an unmounted volume or broken git override never loses orphans). A hand-deleted worktree folder still reports its branch's unreachable commits as `ahead`. The keep-vs-delete dialog (`CloseWorktreeDialog.tsx` + the store's `requestCloseWorktree`/`resolveCloseWorktree`) is dormant in Phase 1 — Phase 4 wires session-worktree close into it.

## Status reporting & quit guard

Quit guard (`src/lib/quit.ts`, started in `App.tsx`): closing the app raises `QuitConfirmDialog.tsx` when a session has a turn in flight (`vibeBusyIds` — the run would be interrupted). Native covers both paths — window close via `onCloseRequested` (every first close is preventDefault'ed, all debounced persists are flushed (`flushAllPersists`), then `close()` is re-issued past a module-level `confirmed` flag so the window-state plugin still saves), and ⌘Q/menu quit via Rust (`lib.rs` run-callback: `ExitRequested` with `code.is_some()` → `prevent_exit()` + emit `app://quit-requested`; `prevent_exit` is a tauri-built-in no-op for the updater's `RESTART_EXIT_CODE`, and code `None` = last window closed passes, which is how a confirmed quit actually exits).

## Orchestrator sensing

Read-only sensing. `src-tauri/src/transcript.rs`: `transcript_read` returns actual message content from codex rollout files (found by the id suffix in the filename, head-scan fallback) — the `event_msg` `user_message`/`agent_message` chat stream preferred, `response_item` messages (instruction wrappers filtered) as fallback. Options: `tail_messages` (default 20), `max_bytes` (default 1 MiB — huge files are seek-tailed to the first complete line, never fully read) and `include_first_user_message` (from a separate capped head read). Texts are char-truncated (umlaut-safe) with an `…` marker. Note: the orchestrator's `read_transcript` TOOL renders LIVE sessions from the vibe store (`renderSessionTranscript`) — `transcript_read` reads rollout files on disk and currently only serves the dev hook. `project_docs` reads README.md/AGENTS.md/CLAUDE.md of a root (worktree paths collapse to the main repo; ~8 KB/file, ~20 KB total; missing files omitted). `src-tauri/src/projects.rs` (codex-only): `discover_projects` merges project folders from Codex `session_meta` cwds, frontend-known folders (project tabs — open and closed —, live session dirs, quick-note folder keys, `worktreeRepos`, `lastCwd` — collected in `lib/orchestrator/native.ts`) and a shallow scan (depth ≤ 2) for `.git` (dir or file); worktree paths collapse onto the repo root, entries dedupe and sort by last activity. It also feeds the "open project" recents (TitleBar + dialog). Both commands are async + `spawn_blocking`, uncached.

Frontend: `src/lib/orchestrator/` — `snapshot.ts` (pure `sessionSnapshot`/`sessionCounts`/`fleetSummaryLine` over the vibe store's entries + busy map), `native.ts` (direct-`invoke` wrappers), `types.ts` (payload + tool-bus types), and `dev.ts`, a dev-only smoke-test hook exposing `window.__orch`.

## Orchestrator tool bus

**The registry is the single source of truth**: `src-tauri/src/orchestrator/registry.rs` defines every tool as `{ name, description, parameters (JSON Schema), timeout_ms }`; `tool_definitions()` serializes the catalog (**9 tools**) for the Codex app-server (`dynamicTools`). The `orchestrator_tools(persona)` command returns `{ instructions, tools }` — the tools plus the compiled system instructions (`persona::build_instructions`). The frontend mirrors the *names* only (`ORCHESTRATOR_TOOL_NAMES` in `types.ts`, executor keys). Argument validation is a deliberately hand-rolled subset (required props, basic `type` tags, `enum`, array `minItems`/`maxItems`/`items`, applied recursively).

**Roundtrip bus** (`orchestrator/bus.rs` + `src/lib/orchestrator/bus.ts` + `executors.ts`): tools are defined in Rust but *execute in the webview*, where the Zustand stores live. `run_tool` validates against the registry, parks a tokio oneshot in a pending map, emits `orchestrator://tool-request` `{ id, tool, args, chat_id }` (`chat_id` = the triggering backend chat, null for dev/internal calls), and awaits the response with the tool's timeout (15 s default; `create_panes` 120 s — it spawns codex processes). The webview bus (`startOrchestratorBus()`, registered once from `App.tsx`) looks up the executor, runs it and **always** answers via `orchestrator_tool_response(id, ok, payload)`. The bus core is tauri-free and unit-tested. `orchestrator_run_tool` is the dev-hook surface (`await __orch.tool("fleet_snapshot", {})`).

**The 9 tools** (executors in `src/lib/orchestrator/executors.ts`; `pane_id`s resolve against the vibe session store, unknown ids error with the list of valid ones): `fleet_snapshot` (`{summary, sessions}` — pure `sessionSnapshot`, exact status working/idle/pending-approval) · `read_transcript` (live session items rendered compactly via `renderSessionTranscript`) · `read_project_docs` (exactly one of `pane_id` — the session's `projectDir` — or `path`) · `read_notes` (quick notes as `{text, done}`) · `git_status` (exactly one of `pane_id`/`path` → live `git_info` + derived `dirty`, or `git:null` with a note) · `list_projects` (`scan_roots` omitted = the user's `orchestratorScanRoots`) · `prompt_pane` (write: submits one turn; a busy session REFUSES — the backend rejects a second turn too — via `assertSessionPromptAllowed`, which also carries the 2 s double-prompt guard) · `create_panes` (write: 1–8 native sessions created sequentially — each start spawns a codex process — with per-entry errors that never abort the batch; initial `prompt`s submit in parallel once all sessions exist; access defaults to `workspace`; sessions are marked `spawnedBy:"conductor"`, get a pool agent name when unnamed, and their PROJECT resolves from the spec's `cwd` — opening its tab without stealing the active one) · `remember` (write-to-disk, → `orchestrator_memory_append`).

## Codex app-server host (`src-tauri/src/codex/`)

The generic layer under every `codex app-server` consumer — the orchestrator brain and the native sessions. Spawn, framing and respawn code exists exactly ONCE, here.

**Wire protocol** (`codex/protocol.rs`, pure + unit-tested): newline-delimited JSON **without** the `"jsonrpc":"2.0"` header (verified against codex 0.144.1). Our requests use an own ascending numeric id; server-initiated request ids are kept as raw JSON and echoed back exactly. `parse_line` classifies each stdout line into response / server request / notification, defensively. The unit tests freeze the live-captured 0.144.1 shapes (reference schema dump + inventory in `docs/codex-protocol/`): `item/started`/`item/completed` with `commandExecution` and `fileChange` items, `item/agentMessage/delta`, `turn/diff/updated`, `turn/plan/updated`, `thread/tokenUsage/updated`, `account/rateLimits/updated` (account-scoped — carries NO threadId) and the approval requests, plus the four decisions `accept`/`acceptForSession`/`decline`/`cancel`. There is NO `item/updated` in the protocol.

**Client & binary resolution** (`codex/host.rs`): `Client::spawn` starts `<program> app-server` via `tokio::process` with one writer task (ordered stdin), a stderr logger and a reader task that resolves responses via a numeric-id pending map and forwards server requests/notifications. Reader EOF = process death: the `alive` flag drops and all pending RPCs fail with a clear error. The binary is resolved to an ABSOLUTE path (Settings override → current $PATH → well-known install dirs → `zsh -lc 'command -v codex'` login-shell probe) because macOS gives the built GUI app a minimal PATH; the child's PATH is enriched with the binary's dir. Resolution is cached process-wide (`set_codex_override` installs the Settings value and drops the cache).

**Thread registry & event routing**: a `Connection` (Client + `initialize` handshake + router task) holds a route table `threadId → EventSink`. The router forwards each server event to the sink registered for its `params.threadId`; routed server requests carry a `Responder` (owning client + echoed id). No sink responsible → notifications are ignored and server requests are refused with -32601. Process death delivers `ThreadEvent::Exited` once per DISTINCT sink and drains the routes.

**Process strategies & respawn**: a `ProcessHost` is one lazily (re)spawned connection slot with a generation counter. The two strategies are just slot allocation: **(a) shared process** — the orchestrator keeps ONE static slot and multiplexes all chat threads over it; **(b) dedicated process per thread** — each session allocates a private slot, so a crash kills only that session. A consumer that sees a different generation than it stored transparently `thread/resume`s before the next turn. `resume_thread` classifies failures typed: `ResumeError::ThreadNotFound` is the consumer's cue to fall back to a fresh `thread/start`.

**Tests**: framing + frozen fixtures in `codex/protocol.rs`; pending map, resolution, routing/exit-dedupe and resume classification in `codex/host.rs`. The `#[ignore]`d spikes (`session_spike`, `sessions_spike`, `appserver_spike`) are the live proofs against the real codex CLI: `cd src-tauri && cargo test <name> -- --ignored --nocapture`.

## Native Codex sessions (`src-tauri/src/codex/sessions.rs`)

Plain agentic Codex sessions driven natively and mirrored into the SwarmZ UI. Process strategy **(b)** — each session owns a PRIVATE `ProcessHost` slot. Unlike the orchestrator, a session keeps codex' STANDARD harness intact — **NO `dynamicTools`, NO `developerInstructions`**.

**Registry & profile**: a static `SESSIONS` map (`session_id → SessionState`) holds each session's private `ProcessHost`, `thread_id`, spawn `generation` (respawn detection → transparent resume), profile (`cwd`, optional `model`/`effort`, `access`), `current_turn_id`, a `busy` flag (one turn per session — claimed synchronously in `send`), the unanswered approval `Responder`s, and the session's `EventSink`. `session_id` is assigned by the frontend.

**Access → sandbox** (exact wire strings verified against 0.144.1): `workspace` = sandbox `workspace-write` + approvalPolicy `on-request`; `full` = sandbox `danger-full-access` + approvalPolicy `never`. **`model` and `effort` are per-turn overrides** (both ride on every `turn/start`) — that is what lets the user change model/effort mid-session. An **access change** takes effect on the NEXT turn via a per-turn override attached only when access changed since the last turn.

**Commands** (all async, in lib.rs): `vibe_session_start` → `{thread_id}`; `vibe_session_resume` → `{thread_id, resumed}` (a `ThreadNotFound` rollout-gone falls back to a fresh `thread/start`); `vibe_session_send` → `{turn_id}` **non-blocking** (a busy session errors); `vibe_session_interrupt`; `vibe_session_respond_approval` (decision ∈ `accept|acceptForSession|decline|cancel`, resolves the remembered `Responder`); `vibe_session_set_access`; `vibe_session_set_model_effort`; `vibe_session_close` (best-effort interrupt, answer every pending approval with `cancel`, unregister, drop the entry).

**Event mapping → `vibe://session-event` `{session_id, kind, data}`** (pure `map_notification`, unit-tested): `turn_started` · `delta` · `message` · `item_started`/`item_completed` (items normalized; commandExecution output capped 64 KB; **userMessage + reasoning dropped**) · `item_output_delta` · `turn_diff` · `plan` · `token_usage` · `approval_request` · `turn_completed` · `turn_failed` · `warning` · `process_exited`.

## Orchestrator brain: Codex app-server

**Process lifecycle** (`src-tauri/src/orchestrator/appserver.rs`, a thin consumer of `src-tauri/src/codex/`): ONE shared `ProcessHost` slot multiplexes all orchestrator chats over a single long-lived `codex app-server` child, spawned lazily on first use — the Settings `codexPath` override is passed by the frontend on every `chat_start`/`chat_status` call. On process death, running turns resolve as failed and every chat gets a `warning` event — the next call respawns; chats transparently `thread/resume` before the next turn (rollouts persist on disk and codex restores the declared dynamic tools from them). A `ThreadNotFound` resume failure surfaces as an error and the frontend controller handles it by starting a new thread.

**Chat ↔ thread mapping**: `orchestrator_chat_start` runs the `initialize` handshake (`capabilities.experimentalApi: true`) and `thread/start` with `cwd` = home, `sandbox: "read-only"`, `approvalPolicy: "never"`, the whole registry adapted to the experimental `dynamicTools` field, and the compiled instructions (persona + memory + operative core) as `developerInstructions`. The persona is passed by the frontend on start/resume and captured on the chat; memory is read fresh each start/resume. `orchestrator_chat_send(chat_id, text, model?, effort?)` prepends a fresh fleet-summary line (`[fleet status: …]`, best-effort via `run_tool("fleet_snapshot")`) to the user text, starts the turn and resolves with `{status, text}` on `turn/completed`; `model`/`effort` are per-turn `turn/start` overrides. `thread/tokenUsage/updated` re-emits as a `token_usage` chat event for the context gauge. `orchestrator_chat_status` reports `{running, version, account}` and never rejects for a dead process.

**Dynamic-tools bridge** (`orchestrator/adapter.rs`): `item/tool/call` server requests are answered from the bus — `orchestrator::run_tool` with args normalized, the result adapted to `{success, contentItems:[{type:"inputText", text}]}`. Each call runs in its own task so deltas keep streaming during tool roundtrips.

**Event stream**: `orchestrator://chat-event` `{chat_id, kind, data}` with kinds `turn_started`, `delta`, `message`, `tool_call` `{tool, args_summary}`, `tool_done` `{tool, ok}`, `token_usage`, `turn_completed`, `turn_failed`, `warning`. Typed wrappers + `onChatEvent` in `src/lib/orchestrator/chat.ts`.

**Approvals & safety**: under read-only sandbox + `approvalPolicy: "never"` + the instructions, approval requests must never occur — if one arrives anyway it is auto-DECLINED and any other server-initiated request is refused with a JSON-RPC error; both emit a `warning` event. Known gap: MCP servers from the user's codex config still attach to orchestrator threads — the instructions forbid using non-SwarmZ tools, but that is instruction-level, not enforced.

## Orchestrator chat UI (Conductor stage)

The user-facing chat over the brain lives in the **Conductor stage** (`components/vibe/ConductorStage.tsx`), reached via ⌘⇧O, the title-bar bot button or the Deck orch dot. The former ⌘⇧O side panel was removed with the grid — the stage is the only orchestrator surface.

**Chat presentation** (`components/orchestrator/ChatView.tsx` — purely a view over the chat-store): a t3code-style human-readable chat. Reading-width column (`CHAT_MAX_W ≈ 46rem`), composer flush beneath it; body text ~13.5px sans, tool/meta lines small mono. Consecutive tool calls fold into one **activity block** (`groupChatMessages` + `toolActivityLabel` in `lib/orchestrator/tool-labels.ts`, pure + unit-tested). A `QuietLine` primitive renders EVERY tool step, status ping and warning identically (signal-triad `StatusIcon`); a lone step renders as one quiet line, multiple steps fold into a collapsible group with session **jump chips first-class** (union row even collapsed, failed steps stay visible). Status pings (`role:"system"`) classify via `systemPingKind(text)`. Unknown tools fall back to `"Used a tool"`; unknown pings to the neutral `info` marker. **File-path pills** (`lib/paths.ts`, `splitTextWithPaths`, pure + unit-tested) render paths in prose as filename pills (full path in the tooltip) inside `OrchestratorMarkdown` — a dependency-free markdown subset built as React nodes, never innerHTML. The shared `ChatMeta` (model/effort chip + context gauge) sits in the stage header; the chip opens the shared `ModelEffortPicker` (per-chat override, applies from the next turn) whose "Available" section comes from the app-server's `model/list` (`codex_list_models`, fetched once per run via `useCodexModels`), with recently-used (from the usage history) + free text as fallback.

**Store slice** (`src/lib/orchestrator/chat-store.ts`, standalone zustand): `chats: OrchestratorChat[]` (`{id, threadId, model?, effort?, title, createdAt, messages, touchedPanes, pendingPings}`, newest first), `activeChatId`, plus in-memory `busy`, `tokenUsage` and the codex `status`. Message roles: `user` / `assistant` (transient `streaming`) / `tool` (`{tool, argsSummary, ok?, paneRefs?}`) / `warning` / `system`. Persisted under `orchestratorChats`, debounced ~800 ms, flushed at quit, hydrated with per-entry graceful degradation — pre-rebuild chats may carry `provider`/`wire` (OpenRouter era); both are dropped tolerantly on hydrate (an old OpenRouter chat keeps its visible history, its model stamp is cleared). Caps: ~200 messages/chat, ~30 chats. `deleteChat` only removes the chat from SwarmZ — the codex thread rollout stays on disk.

**Controller** (`src/lib/orchestrator/controller.ts`, outside React): owns the store-chat ↔ backend-`chat_id` mapping and routes `onChatEvent`. `sendMessage` appends the user message, auto-titles from the first user message, lazily `chatStart`s — or `chatResume`s the persisted `threadId` after a restart; a failed resume warns and starts fresh — then `chatSend`s. **Delta batching**: word-level `delta` events flush at most every ~80 ms; message rows are memoized so only the streaming row re-renders. `tool_call`/`tool_done` append/patch tool messages (FIFO-matched per tool name). **Jump chips**: tool args containing a live session id, and sessions born during a `create_panes` call (order diff around the call), attach `paneRefs`; the UI renders "→ name" chips that jump via `focusSession` and hide once the target is gone. `createChat()` stamps new chats with the Settings codex model/effort defaults (a reused EMPTY chat is re-stamped).

**Status pings**: `bus.rs`'s `ToolRequest` carries the backend `chat_id`, resolved to the store chat (`chatIdForBackend`) and handed to every executor as `ToolCallContext`. `OrchestratorChat.touchedPanes` records every session the chat prompted (`notePromptDelivered`); `pendingPings` (capped 20) is the injection queue — both persisted. The activity watcher (`startVibeSessionActivityWatcher`, a `useVibe.subscribe` outside React, started once from App.tsx) watches busy/pending-approval transitions: a touched session finishing (busy → idle) or raising an approval pings every owning chat — a persisted `system` message (`«name» finished` / `«name» waiting for input`, with a jump chip and a "Review" button that sends a visible summarize-turn) plus an undelivered ping record. Flap guard 3 s, prompt-settle 2 s. On the next `sendMessage`, undelivered pings are marked delivered and prepended as a `[Status update] …` block to the WIRE text only — the stored user bubble stays raw.

## Orchestrator persona & memory

**`build_instructions(persona, memory)`** (`src-tauri/src/orchestrator/persona.rs`) assembles, in fixed order: (1) a **persona header** compiled from the editable fields (name/role/tone/principles — voice only); (2) `OPERATIVE_CORE` — the hard-wired behaviour contract (tools, context discipline, prompting, session doctrine, delivery contract, style). **This block is NOT persona-overridable** — every safety rule (never edit files, never push unprompted, human owns approvals) lives here. The Phase-1 core is the codex-only interim edit of the former word-for-word text (pane/layout/grid sentences removed; content tests updated); Phase 3 replaces it wholesale with the swarm doctrine. (3) A **memory snapshot** (only when non-empty) + the always-present memory-behaviour rule.

**Persona** is `settings.orchestratorPersona` (`{ name, role, tone, principles[], emoji?, accent? }`, unset = the **Maestro** seed — kept in sync between Rust `persona::maestro()` and TS `PERSONA_PRESETS[0]`). Three presets in `src/lib/orchestrator/persona.ts`: **Maestro** 🎼 (default), **Hive** 🐝, **Orchestrator** ⌘. The frontend sends only `name/role/tone/principles` to Rust (`personaWire`); `emoji`/`accent` are UI-only (Conductor card, stage header, composer placeholder).

**Memory** is `orchestrator-memory.md` in the app data dir (`src-tauri/src/orchestrator/memory.rs`): one ISO-date-prefixed entry per line, written ONLY through the explicit **`remember`** tool. Hard caps on write: 20 entries / 2000 chars — overflow drops the OLDEST (FIFO) and the append result says so. Read fresh into every new session. Commands: `orchestrator_memory_read` / `_append` / `_remove` (Settings `MemoryControls` lists entries with per-line delete).

## Vibe data layer (`src/lib/vibe/`)

**Store** (`session-store.ts`, standalone zustand): `sessions: Record<id, VibeSessionEntry>` + `order` (newest first) + `activeId` + in-memory `busy`. Each entry holds the `VibeSession` meta (incl. `projectId`/`agentName`/`spawnedBy`/`worktree` since schema v2 — see *Projects*) and the transcript **normalized by id** (`items: Record<id, VibeItem>` + `order`) — a streaming delta replaces ONLY its own item object (the t3code identity lesson; one row re-renders per delta batch). Per-turn `turnId`/`diff`/`plan`/`tokenUsage` are transient entry fields. **Caps**: 500 items/session, 30 sessions PER PROJECT (`capPerProject`, on create + hydrate); live command output 64 KB, persisted 8 KB. Persisted under `vibeSessions` (`version: 2`), debounced ~800 ms, flushed at quit, hydrated with per-entry degradation — `streaming` never restored, a persisted pending approval hydrates as `cancelled`, and every session runs through the project assignment (v1 persists migrate here).

**Controller** (`controller.ts`, outside React): typed `invoke` wrappers for the eight `vibe_session_*` commands + the single `vibe://session-event` listener. Delta batching ~80 ms. Public surface: `startSession` (resolves the owning project via `openProject` and picks a per-project agent name when unnamed), `resumeSession` (lazy — after a restart the first `sendMessage` resumes the persisted thread; a lost rollout warns and continues fresh), `sendMessage` (a busy session ignores), `interrupt`, `respondApproval` (optimistic), `setAccess`, `setModelEffort`, `closeSession` (flushes the accounting mirror), `focusSession` (switch to the session's project tab, select, leave the Conductor stage), `activateProject`/`activateProjectByIndex` (tab click / ⌘1–⌘9 — aligns the stage selection). Turn lifecycle mirrors into the Deck ticker; `token_usage` mirrors into the usage history (see *Usage & limits*). Busy sessions are quit blockers (`vibeBusyIds`). Dev hook `window.__vibe` (DEV-only; `seed.*` rendering seeds — sessions self-assign a project tab, backend-free — + `bench.snapshot()`).

**Pure helpers**: `ui.ts` (`diffStats`, signal-triad derivation, `totalTokens`, `commandExit`, `approvalCommand`), `names.ts` (agent-name pool, `pickAgentName`, `branchSlugForAgent`), `transcript.ts` (`renderSessionTranscript` for the orchestrator's read_transcript), `mention.ts` (`@session` prefix parser), `triage.ts` (`vibeTriageEntries`), `ui-store.ts` (transient dialog + `stageMode` state), `diff.ts` / `diff-worker.ts` / `diff-highlight.ts` (see below).

## Vibe UI shell (`src/components/vibe/`)

`VibeLayer` — always visible; there is no other view — = `SessionRail` (left, 264px) + `FocusStage` (flex-1) + the vibe dialogs. **Rail**: a pinned "Conductor" card (persona emoji/name, activates the Conductor stage, meta line model · effort · ctx %) then one card per session **of the active project** (Deck triage stays global) — name, project folder, `diffStats` counter, a signal-triad status line (`▸ working` + busy hairline / `⚑ needs you` amber / `✓ finished · Xm` ephemeral green / `· idle`), and a faint meta line. Cards select **primitives only** (the rail's id list is a joined-string signature) so a streaming delta on one session never re-renders the rest. **Stage**: `stageMode` ("conductor"|"session", default conductor) in `ui-store.ts` — conductor mode renders `ConductorStage` (the orchestrator chat + a composer with **`@session` routing**: a message starting with `@<name or id> ` goes straight to that session; plain completion popover while typing the token); session mode renders header (rename, model/effort chip, access chip, context gauge, Stop) → virtualized item feed → floating plan card → composer.

**New-Session dialog** (`NewVibeSessionDialog.tsx`, ⌘T): the folder defaults to the ACTIVE project (picking another folder opens/reuses that project's tab on create; recents from `discoverProjects()`), the agent name comes prefilled from the pool (🎲 reroll, globally collision-free) and stays editable, plus model/effort and the access toggle (`workspace` default). Running a session inside a git worktree = picking the worktree folder (or the WorktreePanel's open-in-session, which attaches the session to the main repo's project and stamps `worktree`).

**Item feed** (`ItemFeed.tsx`): **`virtua`'s `VList`** with stable `itemId` keys and memoized rows that each select ONLY their own item by id. Bottom-pinning is "stick to bottom while at bottom". Renderers: user bubble / assistant (finished messages render as markdown via `OrchestratorMarkdown`; a still-streaming message stays plaintext + caret) / collapsible command card / fileChange card (`DiffCard`) / amber approval card / warning / plan / webSearch.

**Diff stack** (`DiffCard.tsx` + `lib/vibe/diff*`): `@git-diff-view/react` (pinned `^0.1.6`) themed via CSS variables on plain DOM. Off-thread parse + highlight in a 2-worker pool (lowlight class tokens — not Shiki — LRU-cached, deduped by FNV `diffHash`); rows render plain first and only an OPEN row asks the worker. Large sets mount collapsed (>3 files or >160 lines); >512 KB files are byte-capped. `diff.ts` is the single source for +/- counting and diff shaping (incl. the raw-content → unified add-diff synthesis). The stage header carries a `Δ N files +X −Y` chip toggling a `TurnDiffPanel`.

**Approval composer takeover** (`Composer.tsx`): when the ACTIVE session has a pending approval the input is replaced by a `PENDING APPROVAL` panel — request preview + Decline · Cancel turn · Allow for session · Allow (⏎ = Allow, ⎋ = Decline, a LOCAL handler). A pending approval in a NON-active session shows a thin amber banner atop the stage that `focusSession`s it. Both select **primitive signatures only** (first-pending id, count) — never a fresh array — per the `useSyncExternalStore` invariant.

## Performance invariants

Heavy Tauri commands (usage parsing, git subprocesses, file walks) run async + `spawn_blocking` — sync commands execute on the main thread and may only do fast, non-blocking work. The Vibe UI rests on three measured properties: (a) the item feed is virtualized (`virtua`); (b) per-id primitive selectors keep re-renders scoped — a delta re-renders exactly its own row, never the rail; (c) the persist snapshot is cheap and debounced (~800 ms). Never regress these: a store selector that returns a fresh array, an un-memoized feed row, or a per-delta persist would each break one of them.

## Security

CSP is set in `tauri.conf.json` (no remote script/connect sources — the webview talks IPC only). The updater verifies minisign signatures against the pinned pubkey.
