## Transport & backends

`src/lib/transport.ts` exports the native Tauri backend. Backend capabilities are declared in `backend-types.ts` and implemented in `backend-tauri.ts`. Native-only features may skip the backend interface and use `@tauri-apps/api` directly (e.g. `lib/worktree.ts`, `lib/orchestrator/native.ts`, `lib/vibe/controller.ts`, `lib/projects/store.ts` ‚Äî the `canonicalize_path` dedupe resolver).

## Store & persistence

`src/store.ts` holds the shared app state. Persisted: usage history, quick notes, and app settings (`AppSettings`: last used folder, codex/git path overrides, auto-update, worktree repo registry `worktreeRepos`, orchestrator prefs ‚Äî codex model/effort defaults, scan roots, persona, the Phase-5 `autoReviewFinishedLanes` toggle ‚Äî the Phase-6 `reduceMotion` switch, wired to `data-motion="off"` on the root from App.tsx ‚Äî and the Phase-7 GitHub fields: the `githubIntegration` master toggle (default OFF), `githubAutoReviewPrs`, `githubSuggestPrOnFinish`, `githubWatchIntervalSec` and the `ghPath` binary override) ‚Äî edited in `SettingsDialog.tsx`, opened via the title-bar gear or `‚åò,` (Tauri store `swarmz.json`). Always mutate settings through `updateSettings` (merges + persists the whole object). In-memory only: palette/notes/dashboard/GitHub-panel flags, the worktree panel scan, `quitConfirm` (and the whole GitHub store ‚Äî see *GitHub integration*). Window size/position persists natively via `tauri-plugin-window-state`. Sessions live in their own store (`lib/vibe/session-store.ts`, key `vibeSessions`), orchestrator chats in `lib/orchestrator/chat-store.ts` (key `orchestratorChats`), project tabs in `lib/projects/store.ts` (key `projects`), Conductor timers in `lib/orchestrator/timers.ts` (key `conductorTimers`), autonomy budgets in `lib/orchestrator/autonomy.ts` (key `autonomyBudgets` ‚Äî serialize/hydrate stay pure in the module; store.ts wires the load/save + a debounced dirty sink via `registerAutonomyPersist`); all five are flushed by `flushAllPersists` at quit and hydrated from `hydrate()` in a FIXED order ‚Äî autonomy budgets FIRST (every autonomous delivery path gates on the projects store being hydrated, so restored budgets/breakers are guaranteed in place before any turn can pass the budget check) ‚Üí projects ‚Üí vibe sessions ‚Üí orchestrator chats ‚Üí conductor timers (the session migration assigns projectIds into the project store; the Phase-3 chat migration then resolves against both; timers fire missed ones and drop vanished-project entries ONLY when projects verifiably hydrated ‚Äî see *Conductor timers*).

The old pane-era keys (`grid`, `workspaces`, `workspacePresets`, `commandPresets`, `customCommands`, `profiles`) are DELETED once by the schema-v2 migration (`DEAD_STORE_KEYS` in `lib/schema-version.ts` ‚Üí `deleteStoreKeys` in the backend).

The store file itself has a crash net: `storefile.rs` runs in Tauri `setup` before the webview loads ‚Äî a healthy `swarmz.json` refreshes `swarmz.json.bak`, a corrupt one (torn write, disk-full) is moved aside as `swarmz.json.corrupt-<ts>` and the backup restored, so the plugin's silent treat-unreadable-as-empty behavior can no longer factory-reset the app.

`swarmz.json` also carries a top-level **`schemaVersion`** key (a plain number, currently `2`) ‚Äî the migration anchor, deliberately WITHOUT a migration framework. The pure hooks live in `src/lib/schema-version.ts` (unit-tested): `normalizeSchemaVersion` (valid stored versions ‚Äî even NEWER ones ‚Äî are left untouched, so a downgraded build never clobbers a newer store's stamp) and `planSchemaMigration`, which `hydrate()` runs first thing: a raw version `>= 2` plans nothing; `1` or missing/invalid (a possibly pre-rebuild store) plans the one-time dead-key cleanup + the `2` stamp. The plan is idempotent. The actual VALUE migration (sessions ‚Üí projects) does not run here ‚Äî it lives tolerantly in the vibe hydrator (see *Projects*). The `storefile.rs` rescue path is version-agnostic and stays untouched.

## Projects & swarm data model (Phase 2)

**Entity** (`Project` in `types.ts`, persisted key `projects` via `lib/projects/store.ts`): `{ id, dir, name, order, lastActiveAt, createdAt, closedAt? }`. `dir` is the CANONICAL folder path ‚Äî `openProject` resolves it through the Rust `canonicalize_path` command (symlinks, `/var` aliasing; falls back to the normalized input when the folder is gone) and dedupes on it, so the same folder always maps to the same project. `name` is the folder basename with a lowest-free numeric suffix on collision ("api", "api 2"). `order` is the tab position among OPEN projects; `activeProjectId` persists. Pure logic (dedupe key, naming, reorder patches, migration) lives in `lib/projects/core.ts` (unit-tested); the zustand store around it debounces persists (~500 ms, flushed by `flushAllPersists`).

**Open vs closed**: closing a project TAB only stamps `closedAt` ‚Äî the entity stays persisted and its sessions stay in the session store; reopening the same folder (picker, recents, a triage jump) flips the SAME project open again and the sessions reappear. `setActiveProject` auto-reopens.

**Sessions belong to projects** (schema v2): `VibeSession` carries `projectId`, `agentName` (the generated identity ‚Äî immutable, branch names derive from it in Phase 4), `spawnedBy: "user" | "conductor"` and `worktree: { root, branch, shared } | null` (carried now, filled by the WorktreePanel flow and Phase 4's tools). **Hydrate migration**: every hydrated session runs through `useProjects.adoptSessions` (pure core: `assignSessionsToProjects`) ‚Äî a valid `projectId` passes through; pre-v2 sessions and sessions whose project record was lost get projects derived from their `projectDir`s (deduped, matched against open AND closed projects, created OPEN otherwise); `agentName` defaults to the old display name, `spawnedBy` to "user". The stamped result persists immediately (`vibeSessions` writes `version: 2`). The session cap is **30 per project** (`MAX_SESSIONS_PER_PROJECT`, enforced on create and on hydrate).

**Agent names** (`lib/vibe/names.ts`, pure + unit-tested): a 130+ pool of short international first names. `pickAgentName(taken, rng?)` picks collision-free (case-insensitive; per project in the controller, globally strict in the New-Session dialog), suffixing "Maya 2" once the pool is exhausted; `branchSlugForAgent(name, task?)` ‚Üí `swarm/maya-checkout` (NFKD diacritics-stripped ASCII slugs, task fragment capped at a word boundary). The dialog prefills a generated name with a üé≤ reroll; `spawn_agents` sessions without a `name` get a pool name too.

**Scoping**: the Fleet grid shows only the ACTIVE project's sessions (primitive status-signature selector; no active project = show all, a degenerate pre-migration state). The Deck triage/counters stay GLOBAL across projects ‚Äî `focusSession` switches to the session's project tab first (reopening a closed one), then selects it; `activateProject` (tab click, ‚åò1‚Äì‚åò9, palette) aligns the stage: selection restores the project's REMEMBERED session (`activeIdByProject`, persisted; stamped by `setActive`/`createSession`, validated on hydrate), else the project's newest; a session-less project lands on the Conductor. The orchestrator's `spawn_agents` attaches sessions to the Conductor's own project when `cwd` is omitted/equal to the project folder; an explicit foreign `cwd` resolves its own project (NOT the active tab) and never steals the active tab (`activate: false` for conductor spawns).

**TitleBar** (48 px, Vibe v3): the ‚åòB Conductor-sidebar toggle, the hexagon brand mark + "SwarmZ" wordmark + `/`, then one tab per open project with a live dot (busy = accent `zpulse`, idle = faint), `busy/total` count, `‚öën` needs-you badge, close X (middle-click too), double-click rename, drag reorder; the + button opens a folder picker / discovery-recents dropdown ("Open project" also lives in the ‚åòK palette, along with tab switching). Right side: the amber needs-you pill (`N need you`, click = ‚åò‚áßA jump), the ‚åòK button, update pill, usage/notes/worktrees/settings buttons and the accent "New agent" CTA. Tabs are plain `<div>`s and opt out of the header drag region via `data-tauri-drag-region="false"`. **Closing with busy sessions** never blocks and never stops anything, but routes through `requestCloseProject` (vibe controller): N busy sessions raise the `CloseProjectConfirm` dialog (busy count + "they keep running in the background") before the tab hides.

## Quick notes

**Quick notes** (‚åòN, title-bar note button, `QuickNotesPanel.tsx`): a right-side slide-in drawer holding lightweight checklists ‚Äî one **global** list plus one **per project folder** (store key `quickNotes` ‚Üí `QuickNotesData`, folder lists keyed by repo root). The drawer opens in the scope of the active session's `projectDir` (fallback Global) and shows a chip per scope. Items (`NoteItem`) check off, edit inline, reorder by dragging the grip, and can toggle to plain text lines (`plain`); "Clear done" drops every checked item, emptied folder lists drop their key. Every mutation persists immediately. The drawer is a `role="dialog"` so the global-shortcut guard kicks in; ‚åòN closing it is special-cased in the dialog branch of the `App.tsx` keydown handler.

## Usage & limits

Codex usage is parsed from `~/.codex/sessions/**/*.jsonl` by `src-tauri/src/codex_usage.rs` (which also owns the shared usage types since the Claude parser was removed), using `session_meta`, `turn_context`, `event_msg/token_count`, `task_started` and `task_complete` rollout events for session id, cwd, model, tokens, context, limits and activity. The usage commands (`usage_for_dir`, `usage_for_session`, `usage_totals`) are async + `spawn_blocking` and codex-only; they currently have no frontend caller ‚Äî live session accounting runs frontend-side instead (see below). Codex ChatGPT-plan rollouts do not include reliable USD cost, so cost stays zero (there is no pricing table anymore).

Codex limit windows are **account-level**: `codex_account_limits` finds the newest `token_count.rate_limits` event across ALL of `~/.codex/sessions/**/*.jsonl` ‚Äî files are iterated newest-mtime-first, each only via a bounded 256 KiB tail read, stopping at the first hit ‚Äî and is polled every 60 s by `src/lib/limits.ts` (paused while hidden) into the Deck meters. `limits: null` = no data ever seen (the Deck shows a quiet `CX ‚Äî` placeholder); data older than 1 h gets an "as of HH:MM" tooltip suffix.

The **usage dashboard** (`UsageDashboard.tsx`, title-bar chart button) shows the persisted all-time history (store key `usageHistory`, Codex entries; the pre-rebuild Claude/shell entries are filtered from view). **Session accounting** feeds it live: the vibe controller mirrors each session's `token_usage` events into `recordUsageHistory` (keyed `codex:<session id>`, `runtime:"codex"`, `by_model` from the session's model override or "codex default") ‚Äî debounced ~2 s per session, flushed once more on close. The thread's `total` bucket is cumulative, so each write REPLACES the entry; cached input counts as cache reads and is subtracted from fresh input; `message_count` = user turns in the transcript; cost stays 0 (no pricing).

## Deck & fleet events

The **Deck v2** (`components/Deck.tsx`) is the permanent 32 px status bar under the fleet. Title bar = place/navigation, Deck = system status (see DESIGN.md) ‚Äî all `text-11 font-mono tabular-nums`, it never grows. Anatomy left‚Üíright:

1. **Fleet counters** (GLOBAL across all projects ‚Äî the Deck aggregates the whole swarm, unlike the project-scoped grid): `‚ñ∏ working` (accent-tinted while >0) ¬∑ `‚öë needs-you` ¬∑ `‚úì finished` (last 5 min) ¬∑ `¬∑ idle`, derived from ONE primitive status signature + a 30 s decay tick. The ‚öë counter IS the triage queue: at N>0 it turns amber and opens the popover listing every session with a pending approval, oldest waiting first (`vibeTriageEntries` in `lib/vibe/triage.ts`); arrow keys rove, Enter jumps (`focusSession` ‚Äî which switches to the session's project tab first). ‚åò‚áßA jumps to the oldest directly.
2. **Event ticker**: the last 3 fleet events inline (newest last), each clickable to jump while its session still exists; a history icon opens a popover with the last ~30.
3. **Equalizer**: five staggered `animate-zeq` accent bars ‚Äî visible only while any agent or Conductor works (the ambient "fleet is alive" signal).
4. **Conductor dot**: `conductor` + dot ‚Äî AGGREGATED across all projects' Conductors: `--err` when the ACTIVE project's autonomy circuit breaker is latched (read via `subscribeAutonomy`/`autonomyTripped` through `useSyncExternalStore` ‚Äî a primitive-boolean snapshot; tooltip says autonomy is paused until the human writes), `--attn` when any chat has undelivered pings, accent (`zpulse`) while any turn runs, faint otherwise; click = `showConductor` (fleet + sidebar of the active project).
5. **PR indicator** (Phase 7, only while the GitHub integration is ON): `‚áÖ N` open PRs of the ACTIVE project, err-tinted `√ón` while any PR's checks fail ‚Äî a primitive `deckPrSignature` selector string, click opens the GitHub panel.
6. **Meters**: the account-level Codex meters (`cx 5h`, `cx wk`) with full-word tooltips, then the clock (HH:MM, 30 s tick).

**Toasts** (`components/Toasts.tsx`, top-right, z-70): transient event cards fed from the same fleet feed ‚Äî finished (ok) / needs-approval (attn) / agent-started (accent ‚ú¶), `animate-ztoast`, auto-dismiss 5 s, click jumps to the agent. Initialized past the current feed so a mount never replays history; orch-prompt/exited events stay ticker-only.

**Fleet event feed** (`src/lib/events.ts`, a standalone zustand store): `FleetEvent { id, at, kind, sessionId, sessionName, label?, url? }` with kinds `finished` / `waiting` / `orch_prompt` / `created` / `exited` / `pr` (Phase 7 ‚Äî sessionless: `sessionId: ""`, a pre-built `label` and the PR `url` the chip opens; toasts skip it), capped at 50, **in-memory only by design**. Producers: the vibe controller (`approval_request`‚Üíwaiting, `turn_completed`‚Üífinished, `turn_failed`‚Üíexited), `notePromptDelivered` in `orchestrator/executors.ts` (`orch_prompt`) and the GitHub controller (`pr` per watcher change note). Repeated `waiting` for the same session within 3 s is deduped in the feed itself (aligned with the ping flap guard).

## Git status

`git_info` (`src-tauri/src/git.rs`) is a strictly read-only snapshot (repo, branch, ¬±lines, untracked, `origin` URL) that shells out to the git binary (prefers the Settings `gitPath` override, else `/usr/bin/git`; every subprocess runs through `output_with_timeout`). There is no polling anymore ‚Äî the orchestrator's `git_status` tool and the worktree flows call it on demand.

## Git worktrees

`src-tauri/src/worktree.rs` + `src/lib/worktree.ts`, native-only direct `invoke` ‚Äî deliberately *writing* git, unlike the read-only git.rs. `worktree_add` creates `<repo>/.worktrees/<slug>` via `git worktree add -b` off the current HEAD (main root resolved via `--git-common-dir`), excludes `.worktrees/` via `.git/info/exclude` (never the tracked `.gitignore`), and copies every untracked file (`ls-files --others`, incl. gitignored `.env`s) minus the heavyweight `SKIP_DIRS`. The human way to run a session inside a worktree is picking the worktree folder in the New-Session dialog (or the WorktreePanel's open-in-session); since Phase 4 the Conductor creates and manages worktrees itself through its tools (`spawn_agents` placement "new"/"shared:<agent>", `create_worktree`, `assign_worktree` ‚Äî which retunes a live session via `thread/settings/update {cwd}` ‚Äî, `worktree_status`, `cleanup_worktree`), stamping every session's `worktree` meta (`{root, branch, shared}`).

**Management panel** (`WorktreePanel.tsx`): a title-bar `FolderGit2` button, visible only while ‚â•1 worktree exists, lists all SwarmZ worktrees grouped per repo (dirty/ahead state, refreshed on open) with open-in-session (the session runs in the worktree but belongs to the MAIN repo's project tab and carries the `worktree` meta) or jump-to-session, reveal-in-Finder, per-row finish/delete and a "Clean up safe" bulk action for clean unattached worktrees (a live session's `projectDir` counts as attached). Risky rows (uncommitted changes or local-only commits) require a second delete click; deletion re-checks status at execution time ‚Äî on doubt the worktree is kept. **`worktree::remove` takes a `force` flag**: user-confirmed panel deletions force; every SILENT path ("Clean up safe", the close-flow's clean branch, the Conductor's `cleanup_worktree`, the spawn rollback) passes `force: false` ‚Äî Rust then re-checks dirty/ahead INSIDE the removal call (an uncomputable ahead count ‚Äî `ahead_unknown` in `WorktreeStatus`/`WorktreeEntry` ‚Äî refuses like dirt; a git error is never silently 0) and runs `git worktree remove` WITHOUT `--force`, so git itself refuses work that appeared after the caller's check (the copied gitignored env files don't block a non-force removal ‚Äî probe-verified); on the missing-folder path the gated branch `-D` additionally refuses when the branch still holds unreachable commits. Scanned repo roots persist in `settings.worktreeRepos` (pruned automatically once a repo has no worktrees left; `worktree_list` returns a `WorktreeScan {entries, scanned}` and only roots that actually **scanned** empty are pruned ‚Äî an unmounted volume or broken git override never loses orphans). A hand-deleted worktree folder still reports its branch's unreachable commits as `ahead`. The keep-vs-delete dialog (`CloseWorktreeDialog.tsx` + the store's `requestCloseWorktree`/`resolveCloseWorktree`) is the HUMAN escalation path; the Conductor's `cleanup_worktree` never dialogs ‚Äî it re-checks status at execution time and REFUSES on dirty / local-only commits / unknown state / live occupants (forcing stays a human decision).

## Status reporting & quit guard

Quit guard (`src/lib/quit.ts`, started in `App.tsx`; pure blocker logic in `quit-core.ts`, unit-tested): closing the app raises `QuitConfirmDialog.tsx` when quitting would genuinely interrupt work ‚Äî the hard blockers are busy sessions (`vibeBusyIds`), busy Conductors (`busyConductorProjectNames`), a Conductor timer MID-FIRE (`claimedTimers` ‚Äî its durable `firedAt` claim is stamped, so a quit now would hydrate-drop it; merely PENDING timers stay info-only), in-flight gh/git writes (`github_writes_in_flight`; a FAILED counter query reports `-1` = unknown and confirms, fail closed) and the in-flight background ops from `lib/inflight.ts` (detached codex reviews ‚Äî manual, `review_agent`, auto-review ‚Äî and worktree add/remove, counted via `beginInflight` around `reviewSession`/`addWorktree`/`removeWorktree`, since none of those hold a busy flag). Native covers both paths ‚Äî window close via `onCloseRequested` (every first close is preventDefault'ed, all debounced persists are flushed (`flushAllPersists`), then `close()` is re-issued past a module-level `confirmed` flag so the window-state plugin still saves), and ‚åòQ/menu quit via Rust (`lib.rs` run-callback: `ExitRequested` with `code.is_some()` ‚Üí `prevent_exit()` + emit `app://quit-requested`; `prevent_exit` is a tauri-built-in no-op for the updater's `RESTART_EXIT_CODE`, and code `None` = last window closed passes, which is how a confirmed quit actually exits).

## Orchestrator sensing

Read-only sensing. `src-tauri/src/transcript.rs`: `transcript_read` returns actual message content from codex rollout files (found by the id suffix in the filename, head-scan fallback) ‚Äî the `event_msg` `user_message`/`agent_message` chat stream preferred, `response_item` messages (instruction wrappers filtered) as fallback. Options: `tail_messages` (default 20), `max_bytes` (default 1 MiB ‚Äî huge files are seek-tailed to the first complete line, never fully read) and `include_first_user_message` (from a separate capped head read). Texts are char-truncated (umlaut-safe) with an `‚Ä¶` marker. Note: the orchestrator's `read_agent` TOOL renders LIVE sessions from the vibe store (`renderSessionTranscript`) ‚Äî `transcript_read` reads rollout files on disk and currently only serves the dev hook. `project_docs` reads README.md/AGENTS.md/CLAUDE.md of a root (worktree paths collapse to the main repo; ~8 KB/file, ~20 KB total; missing files omitted). `src-tauri/src/projects.rs` (codex-only): `discover_projects` merges project folders from Codex `session_meta` cwds, frontend-known folders (project tabs ‚Äî open and closed ‚Äî, live session dirs, quick-note folder keys, `worktreeRepos`, `lastCwd` ‚Äî collected in `lib/orchestrator/native.ts`) and a shallow scan (depth ‚â§ 2) for `.git` (dir or file); worktree paths collapse onto the repo root, entries dedupe and sort by last activity. It also feeds the "open project" recents (TitleBar + dialog). Both commands are async + `spawn_blocking`, uncached.

Frontend: `src/lib/orchestrator/` ‚Äî `snapshot.ts` (pure `sessionSnapshot`/`sessionCounts`/`fleetSummaryLine` over the vibe store's entries + busy map), `native.ts` (direct-`invoke` wrappers), `types.ts` (payload + tool-bus types), and `dev.ts`, a dev-only smoke-test hook exposing `window.__orch`.

## Orchestrator tool bus

**The registry is the single source of truth**: `src-tauri/src/orchestrator/registry.rs` defines every tool as `{ name, description, parameters (JSON Schema), timeout_ms }`; `tool_definitions()` serializes the catalog (**31 tools** ‚Äî 24 since Phase 4 + the 7 Phase-7 GitHub tools, project-scoped) for the Codex app-server (`dynamicTools`). Per-tool timeouts follow the work behind the tool: 15 s default, `prompt_agent` 30 s, worktree ops 150 s, `spawn_agents` 600 s (the bus timeout can't CANCEL the webview batch ‚Äî a too-tight deadline would leave the batch running while the model retries into duplicate agents, so it must outlast the 8-agent worst case incl. shared-lane waits; the batch's name reservation additionally fail-fasts a same-names retry), `review_agent` 600 s, GitHub reads 90 s (network ‚Äî must outlast the executor's CHAINED gh subprocess deadlines: `github_status` runs auth + repo + PR list, `read_pr` view + diff; the bus timeout can't cancel the webview executor), `create_pr` 300 s (a push rides inside, plus worktree resolve + repo info + `gh pr create`), `review_pr` 600 s. The `orchestrator_tools(persona, project)` command returns `{ instructions, tools }` ‚Äî the tools plus the compiled system instructions (`persona::build_instructions`). The frontend mirrors the *names* only (`ORCHESTRATOR_TOOL_NAMES` in `types.ts`, executor keys). Argument validation is a deliberately hand-rolled subset (required props, basic `type` tags, `enum`, array `minItems`/`maxItems`/`items`, applied recursively).

**Roundtrip bus** (`orchestrator/bus.rs` + `src/lib/orchestrator/bus.ts` + `executors.ts`): tools are defined in Rust but *execute in the webview*, where the Zustand stores live. `run_tool` validates against the registry, parks a tokio oneshot in a pending map, emits `orchestrator://tool-request` `{ id, tool, args, chat_id, project_id }` (`chat_id` = the triggering backend chat, `project_id` = the Conductor instance's project ‚Äî both null for dev/internal calls), and awaits the response with the tool's timeout (per the registry's per-tool budgets ‚Äî `spawn_agents` creates worktrees and spawns codex processes, `review_agent` runs a whole review turn). The webview bus (`startOrchestratorBus()`, registered once from `App.tsx`) looks up the executor, runs it with the `{chatId, projectId}` context and **always** answers via `orchestrator_tool_response(id, ok, payload)`. The bus core is tauri-free and unit-tested. `orchestrator_run_tool` is the dev-hook surface (`await __orch.tool("fleet_snapshot", {})` ‚Äî unscoped).

**Project scoping** (Phase 3): `requireSession` resolves `agent` arguments ONLY among the Conductor's project's sessions ‚Äî by raw id, or by session/agent NAME (optionally `@`-prefixed, case-insensitive) when unique in the project (ambiguity errors with the candidates; unknown ids error with the project's valid list). `fleet_snapshot` returns `{project, summary, sessions, worktrees, timers}` with only that project's data; the summary line Rust prepends to every turn uses the same scope. An unscoped call (dev hook) sees everything; project-scoped tools (spawn/worktrees/timers/plans) error without a project.

**The 31 tools** (executors in `src/lib/orchestrator/executors.ts`, catalog frozen by a registry test):

- **Sensing** ‚Äî `fleet_snapshot` **v2** (project header + scoped `sessionSnapshot` with exact status working/idle/pending-approval, per-session worktree `{branch, shared}`, `worktrees` occupancy (who works where, pure `worktreeOccupancy`), pending `ApprovalBrief`s with their routing class, and the project's timers) ¬∑ `read_agent` (live session items rendered compactly via `renderSessionTranscript`) ¬∑ `read_project_docs` (exactly one of `agent`/`path`) ¬∑ `read_notes` ¬∑ `git_status` (exactly one of `agent`/`path` ‚Äî worktree cwds work) ¬∑ `list_projects` (`scan_roots` omitted = the user's `orchestratorScanRoots`).
- **Agents** ‚Äî `spawn_agents` (1‚Äì8 agents, each `{task, worktree, model?, effort?, access?, name?, expect_report?}`; `expect_report: true` (Phase 5) constrains the task turn's final message to the agent-report `outputSchema` and appends the report suffix to the task ‚Äî see *Autonomy loop*; placement `"new"` = a fresh worktree on a `swarm/<name>-<task>` branch (collision-suffixed, env copied, repo registered for the panel), `"shared:<agent>"` = join that agent's worktree (both marked `shared`; may reference an agent spawned earlier in the SAME batch), `"none"` = the project folder (analysis/read tasks); names auto from the pool with a CASE-INSENSITIVE batch reservation (an explicit name colliding with a live session or an earlier batch entry fails ITS spec ‚Äî duplicate names would make `shared:<name>` and every name-based tool ambiguous), effort defaults to `medium`, access to `workspace`, `spawnedBy:"conductor"`; sequential creation, per-entry errors never abort the batch, a FRESH worktree whose session fails to start is rolled back (gated non-force removal; an unremovable one is named in the error for `cleanup_worktree`); initial tasks kick off in parallel ACROSS lanes but strictly serialized WITHIN a shared worktree ‚Äî the next same-lane turn starts only after the previous agent's turn finished (`waitForSessionIdle`, bounded ~8 min; a still-busy lane hands the remaining tasks back as per-agent warnings instead of starting a second writer)) ¬∑ `prompt_agent` (idle ‚Üí next turn; **BUSY ‚Üí `turn/steer`** ‚Äî the running turn absorbs the text; the result reports `mode: "turn"|"steered"`; the 2 s double-prompt guard stays, the old busy-refusal is gone; `expect_report?` applies only when the prompt starts a FRESH turn ‚Äî a steered running turn keeps its own format) ¬∑ `interrupt_agent` ¬∑ `close_agent` (worktree stays ‚Äî the result says to clean it separately) ¬∑ `set_agent_config` (model/effort/access mid-session; `""` clears an override; applies next turn) ¬∑ `review_agent` (native detached `review/start` over the agent's work ‚Äî target `uncommitted` (default) / `branch:<base>` / `commit:<sha>`; returns the prioritized findings report) ¬∑ `decide_approval` (ROUTINE approvals only ‚Äî enforced SERVER-SIDE via `respond_approval(require_routine: true)`, the frontend class check is just the fast lane; - **Worktrees** ‚Äî `create_worktree` (fresh worktree + branch, env copy) ¬∑ `assign_worktree` (re-home an agent: refuses BUSY sessions, backend `thread/settings/update {cwd}` FIRST and the store meta only on its ack; co-tenants flip `shared`) ¬∑ `worktree_status` (project worktrees with dirty/ahead ‚Äî an uncomputable count surfaces as `ahead_unknown` ‚Äî + occupants) ¬∑ `cleanup_worktree` (**safe-gated**: refuses live occupants, re-checks dirty/ahead at execution time and treats `ahead_unknown` as "may hold work"). The removal itself goes through `worktree::remove(force: false)` ‚Äî Rust re-checks dirty/ahead/`ahead_unknown` inside the call and runs `git worktree remove` WITHOUT `--force`, so even work appearing after the checks refuses (never forces; forcing stays a human decision).

- **Plans** ‚Äî `write_plan`/`list_plans`/`read_plan` ‚Üí the Rust `plans.rs` commands (see *Conductor plan documents*).
- **GitHub** (Phase 7 ‚Äî ALL gated on the Settings master toggle; see *GitHub integration* for the gating decision) ‚Äî `github_status` (the one always-answering tool: reports a DISABLED integration instead of erroring; otherwise auth digest + repo + open PRs with watched flags) ¬∑ `list_prs` ¬∑ `read_pr` (detail incl. reviews/files/checks + the line-capped unified diff) ¬∑ `create_pr` (exactly one of agent/branch ‚Üí the worktree checkout; Rust pushes PLAIN ‚Äî never force, never the default branch ‚Äî then `gh pr create`; outward-facing, doctrinally user-order-only) ¬∑ `review_pr` (the PR's head branch must live in a project worktree ‚Äî explicit agent or unique branch match; runs the detached codex review against `branch:<base>`; `post:true` submits it as a `gh pr review` with the chosen verdict) ¬∑ `comment_pr` ¬∑ `watch_pr` (verifies the PR exists, registers it in the github store; watcher changes then wake the Conductor ‚Äî one app run, timers are the durable alternative).

## Conductor plan documents (`src-tauri/src/plans.rs`)

The Conductor's ONE sanctioned write surface ‚Äî the precisely-worded exception to the "never edit files" guardrail. `conductor_plan_write(project_dir, title, markdown)` stores Markdown under `<project>/.swarmz/plans/<slug>.md`; the slug is DERIVED in Rust from the title (lowercase `a-z0-9-`, ‚â§64 chars, no dots/slashes) and `read`/`list` re-validate it, so a hostile title or slug can never escape the folder (unit-tested with `../` payloads). The project dir comes from the trusted chat context, never from the model. Content is capped at 256 KiB, titles at 200 chars, an H1 title is ensured, same title replaces; a DIFFERENT title colliding on the same slug gets a stable hash suffix instead of silently overwriting the other document. `.swarmz/` is excluded from git via the repo-local `.git/info/exclude` (best-effort, same mechanism as the worktrees ‚Äî never the tracked `.gitignore`), so plans don't dirty the user's `git status`. Foreign files in the folder are never listed/served.

**No-follow confinement** (Phase-4 security pass ‚Äî an AGENT can plant symlinks in its workspace, e.g. `.swarmz/plans/x.md ‚Üí ../../src/main.rs` or `.swarmz/plans ‚Üí /tmp`): the `.swarmz` and `plans` path components refuse when they are symlinks; writes refuse a non-regular file at the target and land ATOMICALLY via a fresh temp file + rename (never writing THROUGH anything); reads open with `O_NOFOLLOW` after a `symlink_metadata` regular-file check (a FIFO/device with a safe name can't hang the open) and are BOUNDED (oversized foreign files error instead of flooding memory; the title scan reads ‚â§4 KiB); `list` uses the no-follow `DirEntry::file_type()`, so a safe-named symlink to a host file is skipped, never leaked. All of it unit-tested with planted symlinks, a symlinked plans dir and a FIFO.

## Conductor timers (`src/lib/orchestrator/timers.ts` + `timers-core.ts`)

Follow-up timers the Conductor sets itself (`set_timer`). A standalone zustand store persisted under `conductorTimers` (debounced, flushed at quit); pure logic (arg resolution `resolveFireAt` with exactly-one-of `delay_seconds`/`at_iso`, sanitizing incl. the `firedAt` claim, due-splitting, rendering) lives in `timers-core.ts`, unit-tested. **Semantics**: timers persist across restarts but only FIRE while the app runs; wakeups are `setTimeout`s (clamped + re-checked for >24-day spans); on hydrate, DUE timers fire immediately, marked as missed. Firing delivers an **autonomous Conductor turn** in the timer's project (visible `‚è∞ Timer fired:` system marker + a `[timer fired‚Ä¶]` wire text) via the controller's `deliverTimerTurn` (which passes the autonomy budget, and since Phase 5 runs INSIDE the project's autonomous serialization chain ‚Äî `runExclusiveAutonomous` ‚Äî so a timer turn never interleaves with an event-triggered turn in the same chat). The delivery + notice functions are REGISTERED from App.tsx (`registerTimerDelivery`/`registerTimerNotice`) instead of imported ‚Äî that keeps the import graph acyclic (store ‚Üí timers ‚Üí controller ‚Üí store would otherwise be a runtime cycle).

**Safety rails** (Phase-4 security pass, hardened in Phase 5):

- **Idempotency / at-most-once**: an in-memory `firing` claim guards against double wakeups within a run, and a durable **`firedAt` stamp is persisted (flushed, not debounced) IMMEDIATELY BEFORE the dispatch, INSIDE the serialization chain** ‚Äî `fireTimer` hands `deliverTimerTurn` a `claim` callback that the delivery invokes only once the project's autonomous chain reached the timer (never before the possibly-minutes-long chain wait: a quit while the timer queues behind a long turn/review must re-fire it as missed, not hydrate-drop a never-delivered timer). A crash between the claimed dispatch and the timer's removal must not re-deliver the same `tm-*` on the next launch. Hydrate DROPS claimed timers with a visible chat notice ("may have already fired‚Ä¶ dropped to avoid a double delivery") instead of re-firing them; a "retry" outcome clears the stamp again (nothing ran).
- **Cancel re-check**: the `claim` callback re-checks that the timer still EXISTS ‚Äî a timer cancelled while queued (e.g. `cancel_timer` called by the very autonomous turn the timer waited behind) aborts as "drop" instead of firing anyway after the turn ends.
- **Honest delivery**: `dispatchTurn` propagates failure ‚Äî a turn that never started (codex unavailable, spawn failure) answers "retry", never "delivered"; the timer stays persisted.
- **Bounded retries**: "retry" (busy chat, open circuit breaker, failed dispatch) re-arms every 30 s AT MOST 20 times (~10 min); after that the timer is removed and **expires visibly** (`‚è∞ Timer expired undelivered‚Ä¶` system message via the notice sink). A quit mid-retry re-fires it as missed next launch.
- **Hydration honesty**: `loadConductorTimers` RETHROWS store read errors (null = only a genuinely missing key) ‚Äî an unreadable store keeps the in-memory state untouched and persists nothing; the vanished-project filter applies ONLY when the projects store verifiably `hydrated` (a failed project hydration must never bulk-erase timers ‚Äî the Phase-3 chat-readiness lesson), and `deliverTimerTurn` answers "retry" while projects are not hydrated.

## Approval routing (Phase 4)

Every approval request a session raises is CLASSIFIED in Rust (`codex/sessions.rs::classify_approval`, pure + unit-tested against an adversarial payload suite) and the `approval_request` event carries `escalation: "routine" | "destructive"` (stored on the approval item, hydrated tolerantly ‚Äî missing = destructive).

The classifier is **FAIL-CLOSED by construction** ("im Zweifel IMMER Mensch" ‚Äî a false destructive costs one human click, a false routine is unacceptable). **Commands**: a strict quote-aware tokenizer rejects the whole command on ANY shell metasyntax outside single quotes (`| & ; < > ( ) { } $ \` \\`, newlines, unterminated quotes ‚Äî no best-effort segment parsing; `ls || find . -delete`, `echo $(‚Ä¶)`, `cat /dev/null > x` are all human) and on any backslash escape (`r\m -rf` can't hide a head). Routine then requires the (bare or `/bin`//`/usr/bin`-prefixed ‚Äî never `./x` or other local paths) head to be on the tiny read-only allowlist ‚Äî `ls cat head tail grep rg wc pwd which echo` ‚Äî or one of the gated forms: `git` with a read-only subcommand (`status diff log show branch remote rev-parse describe blame`) AND read-only flags only (no `git -c`/`-C` global options, no `--output`/`-o`/`--ext-diff`, `branch` only in pure list forms, `remote` only bare/`-v`/`show`), `cargo test|check|clippy|build`, `pnpm test|run test`, `tsc`, `node --version`. NO interpreters (`python*`, `node -e/script`, `ruby`, shells as targets), NO `env` (rejected wholesale, never stripped ‚Äî `env` can rewrite PATH/DYLD and subvert an allowlisted head), NO `find`/`xargs`/`sed`/`tee`/`cp`/`mv`/`touch`/`mkdir`/`chmod`. Shell wrappers unwrap ONLY as exactly `sh|bash|zsh -c|-lc '<one script arg>'` (recursively classified, depth-capped); anything else is classified by its OUTER head. **Phase 7 adds `gh`** (`gh_is_routine`, same fail-closed construction): read-only queries of the current repo are routine unconditionally (`gh pr list|view|diff|checks|status`, `gh repo view`, `gh auth status`, `gh issue|run list|view` ‚Äî every `-`-flag must be on a tiny allowlist (`--json`, `--jq`, `--limit`, `--state`, `--comments`), parsed with VALUE-CONSUMING flag handling (`gh_parse_args`: a `--flag value` spelling consumes its value, so flag values are never mistaken for positionals and can never smuggle an action flag); `--web`, `-R/--repo` retargeting and `--body-file` (file exfiltration into a public comment) are NOT on it); POSITIONAL SELECTORS are restricted to ONE bare PR/issue/run NUMBER or absence (`gh_selector_ok`) ‚Äî gh also accepts `<url>|<branch>` selectors, and a URL retargets the command at an arbitrary FOREIGN repo (`gh pr comment <url> --body ‚Ä¶` would post, `gh pr review <url> --approve` would approve, anywhere on the host), so URLs, branch names, extra positionals and any `gh repo view <owner/repo>` positional are destructive; `gh pr comment --body ‚Ä¶` and `gh pr review` with EXACTLY ONE of `--approve|--request-changes|--comment` are routine ONLY while the GitHub integration is enabled (`classify_approval(kind, params, cwd, gh_writes)` ‚Äî the call site reads the Rust-side `INTEGRATION_ENABLED` flag, never anything frontend-claimed); EVERYTHING else gh ‚Äî `merge`, `close`, `create`, `ready`, `edit`, any `gh api` (arbitrary REST/GraphQL, any method), repo/release/secret/workflow/alias mutations, unknown subcommands or flags ‚Äî is destructive, frozen by an adversarial suite in both toggle states. A routine class that exists ONLY because the toggle was on at arrival is stored as `gh_write_gated` next to the Responder, and the strict respond path (`routine_gate`) RE-CHECKS the LIVE flag ‚Äî an approval pending across a disable downgrades back to human-only instead of releasing a stale gh write. Sensitive substrings (`.env`, `id_rsa`, `.ssh`, `credentials`, `secret`, `token`, `/etc/`, ‚Ä¶) force human regardless of head, and a request `cwd` outside the session's assigned directory (canonicalized `path_within`) is destructive even for read-only heads. **fileChanges**: routine only when EVERY change is a pure create (`add` of a not-yet-existing path) or an in-place edit (`update` without a move/rename), on a non-sensitive path STRICTLY inside the session's canonicalized cwd ‚Äî the containment check resolves symlinks (leaf AND ancestors; a link out of the tree classifies by its TARGET), rejects `..` lexically before any fs lookup, and fails closed on broken links/IO doubt; deletes, renames, overwriting adds and unknown kinds are always human.

**Server-anchored enforcement**: the class is stored in Rust NEXT to the blocked approval `Responder` (`PendingApproval {responder, escalation}`). The Conductor's `decide_approval` answers through `vibe_session_respond_approval(require_routine: true)` ‚Äî the class check and the responder removal happen atomically under the session lock, so a destructive approval can never be answered through the Conductor path no matter what the frontend claims; the frontend's own `escalation` field is a courtesy copy for UI + a fast pre-check, never the authority. The strict path marks nothing optimistically ‚Äî the item status commits only after Rust's ack, and errors (already resolved by the human, destructive, dead session) propagate into the tool result.

**Routing**: the activity watcher (orchestrator controller) escalates a NEW pending ROUTINE approval as an autonomous Conductor turn (`‚öë Approval escalated` marker + a wire text asking for `decide_approval`; the request summary is flattened to a single quoted line ‚Äî untrusted data, see *Autonomy loop*), **gated on conductor-involvement like every loop event** ‚Äî a purely human-created, never-Conductor-touched session never wakes the loop and never has its approvals decided autonomously (its card + Deck entry behave exactly as before). Routine approvals pending ACROSS a restart get a one-time seed escalation when the watcher starts / the session hydrates (same gate, same dedupe) ‚Äî without it a hydrated pending approval never transitions again and the agent hangs until the human notices the Deck. Since Phase 5 the escalation routes through the **trigger router** (kind `approval`, subject = the approval id ‚Äî per-project serialization + bounded retries; the `escalatedApprovals` set stays the cross-lifetime dedupe: DELIVERED escalations never repeat, retried-out/dropped ones may re-escalate while still pending, and the build re-checks the approval at delivery time so one already decided drops silently; the autonomy budget applies, see below). **The human card stays fully authoritative for BOTH classes** ‚Äî the Deck queue, the approval takeover and the notification are untouched; the Conductor is an additional fast lane for routine, never a replacement. Codex' native `approvalsReviewer:"auto_review"` (a guardian subagent) was evaluated as an alternative and NOT used: it decides inside codex without the Conductor's project context/memory, offers no hard human-only class, and would bypass SwarmZ's own doctrine + UI; the client-side classification keeps the human contract enforceable and testable.

## Autonomy loop (Phase 5)

The Conductor acts like a human engineering lead WITHOUT being prompted: it reacts to fleet events, judges finished work, distributes follow-ups, reports back and learns. Six triggers exist, ALL funneled through one budget-gated core:

| trigger (`AutonomousTriggerKind`) | fires when | wire text |
|---|---|---|
| `agent-finished` | a conductor-involved session goes busy ‚Üí idle (no pending approval); a FAILED/process-exited turn triggers too, but its wire says "ended its turn WITHOUT completing ‚Ä¶ do NOT report this as success" (no report parsing, no auto-review); an INTERRUPTED turn is a deliberate stop and never triggers | `[agent finished]` ‚Äî report/last message + diff line + optional auto-review + the lead contract |
| `agent-blocked` | same transition, but the agent asks for direction (report `needs_human`, or the free-text fallback: the final message's last line ends in `?` with more than one word before it ‚Äî a bare "Continue?" stays `agent-finished`) | `[agent needs direction]` ‚Äî the question + "decide it yourself when the goal covers it, else ONE compact question to the user" |
| `approval` | a new pending ROUTINE approval (see *Approval routing*) | `[approval escalation]` ‚Äî decide_approval or hand to the human |
| `timer` | a Conductor timer fires (see *Conductor timers*) | `[timer fired ‚Ä¶]` |
| `idle` | a conductor-involved session sat idle ‚â•10 min with uncommitted work (non-empty turn diff), no pending approval ‚Äî ONE nudge per idle stretch (re-armed by the next busy transition; `lastBusyEndAt` is transient, so restarts never replay stale nudges) | `[idle check]` ‚Äî finish the lane or report the stall, "do NOT nudge again" |
| `pr-changed` (Phase 7) | the PR watcher detects a real change on a PR the Conductor WATCHES (`watch_pr` ‚Äî conductor-involvement by construction), or a NEW PR opens while the auto-review-PRs toggle is on (the user's standing order) ‚Äî integration ON only, re-checked at delivery | `[pr update]` ‚Äî judge the change (read_pr/review_pr), merge/close stays the user's; the auto-review variant instructs the review |

**Trigger router** (`lib/orchestrator/triggers-core.ts` pure + unit-tested, wired in `triggers.ts`): every EVENT trigger goes through `enqueueAutonomousTrigger({projectId, kind, subjectId, prepare?, build, onSettled?})` ‚Äî (1) **dedupe** per `(projectId, kind, subjectId)` while pending/running (double events, two windows fire once). Finish triggers key their subject per **busy-cycle generation** (`session#gen`, a monotonic per-session counter): turn B's finish is never swallowed by turn A's still-retrying trigger, and a superseded generation drops at delivery instead of shipping stale context; the finished/blocked pair enqueues under ONE neutral kind (`agent-finished`) and `build()` refines the kind at delivery (`BuiltTrigger.kind`) ‚Äî classification is delivery-fresh, dedupe stays subject-based. (2) **eligibility**: projects store not hydrated = retry (a load race never eats a trigger), record gone = drop, and the event kinds (`agent-finished`/`agent-blocked`/`idle`) stay QUIET for closed tabs (drop ‚Äî the status ping still lands in the chat for later; timers and approval escalations keep delivering into closed projects: a timer is an explicit promise and a routine approval blocks a running agent); (3) **prepare + per-project serialization**: the optional `prepare()` phase (SLOW work ‚Äî the auto-review, up to minutes) runs OUTSIDE the chain so it never starves approvals/timers queued behind it; autonomous work itself runs strictly one at a time per project on a promise chain ‚Äî the `build()` closure (fast, fresh context assembly: last message, report parsing, classification, diff) runs INSIDE the chain right before the turn; a `null` build drops silently (subject vanished / lane re-tasked / generation superseded); (4) **bounded retries**: "retry" (busy chat, open breaker, failed dispatch) re-arms every 30 s at most 10 times, then drops ‚Äî the Phase-4 status ping remains as the fallback signal. The runner is REGISTERED from App.tsx (`registerAutonomousRunner(runAutonomousTurn)`) ‚Äî the same acyclic-imports pattern as the timers. Timers share only the serialization (`runExclusiveAutonomous`), keeping their own persisted retry machinery.

**The turn core**: `runAutonomousTurn(projectId, marker, wireText, trigger)` resolves (or creates) the project's active chat ‚Äî resolving also claims the project's per-launch fresh-chat slot (`freshenedProjects`), so autonomous content delivered before the stage first shows (missed timers, early finishes) stays visible in THAT chat instead of being buried behind the fresh-start chat ‚Äî refuses busy chats and sub-5 s gaps with "retry", passes the **autonomy budget**, then appends the visible system marker ‚Äî stamped **`autonomous: true` + `trigger`** on the message (persisted + hydrated; the ChatView renders the `‚ö° autonomous` marker chip ‚Äî see DESIGN.md) ‚Äî and dispatches through the same honest `dispatchTurn` the user path uses. Budget accounting is a **reservation**: booked before the dispatch, kept once the turn actually STARTED (a mid-turn failure still burned real work), RELEASED on a definitive pre-start failure (`releaseAutonomousTurn` ‚Äî five dead-codex attempts must not trip the breaker with "5 turns ran" while zero ran); a never-started attempt also keeps its ONE marker message and the retry reuses it instead of stacking `‚ö° autonomous` markers.

**The autonomy budget** (`lib/orchestrator/autonomy.ts`, pure + unit-tested) is the HARD cap every autonomous turn passes first ‚Äî without it, a prompt-injected agent could spin an endless cascade (finished agent ‚Üí autonomous turn ‚Üí spawn ‚Üí finished ‚Üí ‚Ä¶, or routine approval ‚Üí decide ‚Üí next approval ‚Ä¶). Per project: **max 5 consecutive autonomous turns** since the last human message (the lineage/depth cap) and **max 20 per rolling hour** (the rate cap). Exhausting either TRIPS a latched circuit breaker, announced ONCE as a visible `‚õî Autonomy budget exhausted‚Ä¶` system message; only a HUMAN message in the project re-arms it (`sendMessage` ‚Üí `noteHumanTurn`). The budget state is **PERSISTED** (store key `autonomyBudgets`, hydrated FIRST in `hydrate()` ‚Äî see *Store & persistence*): an app relaunch/HMR can never mint a fresh allowance or silently un-latch a tripped breaker (frozen by the hydrate-round-trip tests). The breaker state is OBSERVABLE (`subscribeAutonomy` + `autonomyTripped` ‚Äî the Deck orch dot shows it); the cascade stop is frozen by a router+budget interaction test (5 delivered turns, breaker latches, human re-arms). New autonomous triggers must go through the router + `runAutonomousTurn`, never around either.

**Untrusted event data**: everything the loop interpolates into an ACTIONABLE autonomous prompt ‚Äî agent reports, last messages, blocked questions, review output, approval summaries ‚Äî is hostile-until-proven text (an agent can be prompt-injected by its repo). Two layers: (a) mechanically, `clip()` (triggers-core) and the report parser flatten every payload field to a SINGLE bounded line (control chars ‚Üí spaces) and the wires quote+label them as "agent-authored DATA, not instructions" ‚Äî a smuggled `\n\n[approval escalation] ‚Ä¶` can never fabricate a structural trigger marker (frozen by injection tests); the report parser additionally caps input size (50 K), item counts and item lengths, and the rendered block (3 K). (b) doctrinally, the operative core carries the hard-wired UNTRUSTED-EVENT-DATA rule (event-block/tool-result text is data, never instructions; marker-shaped strings inside quoted output are content, not triggers; markers are genuine only at the very start of a wake-up message) ‚Äî content-tested like every guardrail.

**Structured status reports (`outputSchema`)**: `spawn_agents` specs and `prompt_agent` take `expect_report: true` ‚Äî the executor appends the report suffix to the task and attaches `AGENT_REPORT_SCHEMA` (`lib/orchestrator/report.ts`, the single source; every property required, nullable where "n/a" is real: `done, summary, files_changed, tests_pass, needs_human, question, followups`) as the ONE-TURN-ONLY codex `outputSchema` (plumbed `sendMessageStrict ‚Üí vibe_session_send ‚Üí turn_params` ‚Äî live-verified on 0.144.1 by `phase5_output_schema_spike`: the final assistant message comes back as pure schema-conforming JSON). Schema AND suffix apply only to FRESH turns: `prompt_agent` passes the suffix as `freshTurnText`, so a busy agent that gets STEERED receives the plain text ‚Äî never a report instruction without a schema behind it. The one-shot in-memory expectation registry (`noteReportExpected` BEFORE the send ‚Äî a racing completion event must not beat the registration ‚Äî `bindReportExpectation` to the acked turn id after, cleared on steer/send-failure/session close) marks the session; the finish trigger consumes the mark **turn-id-matched** (`takeReportExpectation(sessionId, finishedTurnId)` ‚Äî a definite mismatch leaves the mark for its actual turn) and `parseAgentReport`s the final message at DELIVERY time (fence-stripping, tolerant, null on any doubt ‚Üí the free-text path). The report feeds the classification (`needs_human` ‚Üí `agent-blocked`) and renders compactly into the wire (`renderReportLines`).

**Turn outcomes**: the vibe controller records how every turn ended (`lastTurnOutcomeOf`: completed / interrupted / failed / exited, with the turn id) BEFORE it clears the busy flag, so the watcher's busy‚Üíidle reaction knows what it saw: completed = the normal loop event; failed/exited = the trigger fires with the explicit failure wire (partial output, "do NOT report this as success", no report parsing, no auto-review); interrupted = no trigger at all (a deliberate stop), the spent report expectation is discarded.

**Auto-review** (`settings.autoReviewFinishedLanes`, default off, Settings ‚Üí Orchestrator): when ON and a finished (COMPLETED ‚Äî never failed) lane has a diff signal, the `agent-finished` trigger runs `reviewSession(id, "uncommitted")` (the detached codex review) in its `prepare()` phase ‚Äî OUTSIDE the serialization chain, so a 570 s review can't starve routine approvals/timers queued in the same project ‚Äî memoized across retries, and `build()` embeds the findings (or the failure) into the wire: the Conductor reports REVIEWED work. A lane re-tasked while the review ran (busy again at delivery) drops the turn.

**Learning loop**: the operative core's autonomous-turns section + the memory-behaviour reflection doctrine teach the Conductor to store preference observations after work cycles; mechanically, every 3rd DELIVERED finish/blocked turn per project (`shouldNudgeReflect`, counted via the router's `onSettled`) appends the soft `[learning]` nudge to the wire ‚Äî "store ONE concise observation via remember ‚Ä¶ skip entirely when nothing new emerged". No auto-write: memory still moves only through the explicit `remember` tool.

**Conductor-involvement**: only sessions the Conductor has business with wake the loop ‚Äî `spawnedBy: "conductor"`, or a chat of the project prompted them (`touchedPanes`). Purely human-driven sessions never trigger autonomous turns ‚Äî the gate covers ALL loop events including the routine-approval escalation (a human session's approvals stay exclusively with the human's card).

**Live verification**: `phase5_output_schema_spike` (sessions.rs) and `phase5_autonomy_loop_spike` (appserver.rs) are the `#[ignore]`d live proofs ‚Äî the loop spike runs a REAL agent task with the report schema, then feeds the exact `[agent finished]` wire into a REAL Conductor thread (production instructions + dynamicTools) and asserts it senses (fleet_snapshot/read_agent/git_status), reports by NAME and never leaks a session id.

## GitHub integration (Phase 7)

SwarmZ knows and (opt-in) manages a project's GitHub context over the **locally installed `gh` CLI** ‚Äî there is NO OAuth, no login flow, and no token ever touches SwarmZ; only the user's existing gh state is read. Two layers with different gates:

**Read-only detection ‚Äî always available.** `src-tauri/src/github.rs` (the git.rs subprocess pattern: `output_with_timeout`, `spawn_blocking`, gh resolved via the Settings `ghPath` override ‚Üí homebrew paths ‚Üí PATH) answers EVERY command with a typed `GhOutcome`: `ok` / `not_installed` / `not_authenticated` / `no_remote` / `error` ‚Äî a missing or logged-out gh degrades into an actionable state, never a crash or hang. Reads: `gh_auth_status` (`gh auth status --json hosts` + version), `gh_repo_info` (`gh repo view --json` ‚Üí owner/name, url, default branch, visibility), `gh_pr_list` (`gh pr list --json` with number/title/author/refs/draft/mergeable/reviewDecision/statusCheckRollup), `gh_pr_view` (detail incl. body (capped), files, reviews and the `gh pr diff` output ‚Äî STREAMED into a bounded buffer (`drain_capped`): the 512 KiB cap applies WHILE reading, the excess is drained and discarded, the clipped text trims to a line boundary so hunks stay well-formed; a monster diff never fully buffers). A VANISHED working directory (worktree-removal race) reports "the folder no longer exists", never a false `not_installed`. `summarize_checks` collapses the heterogeneous `statusCheckRollup` (CheckRun + StatusContext entries) into passing/failing/pending ‚Äî repeated runs dedupe per WORKFLOW + check name (same-named jobs in different workflows are different checks; nameless entries never collapse onto each other), the newest `startedAt` wins (like the GitHub UI) ‚Äî fixture-frozen against real gh 2.95.0 output and live-verified by the `#[ignore]`d `github_live_spike` (real repo, real PR).

**The frontend layer** (`src/lib/github/`): `store.ts` is a small in-memory zustand store (gh is the source of truth ‚Äî nothing persists): the auth digest, per-project `{repoStatus, repo, prs, prsFetchedAt, prsError}` ‚Äî `prsError` tracks a FAILED PR fetch independently of the repo detection, so a rate-limited/broken `gh pr list` shows as a stale-list warning instead of a false "no open PRs" ‚Äî and the Conductor's watched-PR sets. `controller.ts` (outside React, started from App.tsx) does lazy per-project detection (panel open, project open, integration flip; parallel calls collapse onto ONE shared in-flight promise that callers ‚Äî notably the `github_status` executor ‚Äî can AWAIT instead of reading a transient "loading"; a thrown detection lands as a terminal error state, never eternal "loading"), and the **GitHub panel** (`components/GitHubPanel.tsx`, title-bar button, right-side drawer) renders it read-only regardless of the toggle: repo header with clickable link + gh auth chip, open PRs with signal-triad badges (checks ok/err/warn, review decision, draft, conflicts), and a PR detail view whose diff goes `gh pr diff` ‚Üí `splitUnifiedDiff` ‚Üí the SAME `TurnDiffFiles`/@pierre/diffs stack as the transcripts. Typed empty states cover no-gh / not-logged-in / no-remote / no-PRs / a failed PR fetch (and a stale-list warning banner when a refresh fails with PRs already shown).

**Opt-in automation ‚Äî the `githubIntegration` master toggle (Settings ‚Üí GitHub, default OFF)** arms four things at once:

1. **The Conductor's 7 GitHub tools** (see *the 31 tools*). GATING DECISION: the tools stay STATICALLY in the registry and are gated at RUNTIME ‚Äî the executors hard-refuse with "integration disabled" while the toggle is off (`github_status` alone answers, reporting the disabled state so the Conductor can explain it). Dynamic registration was rejected deliberately: `dynamicTools` are declared at `thread/start` and live in the rollout, so a mid-run Settings flip could never retract them from running threads ‚Äî the runtime refusal is the only gate that always holds. The persona core's "## GitHub" section teaches the gated reality (never retry a refusal; GitHub is part of the workspace when ON; create_pr/comment_pr/posted review_pr are the precise narrow exception to "you have no outward tools" ‚Äî user order/standing instruction only; merge/close/force are the human's alone, no tool exists).
2. **The Rust write gate**: the frontend mirrors the toggle via `github_set_integration` into a fail-closed `INTEGRATION_ENABLED` flag; `gh_pr_create`/`gh_pr_comment`/`gh_pr_review` refuse server-side while it is off ‚Äî whatever the frontend claims. The gate is RACE-FREE on the falling edge: every write holds a `WRITE_GATE` read guard for its whole duration, `set_integration(false)` flips the flag first (new writes refuse instantly) and then takes the write guard ‚Äî the disable command (async + spawn_blocking) acks only after every in-flight `git push`/gh write DRAINED; the frontend disarms that edge immediately (un-debounced, awaited, retried and surfaced on failure ‚Äî never a swallowed IPC error). `gh_pr_create` additionally pushes PLAIN only (never `--force`), refuses the repo's DEFAULT branch and FAILS CLOSED when the default branch is unknown (`ensure_lane_branch` ‚Äî a malformed `gh repo view` can't fall open past the check into the push), and a failed push's stderr goes through `redact_credentials` (URL userinfo `user:token@`, gh token shapes ghp_/github_pat_/‚Ä¶, ‚â•40-char opaque hex/base64 runs, then a length cap) before it can reach the webview or a Conductor transcript. There is deliberately NO merge/close command anywhere.
3. **The agent-approval routing**: `classify_approval` gains the gh head ‚Äî with the flag ON, agent-run `gh pr comment --body`/`gh pr review --approve|--request-changes|--comment` classify routine (the Conductor's `decide_approval` fast lane); OFF, every gh write is destructive (see *Approval routing*).
4. **The PR watcher**: `github_watch_configure` is declarative ‚Äî the frontend controller hands Rust every OPEN project with a VERIFIED GitHub repo plus the Settings interval (floor 30 s); an empty list stops polling AND drops the per-repo state (the frontend clears the list on disable, so a re-enable starts from a fresh silent baseline), and the ticker also stays silent while the flag is off. Every (re)configure bumps a config GENERATION; a poll that was in flight across a reconfigure/disable re-checks generation + toggle + repo membership AFTER its await and discards stale results ‚Äî no cache mutation, no event for a repo that was just un-watched. Each poll diffs per-PR signatures (`diff_pr_sets`, pure + unit-tested: opened / closed-or-merged / checks / review decision / draft flips; the first poll is a silent BASELINE, carried as an EXPLICIT `first_poll` flag so a zero-PR baseline still reports the first PR opened after it) and emits `github://pr-changed` with the fresh PR snapshot + the change notes. The frontend routes it: PR cache update always; a Deck-ticker `pr` event per change (sessionless ‚Äî the chip opens the PR URL; toasts skip the kind); and an autonomous **`pr-changed` trigger** for PRs the Conductor WATCHES (`watch_pr`) or ‚Äî with the `githubAutoReviewPrs` toggle ‚Äî for newly OPENED PRs. Both variants go through the trigger router (dedupe `pr-<n>`, closed-tab muting, bounded retries) and `runAutonomousTurn`'s budget like every loop event; the wire labels PR titles as UNTRUSTED GitHub-authored data ‚Äî clip-flattened AND JSON-string-serialized, so a title full of quotes/backslashes can never visually escape the data literal and pose as wire text ‚Äî and the build re-checks the toggle AND the watch at delivery (a PR un-watched while the trigger sat queued stays silent; the final closed-notice of a watched PR is exempt ‚Äî its watch drop is bookkeeping, not a decision). Watches are in-memory for ONE app run by design (the tool result says so and points at timers for durable follow-ups).

Two smaller surfaces ride on the toggle: the **Deck PR indicator** (`‚áÖ N` open PRs of the ACTIVE project, err-tinted `√ón` while any PR's checks fail; primitive `deckPrSignature` selector; click opens the panel) and the **suggest-PR-on-finish** line (`githubSuggestPrOnFinish`): a COMPLETED conductor-involved lane on a worktree branch WITHOUT an open PR gets the `[github]` suggestion appended to its agent-finished wire ‚Äî a suggestion only, `create_pr` stays bound to the user's order.

## Codex app-server host (`src-tauri/src/codex/`)

The generic layer under every `codex app-server` consumer ‚Äî the orchestrator brain and the native sessions. Spawn, framing and respawn code exists exactly ONCE, here.

**Wire protocol** (`codex/protocol.rs`, pure + unit-tested): newline-delimited JSON **without** the `"jsonrpc":"2.0"` header (verified against codex 0.144.1). Our requests use an own ascending numeric id; server-initiated request ids are kept as raw JSON and echoed back exactly. `parse_line` classifies each stdout line into response / server request / notification, defensively. The unit tests freeze the live-captured 0.144.1 shapes (reference schema dump + inventory in `docs/codex-protocol/`): `item/started`/`item/completed` with `commandExecution` and `fileChange` items, `item/agentMessage/delta`, `turn/diff/updated`, `turn/plan/updated`, `thread/tokenUsage/updated`, `account/rateLimits/updated` (account-scoped ‚Äî carries NO threadId) and the approval requests, plus the four decisions `accept`/`acceptForSession`/`decline`/`cancel`. There is NO `item/updated` in the protocol.

**Client & binary resolution** (`codex/host.rs`): `Client::spawn` starts `<program> app-server` via `tokio::process` with one writer task (ordered stdin), a stderr logger and a reader task that resolves responses via a numeric-id pending map and forwards server requests/notifications. Reader EOF = process death: the `alive` flag drops and all pending RPCs fail with a clear error. The binary is resolved to an ABSOLUTE path (Settings override ‚Üí current $PATH ‚Üí well-known install dirs ‚Üí `zsh -lc 'command -v codex'` login-shell probe) because macOS gives the built GUI app a minimal PATH; the child's PATH is enriched with the binary's dir. Resolution is cached process-wide (`set_codex_override` installs the Settings value and drops the cache).

**MCP opt-out (Phase 6 live-fix)**: every spawned app-server would otherwise inherit the user's global `~/.codex/config.toml` MCP servers, which boot PER THREAD on the first turn ‚Äî the ChatGPT desktop app's `node_repl` (`startup_timeout_sec = 120`) plus the built-in `codex_apps` raced the Conductor's FIRST dynamic tool call into codex-router errors (`timeout_ms must be at least 10000`, "dynamic tool call was cancelled before receiving a response"; a retry then worked because the servers were warm). SwarmZ agents don't use those servers, so `Client::spawn` disables them for ITS process only: `--disable apps` (the built-in apps MCP) plus one `-c mcp_servers.<name>.enabled=false` per server enumerated by a READ-ONLY parse of `$CODEX_HOME/config.toml` (`mcp_disable_args_from`, pure + unit-tested; names that aren't bare TOML keys are skipped with a log line, an unreadable config degrades to just the apps opt-out). The user's config file is NEVER written. Live-verified on 0.144.1 (`mcp_disable_spike`): the obvious `-c mcp_servers={}` does NOT work ‚Äî `-c` table overrides MERGE into the config table instead of replacing it (probed: a dummy entry is added, the user's servers keep booting; a per-thread `thread/start {config:{mcp_servers:{}}}` merges the same way) ‚Äî while the per-name disable yields zero `mcpServer/startupStatus/updated` events and a clean first dynamic tool call.

**Thread registry & event routing**: a `Connection` (Client + `initialize` handshake + router task) holds a route table `threadId ‚Üí EventSink`. The router forwards each server event to the sink registered for its `params.threadId`; routed server requests carry a `Responder` (owning client + echoed id). No sink responsible ‚Üí notifications are ignored and server requests are refused with -32601. Process death delivers `ThreadEvent::Exited` once per DISTINCT sink and drains the routes.

**Process strategies & respawn**: a `ProcessHost` is one lazily (re)spawned connection slot with a generation counter. The strategies are just slot allocation: **(a) slot per project** ‚Äî each project's Conductor multiplexes its chat threads over its own slot (Phase 3; formerly one global slot); **(b) dedicated process per thread** ‚Äî each session allocates a private slot, so a crash kills only that session. A consumer that sees a different generation than it stored transparently `thread/resume`s before the next turn. `resume_thread` classifies failures typed: `ResumeError::ThreadNotFound` is the consumer's cue to fall back to a fresh `thread/start`. `ProcessHost::shutdown()` ends the slot's process GRACEFULLY (the writer channel closes ‚Üí child stdin EOF ‚Üí codex exits ‚Üí the reader delivers `Exited` per sink), backed by a 5 s watchdog that force-kills a child ignoring the EOF (no zombies until app quit) ‚Äî the idle reaper's primitive; the next `ensure()` respawns with a bumped generation.

**Tests**: framing + frozen fixtures in `codex/protocol.rs`; pending map, resolution, MCP-disable args, routing/exit-dedupe and resume classification in `codex/host.rs`. The `#[ignore]`d spikes (`session_spike`, `sessions_spike`, `appserver_spike`, `conductor_instances_spike`, `phase4_worktree_spike`, `phase5_output_schema_spike`, `phase5_autonomy_loop_spike`, `mcp_disable_spike`) are the live proofs against the real codex CLI: `cd src-tauri && cargo test <name> -- --ignored --nocapture`.

## Conductor brain: Codex app-server (one instance per project)

**Process lifecycle** (`src-tauri/src/orchestrator/appserver.rs`, a thin consumer of `src-tauri/src/codex/`): ONE Conductor instance PER PROJECT ‚Äî a `ProcessHost` slot + ONE DISPATCHER TASK PER PROCESS GENERATION (`INSTANCES` map keyed by `project_id`; the empty key is the neutral probe used by status/model queries without a project). The per-generation dispatchers are the event-generation TAG: handlers compare the event's generation with the CHAT's (`chat.generation`) ‚Äî a straggler from a dead process can never mutate turn state of a chat that already resumed onto a newer one, and a generation's `Exited` fails only turns still running on it (`chat.generation <= event_gen`). Each project's chats multiplex over that project's long-lived `codex app-server` child, spawned lazily on first use ‚Äî the Settings `codexPath` override is passed by the frontend on every `chat_start`/`chat_status` call. `chat_send` claims the chat's turn slot (`done_tx`) ATOMICALLY in one lock section, so parallel sends can't overwrite each other. On process death, running turns of THAT project resolve as failed and its chats get `turn_failed`/`warning` events ‚Äî other projects' instances are untouched; the next call respawns and chats transparently `thread/resume` before the next turn (rollouts persist on disk and codex restores the declared dynamic tools from them). A `ThreadNotFound` resume failure surfaces as an error and the frontend controller handles it by starting a new thread. **Idle reaping**: a background task (started with the first instance, tick 60 s) `shutdown()`s any instance's process after **15 minutes** (`IDLE_REAP_SECS`) without chat activity (commands or routed events ‚Äî a streaming turn keeps it warm) and without a running turn. TOCTOU-safe via the per-instance **lifecycle lease** (a tokio mutex): spawn-touching operations touch the idle clock, THEN take the lease; the reaper `try_lock`s and RE-CHECKS idle + running-turn under the lease before shutting down ‚Äî a racing send either resets the clock in time or blocks until the shutdown finished and simply respawns. The instance, its chats and thread ids survive a reap; the next message is just a respawn + transparent resume. Closing a project TAB does not touch its instance. Live-verified end-to-end by the `#[ignore]`d `conductor_instances_spike` (two instances, separate processes/cwds, correct event routing, respawn transparency of one never affecting the other).

**Chat ‚Üî thread mapping**: `orchestrator_chat_start(codex_path, persona, project)` runs the `initialize` handshake (`capabilities.experimentalApi: true`) and `thread/start` with **`cwd` = the PROJECT DIR** (fallback home when the folder is gone), `sandbox: "read-only"`, `approvalPolicy: "never"`, the whole registry adapted to the experimental `dynamicTools` field, and the compiled instructions (persona + project + scoped memory + operative core) as `developerInstructions`. Persona AND project are passed by the frontend on start/resume and captured on the chat (reused for transparent resumes); memory (global + project scope) is read fresh each start/resume. `orchestrator_chat_send(chat_id, text, model?, effort?)` prepends a fresh PROJECT-scoped fleet-summary line (`[fleet status: ‚Ä¶]`, best-effort via `run_tool("fleet_snapshot")` with the chat's `project_id`) to the user text, starts the turn and resolves with `{status, text}` on `turn/completed`; `model`/`effort` are per-turn `turn/start` overrides. `thread/tokenUsage/updated` re-emits as a `token_usage` chat event for the context gauge. `orchestrator_chat_compact(chat_id)` (Phase 8) claims the chat's turn slot and BLOCKS until the compaction's turn/completed (300 s bound); on TIMEOUT the slot is NOT cleared ‚Äî a possibly-still-running compaction turn would otherwise let a later send claim `done_tx` and the stale `turn/completed` resolve the new sender ‚Äî instead a best-effort `turn/interrupt` forces the terminal event and the slot frees only through it (turn/completed or process exit). `orchestrator_chat_status(codex_path, project?)` reports `{running, version, account}`, reuses ANY alive instance before spawning, and never rejects for a dead process; `codex_list_models` does the same.

**Dynamic-tools bridge** (`orchestrator/adapter.rs`): `item/tool/call` server requests are answered from the bus ‚Äî `orchestrator::run_tool` with args normalized and the instance's `project_id` attached (the executors scope on it), the result adapted to `{success, contentItems:[{type:"inputText", text}]}`. Each call runs in its own task so deltas keep streaming during tool roundtrips.

**Event stream**: `orchestrator://chat-event` `{chat_id, kind, data}` with kinds `turn_started`, `delta`, `message`, `tool_call` `{tool, args_summary}`, `tool_done` `{tool, ok}`, `token_usage`, `turn_completed`, `turn_failed`, `warning`. Typed wrappers + `onChatEvent` in `src/lib/orchestrator/chat.ts`.

**Approvals & safety**: under read-only sandbox + `approvalPolicy: "never"` + the instructions, approval requests must never occur ‚Äî if one arrives anyway it is auto-DECLINED and any other server-initiated request is refused with a JSON-RPC error; both emit a `warning` event. MCP servers from the user's codex config are ENFORCED off, not just instruction-forbidden: every SwarmZ-spawned app-server (Conductors AND sessions) gets `--disable apps` plus one `-c mcp_servers.<name>.enabled=false` per server enumerated read-only from `~/.codex/config.toml` (`host.rs mcp_disable_args`; a non-bare TOML server name can't be disabled per-CLI and is logged ‚Äî live-verified by `mcp_disable_spike` on 0.144.1). The instructions' "SwarmZ tools only" rule remains as a second, doctrinal layer.

**Controller** (`src/lib/orchestrator/controller.ts`, outside React): owns the store-chat ‚Üî backend-`chat_id` mapping and routes `onChatEvent`. `sendMessage` appends the user message, auto-titles from the first user message, lazily `chatStart`s ‚Äî or `chatResume`s the persisted `threadId` after a restart, both on the chat's PROJECT instance (`ProjectContextWire {id, dir, name}` resolved live from the project store; a lost record degrades to an id-only context ‚Üí Rust falls back to home as cwd); a failed resume warns and starts fresh ‚Äî then `chatSend`s. **Delta batching**: word-level `delta` events flush at most every ~80 ms; message rows are memoized so only the streaming row re-renders. `tool_call`/`tool_done` append/patch tool messages (FIFO-matched per tool name). **Jump chips**: tool args containing a live session id, and sessions born during a `spawn_agents` call (order diff around the call), attach `paneRefs` ‚Äî both scans are FILTERED to the chat's project, so parallel creation in another project can never attach foreign chips; the UI renders "‚Üí name" chips that jump via `focusSession` and hide once the target is gone. `createChat(projectId?)` (default: the active project; null when none) stamps new chats with the Settings codex model/effort defaults (a reused EMPTY chat is re-stamped); `ensureFreshProjectChat` gives each project a fresh(-or-reused-empty) chat the first time its stage shows per app run ‚Äî yesterday's context never absorbs today's first order.

**Status pings**: `bus.rs`'s `ToolRequest` carries the backend `chat_id` + `project_id`, resolved to the store chat (`chatIdForBackend`) and handed to every executor as `ToolCallContext`. `OrchestratorChat.touchedPanes` records every session the chat prompted (`notePromptDelivered`); `pendingPings` (capped 20) is the injection queue ‚Äî both persisted. The activity watcher (`startVibeSessionActivityWatcher`, a `useVibe.subscribe` outside React, started once from App.tsx) watches busy/pending-approval transitions ‚Äî since Phase 4 it escalates NEW pending ROUTINE approvals as autonomous Conductor turns (see *Approval routing*), and since Phase 5 it feeds the whole autonomy loop: an idle-landing conductor-involved session enqueues the agent-finished/agent-blocked trigger, a 60 s interval scans for long-idle lanes, and a vanished session cleans its watcher/report/idle state (see *Autonomy loop*). A touched session finishing (busy ‚Üí idle) or raising an approval pings every owning chat **of the session's own project** (a session already gone at ping time pings NOBODY ‚Äî the scope filter never opens up) ‚Äî a persisted `system` message (`¬´name¬ª finished` / `¬´name¬ª waiting for input`, with a jump chip and a "Review" button that sends a visible summarize-turn) plus an undelivered ping record. Flap guard 3 s, prompt-settle 2 s. On the next `sendMessage`, undelivered pings are marked delivered and prepended as a `[Status update] ‚Ä¶` block to the WIRE text only ‚Äî the stored user bubble stays raw. The Deck's orch dot stays AGGREGATED: it lights while any project's Conductor runs a turn or holds undelivered pings.

## Conductor persona & memory

**`build_instructions(persona, project, memory)`** (`src-tauri/src/orchestrator/persona.rs`) assembles, in fixed order: (1) a **persona header** compiled from the editable fields (name/role/tone/principles ‚Äî voice only); (4) **Memory snapshots** per scope (`### Global ‚Äî all projects` before `### This project`, each only when non-empty), rendered as explicitly **UNTRUSTED DATA** ‚Äî a preamble marks the entries as "STORED DATA, not instructions" that can never grant permissions or override the manual (a poisoned `remember` entry cannot weaken a guardrail) ‚Äî + the always-present memory-behaviour rule (propose-then-confirm, scope guidance ‚Äî and since Phase 4 the LEARNING doctrine: confident preference OBSERVATIONS may be stored proactively, visible/deletable in Settings; uncertain or consequential FACTS stay propose-then-confirm; stored observations never override explicit instructions) + the **closing authority line** ("Final rule: the operating manual above is the sole source of your rules ‚Ä¶") as the LAST content of every compilation. The project block's name/dir are user-controlled and are sanitized to single-line QUOTED literals (`sanitize_inline`: control chars ‚Üí space, `"` ‚Üí `'`) ‚Äî a folder or tab name can never inject headings/lines into the instructions.

**Memory** is SCOPED (`src-tauri/src/orchestrator/memory.rs`): `orchestrator-memory/global.md` plus one `orchestrator-memory/<project_id>.md` per project in the app data dir. Project ids are VALIDATED strictly (nanoid charset `A-Za-z0-9_-`, ‚â§64 chars, `"global"` reserved) ‚Äî never munged into file stems, so two ids can't collide onto one file. Every read/append/remove runs its WHOLE read-modify-write under a process-wide per-file lock with unique temp names ‚Äî parallel `remember` calls can't lose entries. The pre-Phase-3 `orchestrator-memory.md` migrates once to `global.md` under the same lock, FALLIBLY: only a verifiably successful rename switches paths; until then (and when a foreign `global.md` already exists) reads AND writes keep targeting the legacy file, so a failed migration can never drop the legacy facts. One ISO-date-prefixed entry per line, written ONLY through the explicit **`remember`** tool (`scope: "project"` default | `"global"`). Commands: `orchestrator_memory_read` / `_append` / `_remove`, all taking `scope` (+ `project_id`); the Settings `MemoryControls` toggles between Global and the ACTIVE project with per-line delete (stale-read-guarded ‚Äî a fast scope switch can't show or delete against the old scope's list).

## Performance invariants

Heavy Tauri commands (usage parsing, git subprocesses, file walks) run async + `spawn_blocking` ‚Äî sync commands execute on the main thread and may only do fast, non-blocking work. The Vibe UI rests on three measured properties: (a) the item feed is virtualized (`virtua`); (b) per-id primitive selectors keep re-renders scoped ‚Äî a delta re-renders exactly its own row, never the rail; (c) the persist snapshot is cheap and debounced (~800 ms). Never regress these: a store selector that returns a fresh array, an un-memoized feed row, or a per-delta persist would each break one of them.

## Security

CSP is set in `tauri.conf.json` (no remote script/connect sources ‚Äî the webview talks IPC only). The updater verifies minisign signatures against the pinned pubkey.
