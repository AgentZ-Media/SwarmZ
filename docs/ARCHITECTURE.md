## Transport & backends

`src/lib/transport.ts` exports the native Tauri backend. Backend capabilities are declared in `backend-types.ts` and implemented in `backend-tauri.ts`. Native-only features may skip the backend interface and use `@tauri-apps/api` directly (e.g. `lib/worktree.ts`, `lib/orchestrator/native.ts`, `lib/vibe/native.ts`, `lib/projects/store.ts` — the `canonicalize_path` dedupe resolver). Session lifecycle/event reduction stays in `lib/vibe/controller.ts`; its typed Tauri command boundary lives separately in `lib/vibe/native.ts` and has wire-characterization tests.

## Store & persistence

`src/store.ts` holds the shared app state. Persisted: usage history, quick notes, and app settings (`AppSettings`: last used folder, codex/git path overrides, auto-update, worktree repo registry `worktreeRepos`, orchestrator prefs — codex model/effort defaults, scan roots, the Phase-5 `autoReviewFinishedLanes` toggle — the Phase-6 `reduceMotion` switch, wired to `data-motion="off"` on the root from App.tsx — and the Phase-7 GitHub fields: the `githubIntegration` master toggle (default OFF), `githubAutoReviewPrs`, `githubSuggestPrOnFinish`, `githubWatchIntervalSec` and the `ghPath` binary override) — edited through the thin `SettingsDialog.tsx` composition and domain sections under `components/settings/`, opened via the title-bar gear or `⌘,` (Tauri store `swarmz.json`). Always mutate settings through `updateSettings` (merges + persists the whole object). In-memory only: palette/notes/dashboard/GitHub-panel flags, the worktree panel scan, `quitConfirm` (and the whole GitHub store — see *GitHub integration*). Window size/position persists natively via `tauri-plugin-window-state`. Sessions live in their own store (`lib/vibe/session-store.ts`, key `vibeSessions`), orchestrator chats in `lib/orchestrator/chat-store.ts` (key `orchestratorChats`), project tabs in `lib/projects/store.ts` (key `projects`), Mission Control in `lib/missions/store.ts` (key `missions`) with its write-ahead side-effect outbox in `lib/missions/outbox-store.ts` (key `missionOutbox`), project Runtime Environment contracts in `lib/runtime/store.ts` (key `runtimeEnvironments`), Conductor timers in `lib/orchestrator/timers.ts` (key `conductorTimers`), autonomy budgets in `lib/orchestrator/autonomy.ts` (key `autonomyBudgets` — serialize/hydrate stay pure in the module; store.ts wires the load/save + a debounced dirty sink via `registerAutonomyPersist`). `flushAllPersists` uses `allSettled` over every one of these slices plus settings/notes/usage. Hydration has one fixed order: settings + schema migration → usage history → quick notes → autonomy budgets → projects → vibe sessions → missions → mission outbox → runtime environments → orchestrator chats → conductor timers. Budgets must exist before any later store readiness can permit autonomous delivery; sessions migrate against projects; Mission projection, outbox and Runtime contracts must all be authoritative before Mission dispatch; chats migrate against projects + sessions; timers fire missed work only after projects/chats are safe.

The old pane-era keys (`grid`, `workspaces`, `workspacePresets`, `commandPresets`, `customCommands`, `profiles`) are DELETED once by the schema-v2 migration (`DEAD_STORE_KEYS` in `lib/schema-version.ts` → `deleteStoreKeys` in the backend).

The store file itself has a crash net: `storefile.rs` runs in Tauri `setup` before the webview loads — a healthy `swarmz.json` refreshes `swarmz.json.bak`, a corrupt one (torn write, disk-full) is moved aside as `swarmz.json.corrupt-<ts>` and the backup restored, so the plugin's silent treat-unreadable-as-empty behavior can no longer factory-reset the app.

Mission execution has an additional pure authority boundary in `lib/missions/envelope.ts`: every material revision clears its approval, and an exact envelope revision must receive a unique human approval before starts are admitted. `controller-core.ts` applies that boundary to the event log by deriving the complete authorized snapshot from the latest HUMAN `mission.activated`/`mission.resumed` event. Tasks added or materially edited after that revision are not schedulable until a new human activation; this includes the explicit `allowNoop` task exception. Coding tasks otherwise fail closed unless HEAD differs from the approved base **and** the scoped diff has at least one file. A needs-human/blocked lane can only re-enter through HUMAN `task.requeued`, whose bounded instruction is copied once into the next immutable attempt, followed by a new activation that approves the revised scope. The envelope's task/attempt/token/time/cost/parallel budgets and root/tool/network/GitHub capabilities fail closed; regression/conflict/critical-failure policies decide when work pauses or escalates, while the existing persisted autonomy breaker can veto every start as the emergency stop. `runner-core.ts` combines only durable mission projections, scheduler admissions and that envelope into idempotent launch declarations. Every declaration creates a fresh temporary one-assignment worker (`resumeExistingSession:false`, no persona or durable memory), and duplicate operation ids cannot spawn twice. Command idempotency hashes canonical `{missionId, actor, type, data}` only; retry transport metadata (`eventId`, `revision`, `occurredAt`) may differ, while a real command-payload change still conflicts. Agent completion uses `report-v2.ts`: mission/task/attempt identity plus Git/diff/artifact SHA and command exit-code claims are bounded and parsed as untrusted evidence; even a non-success report must bind to its persisted attempt, and a claimed success cannot settle until independently observed HEAD/diff/command results match. A legacy `tests_pass` boolean is never authority.

The impure lifecycle lives in outside-React `lib/missions/controller.ts` and starts once from `App.tsx`. It waits for Mission/outbox/session/runtime hydration and healthy persistence, runs the pure scheduler with adaptive 1–8 admission plus a hard 48-backend cap, and admits only tasks inside the approved snapshot. The cap consumes the Rust `vibe_session_live_backend_count`: only currently alive session `app-server` processes count; persisted threads, closed lanes and crashed process slots do not permanently exhaust capacity. Autonomous workers require a new or explicitly shared worktree; main-checkout and integration-branch placement are refused. Before it writes an attempt or spawn command it resolves the canonical main repo root and observes its HEAD through the read-only Rust `mission_git_evidence` command; the durable spawn payload contains that base SHA, deterministic branch/path, exact canonical diff command and `copyEnv:false`. Native `worktree_add(baseSha)` creates the branch from that exact full commit id rather than the main checkout's later mutable HEAD, then dispatch verifies exact branch/base again. It starts a secret-free deterministic-id workspace-sandboxed Codex session (`networkAccess:false`) and requires Report v2. Mission sessions are distinct from Conductor lanes and accept exactly one controller-owned turn. Transcript user/assistant/command items persist their backend turn id; settlement selects only the turn in the spawn receipt, so a later foreign report or command cannot become evidence, while a stamped bound report safely reconstructs completion after restart. Worker command events are never gate authority: after final Git evidence is frozen, every approved command is parsed to direct argv and run by the native Acceptance Runner inside a no-network macOS sandbox whose writable root is the exact worktree (not the approved main repo). The fixed app-owned SBPL is passed canonical paths only through bounded `sandbox-exec -D` values. It grants read/execute to a real canonical `<main>/node_modules` and narrowly discovered, ownership/permission-checked `/usr/local` Node/Corepack, user pnpm/Corepack cache, Cargo registry/git cache and only the active Rustup toolchain. The rest of main (including source, `.env` and `.git`), sibling worktrees, host-home credentials and external network remain inaccessible. Every child uses `env_clear`; `HOME`/`TMPDIR` stay worktree-confined, Cargo/Corepack are offline, and caller input cannot override `PATH`, `HOME`, `CARGO_HOME`, `RUSTUP_*`, `COREPACK_*`, `NODE_PATH` or the other reserved process variables. Git evidence is read again after each command and must retain the exact clean HEAD/base/diff hash/file set. Attempt-bound commit, test-result, detached-review and usage artifacts carry `authority:"swarmz_native"`; gate results link those exact artifacts before success can settle. Failed attempts use bounded fresh workers; needs-human/blocked attempts stay terminal until `requeueTask` atomically records the human instruction, an explicit retry-limit extension when exhausted, and a new human approval revision. Breaker/hard-budget/pause/cancel stops interrupt owned running workers. Terminal workers close only after evidence, Mission state and settle receipt are durable; controller cleanup never removes their worktrees.

`lib/runtime/` persists up to 32 validated Runtime Environment specs per project (bounded direct argv setup/cleanup, services, port declarations and database namespace; no shell). Mission activation freezes the selected spec fingerprint in its approval envelope. Mission-bound specs deliberately reject all secret bindings because worker-controlled code could copy a secret into the worktree. Setup runs first; a durable runtime-prepared artifact is flushed before any service/worker turn can start, closing the setup-done/service-not-yet-owned crash window. Service leases bind deterministic instance ids to project, Mission, attempt, main root and worktree; reconcile/cleanup refuse ownership drift and require all processes to disappear. Setup/cleanup/acceptance have no network; services receive localhost only. Native capture is bounded, timed out, process-group terminated and redacts resolved secrets for non-Mission callers.

Every external Mission Control side effect crosses the durable outbox (`lib/missions/outbox*.ts`) before dispatch. Spawn, prompt, settle, integrate and gate commands carry a global idempotency key and move through `pending → claimed → delivered`, or bounded `failed → dead_letter` retries. Claims are write-through leases with claim-token fencing; the normal claim path durably reaps an expired lease and applies deterministic backoff, while a stale completion cannot win after a re-claim. There is one production recovery engine, not a second startup-plan model: `settlement-service.ts` replays due spawn/settle records, reapplies a delivered settlement whose projection event is missing, and fails a running attempt that has no recoverable durable spawn. `recovery-core.ts` validates Mission/task/attempt/session id, project, cwd, worktree, `spawnedBy:"mission"` and workspace access before session reuse; it closes only orphaned Mission-owned lanes, never human or Conductor lanes. Spawn's success-before-outbox-ack window is recovered from the exact persisted prompt/turn binding. Integration independently adopts authoritative success-before-ack Git receipts after proving the expected operation is already effective, so it never applies the patch twice. Unknown/corrupt Mission, outbox, Vibe or Runtime persistence pauses dispatch fail-closed. The outbox has independent strict serialization, record/payload/receipt/16-MB caps, is hydrated immediately after Mission replay, and joins `flushAllPersists` at quit. `outbox-compaction.ts` prevents unbounded history growth: archive writes first insert compact, immutable audit artifacts (turn ids, settlement ids, integration before/after heads, clipped gate stdout/stderr) in the same Mission event batch, then only receipted `delivered` records whose archive and matching terminal effect have crossed the durable Mission-log boundary are removed in deterministic write-through batches. Pending, claimed, failed, dead-letter, reconciliation-needed receipts and every non-archived Mission record are retained. At 9,500 records, admission backpressure refuses new spawn/prompt/integration/gate work with an archive escalation while reserving 500 slots for terminal settlement; the 10,000-record hard cap remains the final bound.

`swarmz.json` also carries a top-level **`schemaVersion`** key (a plain number, currently `3`) — the migration anchor, deliberately WITHOUT a migration framework. The pure hooks live in `src/lib/schema-version.ts` (unit-tested): `normalizeSchemaVersion` leaves valid stored versions — even NEWER ones — untouched, so a downgraded build never clobbers a newer stamp. `planSchemaMigration` runs first in `hydrate()`: pre-v2 stores receive the idempotent dead pane-key cleanup, pre-v3 stores additionally lose only the removed nested `settings.orchestratorPersona`, and every older/invalid stamp advances to `3`. The actual sessions→projects value migration remains in the tolerant Vibe hydrator (see *Projects*); `storefile.rs` rescue is version-agnostic.

Human-approved A/B(/N) work is a first-class `CandidateBatch`, not a UI-only selector. A request freezes 2–8 fresh one-assignment lanes into a new human-approved Mission revision; every candidate gets its own deterministic branch, worktree, session and exact turn. While the batch is incomplete the task remains running, and once all lanes are terminal it becomes `needs_human` rather than inheriting whichever attempt happened to finish last. `candidates.ts` scores only controller-observed, final-HEAD-bound artifacts: native commit/test evidence, a detached exact-commit review (actionable P0–P3 findings fail that evidence), and native-observed usage/duration tie-breakers. Agent reports or arbitrary artifact labels are never selector authority. An unambiguous margin can be confirmed only for the selector's exact winner; an ambiguous/no-eligible result needs a separate bounded, human-authored override event. The selected attempt id is persisted on the task so Integration Train commit discovery cannot silently choose another successful candidate.

Integration Trains are dependency-ordered and checkpointed before the first operation and after every integrated entry. Native Git accepts only the recorded clean integration branch at the exact expected HEAD; operations are cherry-pick or merge (the UI describes the strategy actually used), and combined regression gates run through the same direct-argv sandbox. A blocked train has three explicit human-only remediations in `MissionIntegrationView`: **Retry** the failed entry with a new durable `retryRevision`, **Skip** it with a bounded permanent audit reason, or type-confirm **Rollback** to one human-selected durable checkpoint. Retry/Skip write approval artifacts and mark stale outbox operations delivered as superseded without erasing them. Rollback records approval, refuses an active claim or dirty/wrong branch, uses an expected-HEAD-fenced native reset, persists its completion proof and requeues only entries after the chosen checkpoint; the failed entry remains blocked until Retry/Skip. The autonomous tick never performs destructive rollback or invents a remediation choice.

Mission reminders live in the same append-only Mission event log (`schedule.created/cancelled/claimed/delivery_failed/fired`) and are project-bound. `lib/missions/schedules.ts`, started from `App.tsx`, re-arms future reminders after hydration, fires overdue reminders as missed-on-restart, and flushes a durable `claimed` event immediately before the native OS notification. Native permission denial or a synchronous send error appends `delivery_failed`, releases the claim and schedules visible bounded backoff; only a successful native API return appends `fired`. A crash in the claim/notification window remains a visible claimed-only **uncertain** delivery and is not retried automatically. Cancel/archive atomically cancels every still-pending reminder. The same controller watches derived Mission status transitions and emits local notifications for attention/terminal states without replaying hydrated historic status on launch. Header lifecycle actions use typed-title confirmation; cancellation interrupts owned workers, while archiving is disabled with running work and makes the event log immutable.

## Projects & swarm data model (Phase 2)

**Entity** (`Project` in `types.ts`, persisted key `projects` via `lib/projects/store.ts`): `{ id, dir, name, order, lastActiveAt, createdAt, closedAt? }`. `dir` is the CANONICAL folder path — `openProject` resolves it through the Rust `canonicalize_path` command (symlinks, `/var` aliasing; falls back to the normalized input when the folder is gone) and dedupes on it, so the same folder always maps to the same project. `name` is the folder basename with a lowest-free numeric suffix on collision ("api", "api 2"). `order` is the tab position among OPEN projects; `activeProjectId` persists. Pure logic (dedupe key, naming, reorder patches, migration) lives in `lib/projects/core.ts` (unit-tested); the zustand store around it debounces persists (~500 ms, flushed by `flushAllPersists`).

**Open vs closed**: closing a project TAB only stamps `closedAt` — the entity stays persisted and its sessions stay in the session store; reopening the same folder (picker, recents, a triage jump) flips the SAME project open again and the sessions reappear. `setActiveProject` auto-reopens.

**Sessions belong to projects** (schema v2): `VibeSession` carries `projectId`, `agentName` (a neutral temporary lane label, not a persona), `spawnedBy: "user" | "conductor" | "mission"` and `worktree: { root, branch, shared } | null`. **Hydrate migration**: every hydrated session runs through `useProjects.adoptSessions` (pure core: `assignSessionsToProjects`) — a valid `projectId` passes through; pre-v2 sessions and sessions whose project record was lost get projects derived from their `projectDir`s (deduped, matched against open AND closed projects, created OPEN otherwise); `agentName` defaults to the old display name, `spawnedBy` to "user". The stamped result persists immediately (`vibeSessions` writes `version: 2`). `MAX_SESSIONS_PER_PROJECT = 30` is now a soft UI/archive threshold only: create/hydrate never evict a session automatically, because a large Mission may legitimately exceed it and historical deletion requires an explicit archive/delete flow.

**Temporary lane labels** (`lib/vibe/names.ts`, pure + unit-tested): a deterministic pool of 128 neutral operational labels (`Lane 01` … `Lane 128`). `pickAgentName(taken)` chooses the lowest case-insensitively free label and adds the lowest numeric suffix only after exhaustion; it does not assign a personality or reusable identity. `branchSlugForAgent(name, task?)` produces a bounded ASCII worktree branch such as `swarm/lane-01-checkout`. The new-lane dialog prefills the next label but keeps it editable; `spawn_agents` without an explicit display label uses the same pool.

**Scoping**: the Fleet grid shows only the ACTIVE project's sessions (primitive status-signature selector; no active project = show all, a degenerate pre-migration state). The Deck triage/counters stay GLOBAL across projects — `focusSession` switches to the session's project tab first (reopening a closed one), then selects it; `activateProject` (tab click, ⌘1–⌘9, palette) aligns the stage: selection restores the project's REMEMBERED session (`activeIdByProject`, persisted; stamped by `setActive`/`createSession`, validated on hydrate), else the project's newest; a session-less project lands on the Conductor. The orchestrator's `spawn_agents` attaches sessions to the Conductor's own project when `cwd` is omitted/equal to the project folder; an explicit foreign `cwd` resolves its own project (NOT the active tab) and never steals the active tab (`activate: false` for conductor spawns).

**TitleBar** (48 px, Vibe v3): the ⌘B Conductor-sidebar toggle, the hexagon brand mark + "SwarmZ" wordmark + `/`, then one tab per open project with a live dot (busy = accent `zpulse`, idle = faint), `busy/total` count, `⚑n` needs-you badge, close X (middle-click too), double-click rename, drag reorder; the + button opens a folder picker / discovery-recents dropdown ("Open project" also lives in the ⌘K palette, along with tab switching). Right side: the amber needs-you pill (`N need you`, click = ⌘⇧A jump), the ⌘K button, update pill, usage/notes/worktrees/settings buttons and the accent "New agent" CTA. Tabs are plain `<div>`s and opt out of the header drag region via `data-tauri-drag-region="false"`. **Closing with busy sessions** never blocks and never stops anything, but routes through `requestCloseProject` (vibe controller): N busy sessions raise the `CloseProjectConfirm` dialog (busy count + "they keep running in the background") before the tab hides.

## Quick notes

**Quick notes** (⌘N, title-bar note button, `QuickNotesPanel.tsx`): a right-side slide-in drawer holding lightweight checklists — one **global** list plus one **per project folder** (store key `quickNotes` → `QuickNotesData`, folder lists keyed by repo root). The drawer opens in the scope of the active session's `projectDir` (fallback Global) and shows a chip per scope. Items (`NoteItem`) check off, edit inline, reorder by dragging the grip, and can toggle to plain text lines (`plain`); "Clear done" drops every checked item, emptied folder lists drop their key. Every mutation persists immediately. The drawer is a `role="dialog"` so the global-shortcut guard kicks in; ⌘N closing it is special-cased in the dialog branch of the `App.tsx` keydown handler.

## Usage & limits

Codex usage is parsed from `~/.codex/sessions/**/*.jsonl` by `src-tauri/src/codex_usage.rs` (which also owns the shared usage types since the Claude parser was removed), using `session_meta`, `turn_context`, `event_msg/token_count`, `task_started` and `task_complete` rollout events for session id, cwd, model, tokens, context, limits and activity. The pre-rebuild per-session aggregation commands (`usage_for_dir`, `usage_for_session`, `usage_totals`) were REMOVED in the final audit (R13) — they had no frontend caller because live session accounting runs frontend-side (see below); `codex_usage.rs` now serves only the account-level limits. Codex ChatGPT-plan rollouts do not include reliable USD cost, so cost stays zero (there is no pricing table anymore).

Codex limit windows are **account-level**: `codex_account_limits` finds the newest `token_count.rate_limits` event across ALL of `~/.codex/sessions/**/*.jsonl` — files are iterated newest-mtime-first, each only via a bounded 256 KiB tail read, stopping at the first hit — and is polled every 60 s by `src/lib/limits.ts` (paused while hidden) into the Deck meters. `limits: null` = no data ever seen (the Deck shows a quiet `CX —` placeholder); data older than 1 h gets an "as of HH:MM" tooltip suffix.

The **usage dashboard** (`UsageDashboard.tsx`, title-bar chart button) shows the persisted all-time history (store key `usageHistory`, Codex entries; the pre-rebuild Claude/shell entries are filtered from view). **Session accounting** feeds it live: the vibe controller mirrors each session's `token_usage` events into `recordUsageHistory` (keyed `codex:<session id>`, `runtime:"codex"`, `by_model` from the session's model override or "codex default") — debounced ~2 s per session, flushed once more on close. The thread's `total` bucket is cumulative, so each write REPLACES the entry; cached input counts as cache reads and is subtracted from fresh input; `message_count` = user turns in the transcript; cost stays 0 (no pricing).

## Deck & fleet events

The **Deck v2** (`components/Deck.tsx`) is the permanent 32 px status bar under the fleet. Title bar = place/navigation, Deck = system status (see DESIGN.md) — all `text-11 font-mono tabular-nums`, it never grows. Anatomy left→right:

1. **Fleet counters** (GLOBAL across all projects — the Deck aggregates the whole swarm, unlike the project-scoped grid): `▸ working` (accent-tinted while >0) · `⚑ needs-you` · `✓ finished` (last 5 min) · `· idle`, derived from ONE primitive status signature + a 30 s decay tick. The ⚑ counter IS the triage queue: at N>0 it turns amber and opens the popover listing every session with a pending approval, oldest waiting first (`vibeTriageEntries` in `lib/vibe/triage.ts`); arrow keys rove, Enter jumps (`focusSession` — which switches to the session's project tab first). ⌘⇧A jumps to the oldest directly.
2. **Event ticker**: the last 3 fleet events inline (newest last), each clickable to jump while its session still exists; a history icon opens a popover with the last ~30.
3. **Equalizer**: five staggered `animate-zeq` accent bars — visible only while any agent or Conductor works (the ambient "fleet is alive" signal).
4. **Conductor dot**: `conductor` + dot — AGGREGATED across all projects' Conductors: `--err` when the ACTIVE project's autonomy circuit breaker is latched (read via `subscribeAutonomy`/`autonomyTripped` through `useSyncExternalStore` — a primitive-boolean snapshot; tooltip says autonomy is paused until the human writes), `--attn` when any chat has undelivered pings, accent (`zpulse`) while any turn runs, faint otherwise; click = `showConductor` (fleet + sidebar of the active project).
5. **PR indicator** (Phase 7, only while the GitHub integration is ON): `⇅ N` open PRs of the ACTIVE project, err-tinted `×n` while any PR's checks fail — a primitive `deckPrSignature` selector string, click opens the GitHub panel.
6. **Meters**: the account-level Codex meters (`cx 5h`, `cx wk`) with full-word tooltips, then the clock (HH:MM, 30 s tick).

**Toasts** (`components/Toasts.tsx`, top-right, z-70): transient event cards fed from the same fleet feed — finished (ok) / needs-approval (attn) / agent-started (accent ✦), `animate-ztoast`, auto-dismiss 5 s, click jumps to the agent. Initialized past the current feed so a mount never replays history; orch-prompt/exited events stay ticker-only.

**Fleet event feed** (`src/lib/events.ts`, a standalone zustand store): `FleetEvent { id, at, kind, sessionId, sessionName, label?, url? }` with kinds `finished` / `waiting` / `orch_prompt` / `created` / `exited` / `pr` (Phase 7 — sessionless: `sessionId: ""`, a pre-built `label` and the PR `url` the chip opens; toasts skip it), capped at 50, **in-memory only by design**. Producers: the vibe controller (`approval_request`→waiting, `turn_completed`→finished, `turn_failed`→exited), `notePromptDelivered` in `orchestrator/executor-agents.ts` (`orch_prompt`) and the GitHub controller (`pr` per watcher change note). Repeated `waiting` for the same session within 3 s is deduped in the feed itself (aligned with the ping flap guard).

## Git status

`git_info` (`src-tauri/src/git.rs`) is a strictly read-only snapshot (repo, branch, ±lines, untracked, `origin` URL) that shells out to the git binary (prefers the Settings `gitPath` override, else `/usr/bin/git`; every subprocess runs through `output_with_timeout`). There is no polling anymore — the orchestrator's `git_status` tool and the worktree flows call it on demand.

**Repository-controlled execution is suppressed in EVERY backend git subprocess (audit C1).** SwarmZ's own git runs UNSANDBOXED in the Tauri backend, while hook bodies / fsmonitor configs / diff drivers are workspace-agent-editable content (a husky-style tracked `hooksPath` makes the hook body a plain tracked file a sandbox-permitted edit can rewrite) — so `git worktree add` would run `post-checkout`, `git status` a configured `core.fsmonitor` hook, `git update-ref` a `reference-transaction` hook, all with host authority. Two builders in `git.rs` are the ONLY way the backend spawns git: `git_command` (local ops — status, worktree management, ref queries) prepends `-c core.hooksPath=/dev/null -c core.fsmonitor=false -c core.pager=cat -c protocol.allow=never` and `GIT_TERMINAL_PROMPT=0` (no hook is ever found, no fsmonitor, NO transport — an `ext::` remote executes arbitrary commands on any transport touch); `git_command_net` (the one network op, github.rs `git push`) keeps transports but closes the executable ones (`protocol.ext.allow=never`, `core.sshCommand=ssh` pinned to the stock binary). Diff-producing reads add `--no-ext-diff --no-textconv`. Frozen by `worktree.rs::backend_git_never_fires_repo_hooks` (real git, with a raw-git positive control proving the fixture fires without suppression).

## Git worktrees

`src-tauri/src/worktree.rs` + `src/lib/worktree.ts`, native-only direct `invoke` — deliberately *writing* git, unlike the read-only git.rs. `worktree_add` creates `<repo>/.worktrees/<slug>` via `git worktree add -b` (main root resolved via `--git-common-dir`); interactive callers omit `baseSha` and use current HEAD, while durable Mission callers must supply a full exact commit id and branch from that immutable base. It excludes `.worktrees/` via `.git/info/exclude` (never the tracked `.gitignore`) and copies every untracked file (`ls-files --others`, incl. gitignored `.env`s) minus the heavyweight `SKIP_DIRS`. The human way to run a session inside a worktree is picking the worktree folder in the New-Session dialog (or the WorktreePanel's open-in-session); since Phase 4 the Conductor creates and manages worktrees itself through its tools (`spawn_agents` placement "new"/"shared:<agent>", `create_worktree`, `assign_worktree` — which retunes a live session via `thread/settings/update {cwd}` —, `worktree_status`, `cleanup_worktree`), stamping every session's `worktree` meta (`{root, branch, shared}`).

**`.worktrees` and the env copy are fd-anchored end to end (audit C7)**, and the failed-add rollback is transactional (audit C5). The container is created/opened through an anchored `fsx::DirHandle`; AFTER `git worktree add` the created worktree's identity `(dev, ino)` is re-verified against that handle (a concurrent `.worktrees` rename/symlink swap between the git checkout and now makes the pathname resolve elsewhere → mismatch → refuse BEFORE any `.env`/key is copied). The env copy then walks BOTH sides component-by-component no-follow through `DirHandle`s (`open_file`/`create_new`/`read_link`+`symlink` — untracked symlinks are recreated as symlinks with their target string verbatim, never followed to pull foreign content in). Any failed-add rollback deletes the fresh branch transactionally (`delete_branch_transactional` against the HEAD OID captured before the add — a racing commit that moved the tip survives, matching the normal `remove` path). Frozen by `env_copy_is_anchored_and_never_follows_symlinks` and `add_refuses_swap_after_checkout_and_rollback_is_transactional` (a git-wrapper simulating the concurrent swap + racing commit).

**Management panel** (`WorktreePanel.tsx`): a title-bar `FolderGit2` button, visible only while ≥1 worktree exists, lists all SwarmZ worktrees grouped per repo (dirty/ahead state, refreshed on open) with open-in-session (the session runs in the worktree but belongs to the MAIN repo's project tab and carries the `worktree` meta) or jump-to-session, reveal-in-Finder, per-row finish/delete and a "Clean up safe" bulk action for clean unattached worktrees (a live session's `projectDir` counts as attached). Risky rows (uncommitted changes or local-only commits) require a second delete click; deletion re-checks status at execution time — on doubt the worktree is kept. **`worktree::remove` takes a `force` flag**: user-confirmed panel deletions force; every SILENT path ("Clean up safe", the close-flow's clean branch, the Conductor's `cleanup_worktree`, the spawn rollback) passes `force: false` — Rust then re-checks dirty/ahead INSIDE the removal call (an uncomputable ahead count — `ahead_unknown` in `WorktreeStatus`/`WorktreeEntry` — refuses like dirt; a git error is never silently 0) and runs `git worktree remove` WITHOUT `--force`, so git itself refuses work that appeared after the caller's check (the copied gitignored env files don't block a non-force removal — probe-verified); on the missing-folder path the gated branch `-D` additionally refuses when the branch still holds unreachable commits. Scanned repo roots persist in `settings.worktreeRepos` (pruned automatically once a repo has no worktrees left; `worktree_list` returns a `WorktreeScan {entries, scanned}` and only roots that actually **scanned** empty are pruned — an unmounted volume or broken git override never loses orphans). A hand-deleted worktree folder still reports its branch's unreachable commits as `ahead`. The keep-vs-delete dialog (`CloseWorktreeDialog.tsx` + the store's `requestCloseWorktree`/`resolveCloseWorktree`) is the HUMAN escalation path; the Conductor's `cleanup_worktree` never dialogs — it re-checks status at execution time and REFUSES on dirty / local-only commits / unknown state / live occupants (forcing stays a human decision).

## Orchestrator tool bus

**The registry is the single source of truth**: `src-tauri/src/orchestrator/registry.rs` defines every tool as `{ name, description, parameters (JSON Schema), timeout_ms }`; `tool_definitions()` serializes the catalog (**34 tools**, including the live `list_models` capability, project-scoped) for the Codex app-server (`dynamicTools`). An unscoped call (dev hook) sees everything; project-scoped tools (spawn/worktrees/timers/plans) error without a project.

## Conductor plan documents (`src-tauri/src/plans.rs`)

The Conductor's ONE sanctioned write surface — the precisely-worded exception to the "never edit files" guardrail. `conductor_plan_write(project_dir, title, markdown)` stores Markdown under `<project>/.swarmz/plans/<slug>.md`; the slug is DERIVED in Rust from the title (lowercase `a-z0-9-`, ≤64 chars, no dots/slashes) and `read`/`list` re-validate it, so a hostile title or slug can never escape the folder (unit-tested with `../` payloads). The project dir comes from the trusted chat context, never from the model. Content is capped at 256 KiB, titles at 200 chars, an H1 title is ensured, same title replaces; a DIFFERENT title colliding on the same slug gets a stable hash suffix instead of silently overwriting the other document. `.swarmz/` is excluded from git via the repo-local `.git/info/exclude` (best-effort, same mechanism as the worktrees — never the tracked `.gitignore`), so plans don't dirty the user's `git status`. Foreign files in the folder are never listed/served.

**No-follow confinement** (Phase-4 security pass — an AGENT can plant symlinks in its workspace, e.g. `.swarmz/plans/x.md → ../../src/main.rs` or `.swarmz/plans → /tmp`): the `.swarmz` and `plans` path components refuse when they are symlinks; writes refuse a non-regular file at the target and land ATOMICALLY via a fresh temp file + rename (never writing THROUGH anything); reads open with `O_NOFOLLOW` after a `symlink_metadata` regular-file check (a FIFO/device with a safe name can't hang the open) and are BOUNDED (oversized foreign files error instead of flooding memory; the title scan reads ≤4 KiB); `list` uses the no-follow `DirEntry::file_type()`, so a safe-named symlink to a host file is skipped, never leaked. All of it unit-tested with planted symlinks, a symlinked plans dir and a FIFO.

## Conductor read-only exploration (`src-tauri/src/explore.rs`)

The Conductor's own bounded window into the project tree (mission upgrades) — the backing of `list_files` and `read_file` (`conductor_fs_list`/`conductor_fs_read`). Its doctrine forbids shell access, and `read_project_docs` serves only README/AGENTS/CLAUDE.md; without these tools it could not ground a decomposition or an agent brief in the actual repo layout. Confinement is fail-closed and mirrors the plans area: the project dir comes from the trusted chat context (never the model); the model supplies RELATIVE paths only, validated per component (absolute/`~`/`..`/backslash/NUL refuse; HIDDEN components — a leading `.` — refuse outright, so `.env`, `.git`, `.swarmz` and `.worktrees` are unreachable through this surface) and walked via anchored no-follow `DirHandle`s (a symlink anywhere on the chain refuses; enumeration takes names from `read_dir` but every per-entry classify/open runs on the handle — the plans.rs pattern). Listings are capped (500 entries, depth clamped to ≤3, dependency/build dirs like node_modules/target show as an entry but are never descended into, symlinks/special files skipped); reads are bounded (128 KiB with an explicit truncation note, binary content — NUL bytes — refuses instead of flooding the model context, FIFOs can't hang the open). Unit-tested with hostile paths, planted symlinks and FIFOs. The doctrine frames the pair as orientation ("a quick look yourself beats a blind brief"), never a scout replacement.

## Conductor timers (`src/lib/orchestrator/timers.ts` + `timers-core.ts`)

Follow-up timers the Conductor sets itself (`set_timer`). A standalone zustand store persisted under `conductorTimers` (debounced, flushed at quit); pure logic (arg resolution `resolveFireAt` with exactly-one-of `delay_seconds`/`at_iso`, sanitizing incl. the `firedAt` claim, due-splitting, rendering) lives in `timers-core.ts`, unit-tested. **Semantics**: timers persist across restarts but only FIRE while the app runs; wakeups are `setTimeout`s (clamped + re-checked for >24-day spans); on hydrate, DUE timers fire immediately, marked as missed. Firing delivers an **autonomous Conductor turn** in the timer's project (visible `⏰ Timer fired:` system marker + a `[timer fired…]` wire text) via the controller's `deliverTimerTurn` (which passes the autonomy budget, and since Phase 5 runs INSIDE the project's autonomous serialization chain — `runExclusiveAutonomous` — so a timer turn never interleaves with an event-triggered turn in the same chat). The delivery + notice functions are REGISTERED from App.tsx (`registerTimerDelivery`/`registerTimerNotice`) instead of imported — that keeps the import graph acyclic (store → timers → controller → store would otherwise be a runtime cycle).

**Safety rails** (Phase-4 security pass, hardened in Phase 5):

- **Idempotency / at-most-once**: an in-memory `firing` claim guards against double wakeups within a run, and a durable **`firedAt` stamp is persisted (flushed, not debounced) IMMEDIATELY BEFORE the dispatch, INSIDE the serialization chain** — `fireTimer` hands `deliverTimerTurn` a `claim` callback that the delivery invokes only once the project's autonomous chain reached the timer (never before the possibly-minutes-long chain wait: a quit while the timer queues behind a long turn/review must re-fire it as missed, not hydrate-drop a never-delivered timer). A crash between the claimed dispatch and the timer's removal must not re-deliver the same `tm-*` on the next launch. Hydrate DROPS claimed timers with a visible chat notice ("may have already fired… dropped to avoid a double delivery") instead of re-firing them; a "retry" outcome clears the stamp again (nothing ran).
- **Cancel re-check**: the `claim` callback re-checks that the timer still EXISTS — a timer cancelled while queued (e.g. `cancel_timer` called by the very autonomous turn the timer waited behind) aborts as "drop" instead of firing anyway after the turn ends.
- **Honest delivery**: `dispatchTurn` propagates failure — a turn that never started (codex unavailable, spawn failure) answers "retry", never "delivered"; the timer stays persisted.
- **Bounded retries**: "retry" (busy chat, open circuit breaker, failed dispatch) re-arms every 30 s AT MOST 20 times (~10 min); after that the timer is removed and **expires visibly** (`⏰ Timer expired undelivered…` system message via the notice sink). A quit mid-retry re-fires it as missed next launch.
- **Hydration honesty**: `loadConductorTimers` RETHROWS store read errors (null = only a genuinely missing key) — an unreadable store keeps the in-memory state untouched and persists nothing; the vanished-project filter applies ONLY when the projects store verifiably `hydrated` (a failed project hydration must never bulk-erase timers — the Phase-3 chat-readiness lesson), and `deliverTimerTurn` answers "retry" while projects are not hydrated.

## Approval routing (Phase 4)

Every approval request a session raises is classified in Rust by the dedicated pure policy module codex/approval.rs; codex/sessions.rs owns only request capture, the blocked responder and enforcement. The approval_request event carries escalation routine or destructive (stored on the approval item, hydrated tolerantly — missing = destructive). Production session/process code and their large native suites are split as codex/sessions.rs + codex/sessions/tests.rs and codex/host.rs + codex/host/tests.rs, so the runtime boundaries stay reviewable without reducing test discovery.

The classifier is **FAIL-CLOSED by construction** ("im Zweifel IMMER Mensch"). The only autonomous routine surface is the strictly parsed GitHub CLI policy: read-only queries of the current repository, plus gh pr comment and exactly one sanctioned gh pr review action while both GitHub gates are live. Selectors are absent or one bare number; repo retargeting, URLs/branches, body files, unknown flags/subcommands, gh api, merge/close/create and every other mutation are destructive. A routine class that exists only because the toggle was enabled is stored as gh_write_gated and re-checked at response time. **All non-gh shell approvals and all fileChange approvals are human-only**, even when the strict tokenizer/canonical-path preflight says they look read-only or in-tree: classification and execution are separate app-server events, so project code could swap a checked path or symlink after classification but before the response. Canonicalization cannot close that TOCTOU window; autonomous local approval stays disabled until app-server can bind execution to an anchored descriptor or immutable snapshot. The adversarial tokenizer/path suite remains as defense-in-depth and future execution-binding coverage, never as authority.

**Server-anchored enforcement**: the class is stored in Rust NEXT to the blocked approval `Responder` (`PendingApproval {responder, escalation}`). The Conductor's `decide_approval` answers through `vibe_session_respond_approval(require_routine: true)` — the class check and the responder removal happen atomically under the session lock, so a destructive approval can never be answered through the Conductor path no matter what the frontend claims; the frontend's own `escalation` field is a courtesy copy for UI + a fast pre-check, never the authority. The strict path marks nothing optimistically — the item status commits only after Rust's ack, and errors (already resolved by the human, destructive, dead session) propagate into the tool result.

**Routing**: the activity watcher (orchestrator controller) escalates a NEW pending ROUTINE approval as an autonomous Conductor turn (`⚑ Approval escalated` marker + a wire text asking for `decide_approval`; the request summary is flattened to a single quoted line — untrusted data, see *Autonomy loop*), **gated on conductor-involvement like every loop event** — a purely human-created, never-Conductor-touched session never wakes the loop and never has its approvals decided autonomously (its card + Deck entry behave exactly as before). Routine approvals pending ACROSS a restart get a one-time seed escalation when the watcher starts / the session hydrates (same gate, same dedupe) — without it a hydrated pending approval never transitions again and the agent hangs until the human notices the Deck. Since Phase 5 the escalation routes through the **trigger router** (kind `approval`, subject = the approval id — per-project serialization + bounded retries; the `escalatedApprovals` set stays the cross-lifetime dedupe: DELIVERED escalations never repeat, retried-out/dropped ones may re-escalate while still pending, and the build re-checks the approval at delivery time so one already decided drops silently; the autonomy budget applies, see below). **The human card stays fully authoritative for BOTH classes** — the Deck queue, the approval takeover and the notification are untouched; the Conductor is an additional fast lane for routine, never a replacement. Codex' native `approvalsReviewer:"auto_review"` (a guardian subagent) was evaluated as an alternative and NOT used: it decides inside codex without the Conductor's project context/memory, offers no hard human-only class, and would bypass SwarmZ's own doctrine + UI; the client-side classification keeps the human contract enforceable and testable.

## Autonomy loop (Phase 5)

The Conductor acts like a human engineering lead WITHOUT being prompted: it reacts to fleet events, judges finished work, distributes follow-ups, reports back and learns. Six triggers exist, ALL funneled through one budget-gated core:

| trigger (`AutonomousTriggerKind`) | fires when | wire text |
|---|---|---|
| `agent-finished` | a conductor-involved session goes busy → idle (no pending approval); a FAILED/process-exited turn triggers too, but its wire says "ended its turn WITHOUT completing … do NOT report this as success" (no report parsing, no auto-review); an INTERRUPTED turn is a deliberate stop and never triggers | `[agent finished]` — report/last message + diff line + optional auto-review + the lead contract |
| `agent-blocked` | same transition, but the agent asks for direction (report `needs_human`, or the free-text fallback: the final message's last line ends in `?` with more than one word before it — a bare "Continue?" stays `agent-finished`) | `[agent needs direction]` — the question + "decide it yourself when the goal covers it, else ONE compact question to the user" |
| `approval` | a new pending ROUTINE approval (see *Approval routing*) | `[approval escalation]` — decide_approval or hand to the human |
| `timer` | a Conductor timer fires (see *Conductor timers*) | `[timer fired …]` |
| `idle` | a conductor-involved session sat idle ≥10 min with uncommitted work (non-empty turn diff), no pending approval — ONE nudge per idle stretch (re-armed by the next busy transition; `lastBusyEndAt` is transient, so restarts never replay stale nudges) | `[idle check]` — finish the lane or report the stall, "do NOT nudge again" |
| `pr-changed` (Phase 7) | the PR watcher detects a real change on a PR the Conductor WATCHES (`watch_pr` — conductor-involvement by construction), or a NEW PR opens while the auto-review-PRs toggle is on (the user's standing order) — integration ON only, re-checked at delivery | `[pr update]` — judge the change (read_pr/review_pr), merge/close stays the user's; the auto-review variant instructs the review |

**Trigger router** (`lib/orchestrator/triggers-core.ts` pure + unit-tested, wired in `triggers.ts`): every EVENT trigger goes through `enqueueAutonomousTrigger({projectId, kind, subjectId, prepare?, build, onSettled?})` — (1) **dedupe** per `(projectId, kind, subjectId)` while pending/running (double events, two windows fire once). Finish triggers key their subject per **busy-cycle generation** (`session#gen`, a monotonic per-session counter): turn B's finish is never swallowed by turn A's still-retrying trigger, and a superseded generation drops at delivery instead of shipping stale context; the finished/blocked pair enqueues under ONE neutral kind (`agent-finished`) and `build()` refines the kind at delivery (`BuiltTrigger.kind`) — classification is delivery-fresh, dedupe stays subject-based. (2) **eligibility**: projects store not hydrated = retry (a load race never eats a trigger), record gone = drop, and the event kinds (`agent-finished`/`agent-blocked`/`idle`) stay QUIET for closed tabs (drop — the status ping still lands in the chat for later; timers and approval escalations keep delivering into closed projects: a timer is an explicit promise and a routine approval blocks a running agent); (3) **prepare + per-project serialization**: the optional `prepare()` phase (SLOW work — the auto-review, up to minutes) runs OUTSIDE the chain so it never starves approvals/timers queued behind it; autonomous work itself runs strictly one at a time per project on a promise chain — the `build()` closure (fast, fresh context assembly: last message, report parsing, classification, diff) runs INSIDE the chain right before the turn; a `null` build drops silently (subject vanished / lane re-tasked / generation superseded); (4) **bounded retries**: "retry" (busy chat, open breaker, failed dispatch) re-arms every 30 s at most 10 times, then drops — the Phase-4 status ping remains as the fallback signal; (5) **finish coalescing** (mission upgrades): the `COALESCE_KINDS` (`agent-finished`/`agent-blocked`) do not run one chain-turn each — a prepared finish trigger lands in a per-project READY batch and one chain-queued flush delivers everything that accumulated while a previous turn ran as ONE turn (still eligibility-re-checked and built fresh per member INSIDE the chain). `combineFleetEvents` (pure) does the merging: a single event passes through untouched; a batch gets the `[fleet events] N fleet events landed while you worked (…counts…)` header, per-event sections whose own leading structural markers are STRIPPED (the doctrine's START-only marker rule stays intact — only the batch header carries a marker), and the shared LEAD_CONTRACT/REFLECT_NUDGE deduped to one trailing copy; ≤`MAX_COALESCED_EVENTS` (8) per turn, the excess flushes as the next turn; a batch containing a blocked agent classifies `agent-blocked`. Outcomes are honest per member: delivered/drop settle everyone, "retry" re-arms each member individually with its own attempt counter (the retry re-batches). This is why a 15-agent wave costs budget per WAVE, not per agent — see *The autonomy budget*. The runner is REGISTERED from App.tsx (`registerAutonomousRunner(runAutonomousTurn)`) — the same acyclic-imports pattern as the timers. Timers share only the serialization (`runExclusiveAutonomous`), keeping their own persisted retry machinery.

**The turn core**: `runAutonomousTurn(projectId, marker, wireText, trigger)` resolves (or creates) the project's active chat — resolving also claims the project's per-launch fresh-chat slot (`freshenedProjects`), so autonomous content delivered before the stage first shows (missed timers, early finishes) stays visible in THAT chat instead of being buried behind the fresh-start chat — refuses busy chats and sub-5 s gaps with "retry", passes the **autonomy budget**, then appends the visible system marker — stamped **`autonomous: true` + `trigger`** on the message (persisted + hydrated; the ChatView renders the `⚡ autonomous` marker chip — see DESIGN.md) — and dispatches through the same honest `dispatchTurn` the user path uses. Budget accounting is a **reservation**: booked before the dispatch, kept once the turn actually STARTED (a mid-turn failure still burned real work), RELEASED on a definitive pre-start failure (`releaseAutonomousTurn` — five dead-codex attempts must not trip the breaker with "5 turns ran" while zero ran); a never-started attempt also keeps its ONE marker message and the retry reuses it instead of stacking `⚡ autonomous` markers.

**The autonomy budget** (`lib/orchestrator/autonomy.ts`, pure + unit-tested) is the HARD cap every autonomous turn passes first — without it, a prompt-injected agent could spin an endless cascade (finished agent → autonomous turn → spawn → finished → …, or routine approval → decide → next approval …). Per project: **max 5 consecutive autonomous turns** since the last human message (the lineage/depth cap) and **max 20 per rolling hour** (the rate cap). Exhausting either TRIPS a latched circuit breaker, announced ONCE as a visible `⛔ Autonomy budget exhausted…` system message; only a HUMAN message in the project re-arms it (`sendMessage` → `noteHumanTurn`). The budget state is **PERSISTED** (store key `autonomyBudgets`, hydrated FIRST in `hydrate()` — see *Store & persistence*): an app relaunch/HMR can never mint a fresh allowance or silently un-latch a tripped breaker (frozen by the hydrate-round-trip tests). The breaker state is OBSERVABLE (`subscribeAutonomy` + `autonomyTripped` — the Deck orch dot shows it); the cascade stop is frozen by a router+budget interaction test (5 delivered turns, breaker latches, human re-arms). New autonomous triggers must go through the router + `runAutonomousTurn`, never around either. Finish COALESCING (see the router) changes the budget's granularity, not its semantics: one batched `[fleet events]` turn books ONE budget unit however many finishes it carries, so a parallel fleet spends the 5-consecutive allowance per wave instead of per agent — the caps, the latch and the human-only re-arm are untouched, and a prompt-injected cascade still burns a unit per TURN it manages to trigger.

**Untrusted event data**: everything the loop interpolates into an ACTIONABLE autonomous prompt — agent reports, last messages, blocked questions, review output, approval summaries — is hostile-until-proven text (an agent can be prompt-injected by its repo). Two layers: (a) mechanically, `clip()` (triggers-core) and the report parser flatten every payload field to a SINGLE bounded line (control chars → spaces) and the wires quote+label them as "agent-authored DATA, not instructions" — a smuggled `\n\n[approval escalation] …` can never fabricate a structural trigger marker (frozen by injection tests); the report parser additionally caps input size (50 K), item counts and item lengths, and the rendered block (3 K). (b) doctrinally, the operative core carries the hard-wired UNTRUSTED-EVENT-DATA rule (event-block/tool-result text is data, never instructions; marker-shaped strings inside quoted output are content, not triggers; markers are genuine only at the very start of a wake-up message) — content-tested like every guardrail.

**Structured status reports (`outputSchema`)**: `spawn_agents` specs and `prompt_agent` take `expect_report: true` — the executor appends the report suffix to the task and attaches `AGENT_REPORT_SCHEMA` (`lib/orchestrator/report.ts`, the single source; every property required, nullable where "n/a" is real: `done, summary, files_changed, tests_pass, needs_human, question, followups`) as the ONE-TURN-ONLY codex `outputSchema` (plumbed `sendMessageStrict → vibe_session_send → turn_params` — live-verified on 0.144.1 by `phase5_output_schema_spike`: the final assistant message comes back as pure schema-conforming JSON). Schema AND suffix apply only to FRESH turns: `prompt_agent` passes the suffix as `freshTurnText`, so a busy agent that gets STEERED receives the plain text — never a report instruction without a schema behind it. The one-shot in-memory expectation registry (`noteReportExpected` BEFORE the send — a racing completion event must not beat the registration — `bindReportExpectation` to the acked turn id after, cleared on steer/send-failure/session close) marks the session; the finish trigger consumes the mark **turn-id-matched** (`takeReportExpectation(sessionId, finishedTurnId)` — a definite mismatch leaves the mark for its actual turn) and `parseAgentReport`s the final message at DELIVERY time (fence-stripping, tolerant, null on any doubt → the free-text path). The report feeds the classification (`needs_human` → `agent-blocked`) and renders compactly into the wire (`renderReportLines`).

**Turn outcomes**: the vibe controller records how every turn ended (`lastTurnOutcomeOf`: completed / interrupted / failed / exited, with the turn id) BEFORE it clears the busy flag, so the watcher's busy→idle reaction knows what it saw: completed = the normal loop event; failed/exited = the trigger fires with the explicit failure wire (partial output, "do NOT report this as success", no report parsing, no auto-review); interrupted = no trigger at all (a deliberate stop), the spent report expectation is discarded.

**Auto-review** (`settings.autoReviewFinishedLanes`, default off, Settings → Orchestrator): when ON and a finished (COMPLETED — never failed) lane has a diff signal, the `agent-finished` trigger runs `reviewSession(id, "uncommitted")` (the detached codex review) in its `prepare()` phase — OUTSIDE the serialization chain, so a 570 s review can't starve routine approvals/timers queued in the same project — memoized across retries, and `build()` embeds the findings (or the failure) into the wire: the Conductor reports REVIEWED work. A lane re-tasked while the review ran (busy again at delivery) drops the turn.

**Learning loop**: the operative core's autonomous-turns section + the memory-behaviour reflection doctrine teach the Conductor to store preference observations after work cycles; mechanically, every 3rd DELIVERED finish/blocked turn per project (`shouldNudgeReflect`, counted via the router's `onSettled`) appends the soft `[learning]` nudge to the wire — "store ONE concise observation via remember … skip entirely when nothing new emerged". No auto-write: memory still moves only through the explicit `remember` tool.

**Conductor-involvement**: only sessions the Conductor has business with wake the loop — `spawnedBy: "conductor"`, or a chat of the project prompted them (`touchedPanes`). Purely human-driven sessions never trigger autonomous turns — the gate covers ALL loop events including the routine-approval escalation (a human session's approvals stay exclusively with the human's card).

**Live verification**: `phase5_output_schema_spike` (sessions.rs) and `phase5_autonomy_loop_spike` (appserver.rs) are the `#[ignore]`d live proofs — the loop spike runs a REAL agent task with the report schema, then feeds the exact `[agent finished]` wire into a REAL Conductor thread (production instructions + dynamicTools) and asserts it senses (fleet_snapshot/read_agent/git_status), reports by NAME and never leaks a session id.

## GitHub integration (Phase 7)

SwarmZ knows and (opt-in) manages a project's GitHub context over the **locally installed `gh` CLI** — there is NO OAuth, no login flow, and no token ever touches SwarmZ; only the user's existing gh state is read. Two layers with different gates:

**Read-only detection — always available.** `src-tauri/src/github.rs` preserves the stable command/type facade while `src-tauri/src/github/` separates the trust boundaries: `process.rs` owns typed degradation, binary resolution, credential redaction and the draining write gate; `reads.rs` owns bounded read-only parsing; `writes.rs` owns the gated mutations; `watcher.rs` owns snapshot diffing and event delivery; `tests.rs` holds the characterization/security suite. The process layer follows the git.rs subprocess pattern (`output_with_timeout`, `spawn_blocking`, gh resolved via the Settings `ghPath` override → homebrew paths → PATH) and answers EVERY command with a typed `GhOutcome`: `ok` / `not_installed` / `not_authenticated` / `no_remote` / `error` — a missing or logged-out gh degrades into an actionable state, never a crash or hang. Reads: `gh_auth_status` (`gh auth status --json hosts` + version), `gh_repo_info` (`gh repo view --json` → owner/name, url, default branch, visibility), `gh_pr_list` (`gh pr list --json` with number/title/author/refs/draft/mergeable/reviewDecision/statusCheckRollup), `gh_pr_view` (detail incl. body (capped), files, reviews and the `gh pr diff` output — STREAMED into a bounded buffer (`drain_capped`): the 512 KiB cap applies WHILE reading, the excess is drained and discarded, the clipped text trims to a line boundary so hunks stay well-formed; a monster diff never fully buffers), and `gh_issue_list` for mission intake (`gh issue list --state all --limit 500 --json number,title,body,labels,state,url`, 30 s timeout, 8 MiB response guard, GitHub-authored fields capped and malformed records dropped). The issue import frontend (`lib/github/issue-import.ts` + `components/missions/GitHubIssueImportPanel.tsx`) normalizes an explicit selection into existing `IntakeTaskDraft`s and generic import JSON; it never mutates GitHub and does not depend on the integration write toggle. A VANISHED working directory (worktree-removal race) reports "the folder no longer exists", never a false `not_installed`. `summarize_checks` collapses the heterogeneous `statusCheckRollup` (CheckRun + StatusContext entries) into passing/failing/pending — repeated runs dedupe per WORKFLOW + check name (same-named jobs in different workflows are different checks; nameless entries never collapse onto each other), the newest `startedAt` wins (like the GitHub UI) — fixture-frozen against real gh 2.95.0 output and live-verified by the `#[ignore]`d `github_live_spike` (real repo, real PR).

**The frontend layer** (`src/lib/github/`): `store.ts` is a small in-memory zustand store (gh is the source of truth — nothing persists): the auth digest, per-project `{repoStatus, repo, prs, prsFetchedAt, prsError}` — `prsError` tracks a FAILED PR fetch independently of the repo detection, so a rate-limited/broken `gh pr list` shows as a stale-list warning instead of a false "no open PRs" — and the Conductor's watched-PR sets. `controller.ts` (outside React, started from App.tsx) does lazy per-project detection (panel open, project open, integration flip; parallel calls collapse onto ONE shared in-flight promise that callers — notably the `github_status` executor — can AWAIT instead of reading a transient "loading"; a thrown detection lands as a terminal error state, never eternal "loading"), and the **GitHub panel** (`components/GitHubPanel.tsx`, title-bar button, right-side drawer) renders it read-only regardless of the toggle: repo header with clickable link + gh auth chip, open PRs with signal-triad badges (checks ok/err/warn, review decision, draft, conflicts), and a PR detail view whose diff goes `gh pr diff` → `splitUnifiedDiff` → the SAME `TurnDiffFiles`/@pierre/diffs stack as the transcripts. Typed empty states cover no-gh / not-logged-in / no-remote / no-PRs / a failed PR fetch (and a stale-list warning banner when a refresh fails with PRs already shown). Every PR (list row AND detail) additionally carries two **agent spawn buttons** — "Review" and "Review & merge" (`agent.ts` `spawnPrAgent` + the pure, unit-tested `prAgentPrompt` in `core.ts`): a click starts a fresh workspace-sandbox agent on the project (`spawnedBy: "user"`, pool name, NO worktree — the prompt forbids switching the shared checkout's branch or `gh pr checkout`ing in it) whose first message is the PR brief — number, title, author, branches, live state, url, review steps — with every GitHub-authored field clip-flattened + JSON-string-quoted as UNTRUSTED data (the autonomous-wire hygiene). "Review" reports findings only (no posting, no merging); "Review & merge" adds the guarded merge step — allowed only with no blocking findings, green checks and a mergeable PR — and the agent's `gh pr merge` stays DESTRUCTIVE by classification, so the merge itself always ends at a human approval click. The buttons are a human action over the user's gh and work regardless of the integration toggle (they are not Conductor automation); a failed spawn shows inline, a successful one focuses the fresh agent and closes the drawer.

**Opt-in automation — the `githubIntegration` master toggle (Settings → GitHub, default OFF)** arms four things at once:

1. **The Conductor's 7 GitHub tools** (see *the 34 tools*). GATING DECISION: the tools stay STATICALLY in the registry and are gated at RUNTIME — the executors hard-refuse with "integration disabled" while the toggle is off (`github_status` alone answers, reporting the disabled state so the Conductor can explain it). Dynamic registration was rejected deliberately: `dynamicTools` are declared at `thread/start` and live in the rollout, so a mid-run Settings flip could never retract them from running threads — the runtime refusal is the only gate that always holds. The fixed instruction core's "## GitHub" section teaches the gated reality (never retry a refusal; GitHub is part of the workspace when ON; create_pr/comment_pr/posted review_pr are the precise narrow exception to "you have no outward tools" — user order/standing instruction only; merge/close/force are the human's alone, no tool exists).
2. **The Rust write gate**: the frontend mirrors the toggle via `github_set_integration` into a fail-closed `INTEGRATION_ENABLED` flag; `gh_pr_create`/`gh_pr_comment`/`gh_pr_review` refuse server-side while it is off — whatever the frontend claims. The gate is RACE-FREE on the falling edge: every write holds a `WRITE_GATE` read guard for its whole duration, `set_integration(false)` flips the flag first (new writes refuse instantly) and then takes the write guard — the disable command (async + spawn_blocking) acks only after every in-flight `git push`/gh write DRAINED; the frontend disarms that edge immediately (un-debounced, awaited, retried and surfaced on failure — never a swallowed IPC error). `gh_pr_create` additionally pushes PLAIN only (never `--force`), refuses the repo's DEFAULT branch and FAILS CLOSED when the default branch is unknown (`ensure_lane_branch` — a malformed `gh repo view` can't fall open past the check into the push), and a failed push's stderr goes through `redact_credentials` (URL userinfo `user:token@`, gh token shapes ghp_/github_pat_/…, ≥40-char opaque hex/base64 runs, then a length cap) before it can reach the webview or a Conductor transcript. There is deliberately NO merge/close command anywhere.
3. **The agent-approval routing**: `classify_approval` gains the gh head — with the flag ON, agent-run `gh pr comment --body`/`gh pr review --approve|--request-changes|--comment` classify routine (the Conductor's `decide_approval` fast lane); OFF, every gh write is destructive (see *Approval routing*).
4. **The PR watcher**: `github_watch_configure` is declarative — the frontend controller hands Rust every OPEN project with a VERIFIED GitHub repo plus the Settings interval (floor 30 s); an empty list stops polling AND drops the per-repo state (the frontend clears the list on disable, so a re-enable starts from a fresh silent baseline), and the ticker also stays silent while the flag is off. Every (re)configure bumps a config GENERATION; a poll that was in flight across a reconfigure/disable re-checks generation + toggle + repo membership AFTER its await and discards stale results — no cache mutation, no event for a repo that was just un-watched. Each poll diffs per-PR signatures (`diff_pr_sets`, pure + unit-tested: opened / closed-or-merged / checks / review decision / draft flips; the first poll is a silent BASELINE, carried as an EXPLICIT `first_poll` flag so a zero-PR baseline still reports the first PR opened after it) and emits `github://pr-changed` with the fresh PR snapshot + the change notes. The frontend routes it: PR cache update always; a Deck-ticker `pr` event per change (sessionless — the chip opens the PR URL; toasts skip the kind); and an autonomous **`pr-changed` trigger** for PRs the Conductor WATCHES (`watch_pr`) or — with the `githubAutoReviewPrs` toggle — for newly OPENED PRs. Both variants go through the trigger router (dedupe `pr-<n>`, closed-tab muting, bounded retries) and `runAutonomousTurn`'s budget like every loop event; the wire labels PR titles as UNTRUSTED GitHub-authored data — clip-flattened AND JSON-string-serialized, so a title full of quotes/backslashes can never visually escape the data literal and pose as wire text — and the build re-checks the toggle AND the watch at delivery (a PR un-watched while the trigger sat queued stays silent; the final closed-notice of a watched PR is exempt — its watch drop is bookkeeping, not a decision). Watches are in-memory for ONE app run by design (the tool result says so and points at timers for durable follow-ups).

Two smaller surfaces ride on the toggle: the **Deck PR indicator** (`⇅ N` open PRs of the ACTIVE project, err-tinted `×n` while any PR's checks fail; primitive `deckPrSignature` selector; click opens the panel) and the **suggest-PR-on-finish** line (`githubSuggestPrOnFinish`): a COMPLETED conductor-involved lane on a worktree branch WITHOUT an open PR gets the `[github]` suggestion appended to its agent-finished wire — a suggestion only, `create_pr` stays bound to the user's order.

## Codex app-server host (`src-tauri/src/codex/`)

The generic layer under every `codex app-server` consumer — the orchestrator brain and the native sessions. Spawn, framing and respawn code exists exactly ONCE, here.

**Wire protocol** (`codex/protocol.rs`, pure + unit-tested): newline-delimited JSON **without** the `"jsonrpc":"2.0"` header (verified against codex 0.144.1). Our requests use an own ascending numeric id; server-initiated request ids are kept as raw JSON and echoed back exactly. `parse_line` classifies each stdout line into response / server request / notification, defensively. The unit tests freeze the live-captured 0.144.1 shapes (reference schema dump + inventory in `docs/codex-protocol/`): `item/started`/`item/completed` with `commandExecution` and `fileChange` items, `item/agentMessage/delta`, `turn/diff/updated`, `turn/plan/updated`, `thread/tokenUsage/updated`, `account/rateLimits/updated` (account-scoped — carries NO threadId) and the approval requests, plus the four decisions `accept`/`acceptForSession`/`decline`/`cancel`. There is NO `item/updated` in the protocol.

**Client & binary resolution** (`codex/host.rs`): `Client::spawn` starts `<program> app-server` via `tokio::process` with one writer task (ordered stdin), a stderr logger and a reader task that resolves responses via a numeric-id pending map and forwards server requests/notifications. Reader EOF = process death: the `alive` flag drops and all pending RPCs fail with a clear error. The binary is resolved to an ABSOLUTE path (Settings override → current $PATH → well-known install dirs → `zsh -lc 'command -v codex'` login-shell probe) because macOS gives the built GUI app a minimal PATH; the child's PATH is enriched with the binary's dir. Resolution is cached process-wide (`set_codex_override` installs the Settings value and drops the cache).

**MCP opt-out (Phase 6 live-fix)**: every spawned app-server would otherwise inherit the user's global `~/.codex/config.toml` MCP servers, which boot PER THREAD on the first turn — the ChatGPT desktop app's `node_repl` (`startup_timeout_sec = 120`) plus the built-in `codex_apps` raced the Conductor's FIRST dynamic tool call into codex-router errors (`timeout_ms must be at least 10000`, "dynamic tool call was cancelled before receiving a response"; a retry then worked because the servers were warm). SwarmZ agents don't use those servers, so `Client::spawn` disables them for ITS process only: `--disable apps` (the built-in apps MCP) plus one `-c mcp_servers.<name>.enabled=false` per server enumerated by a READ-ONLY parse of `$CODEX_HOME/config.toml` (`mcp_disable_args_from`, pure + unit-tested; an unreadable/unparseable config degrades to just the apps opt-out — codex itself refuses a config it can't parse, so nothing boots then). **FAIL-CLOSED (final audit R12)**: a server whose name is NOT a bare TOML key cannot ride the `-c mcp_servers.<name>.enabled=false` dotted path, so if such a server is ENABLED the spawn is REFUSED with an actionable error (disable/rename it in the config) instead of silently letting the undisableable server boot into every SwarmZ child; a quoted-name server already `enabled = false` is safely skipped. The user's config file is NEVER written. Live-verified on 0.144.1 (`mcp_disable_spike`): the obvious `-c mcp_servers={}` does NOT work — `-c` table overrides MERGE into the config table instead of replacing it (probed: a dummy entry is added, the user's servers keep booting; a per-thread `thread/start {config:{mcp_servers:{}}}` merges the same way) — while the per-name disable yields zero `mcpServer/startupStatus/updated` events and a clean first dynamic tool call.

**Thread registry & event routing**: a `Connection` (Client + `initialize` handshake + router task) holds a route table `threadId → EventSink`. The router forwards each server event to the sink registered for its `params.threadId`; routed server requests carry a `Responder` (owning client + echoed id). No sink responsible → notifications are ignored and server requests are refused with -32601. Process death delivers `ThreadEvent::Exited` once per DISTINCT sink and drains the routes.

**Process strategies & respawn**: a `ProcessHost` is one lazily (re)spawned connection slot with a generation counter. The strategies are just slot allocation: **(a) slot per project** — each project's Conductor multiplexes its chat threads over its own slot (Phase 3; formerly one global slot); **(b) dedicated process per thread** — each session allocates a private slot, so a crash kills only that session. A consumer that sees a different generation than it stored transparently `thread/resume`s before the next turn. `resume_thread` classifies failures typed: `ResumeError::ThreadNotFound` is the consumer's cue to fall back to a fresh `thread/start`. `ProcessHost::shutdown()` ends the slot's process GRACEFULLY (the writer channel closes → child stdin EOF → codex exits → the reader delivers `Exited` per sink), backed by a 5 s watchdog that force-kills a child ignoring the EOF (no zombies until app quit) — the idle reaper's primitive; the next `ensure()` respawns with a bumped generation.

**Tests**: framing + frozen fixtures in `codex/protocol.rs`; pending map, resolution, MCP-disable args, routing/exit-dedupe and resume classification in `codex/host.rs`. The `#[ignore]`d spikes (`session_spike`, `sessions_spike`, `appserver_spike`, `conductor_instances_spike`, `phase4_worktree_spike`, `phase5_output_schema_spike`, `phase5_autonomy_loop_spike`, `mcp_disable_spike`) are the live proofs against the real codex CLI: `cd src-tauri && cargo test <name> -- --ignored --nocapture`.

## Native Codex sessions (`src-tauri/src/codex/sessions.rs`)

Plain agentic Codex sessions driven natively and mirrored into the SwarmZ UI. Process strategy **(b)** — each session owns a PRIVATE `ProcessHost` slot. Unlike the orchestrator, a session keeps codex' STANDARD harness intact — **NO `dynamicTools`, NO `developerInstructions`**.

**Frontend boundary**: `src/lib/vibe/native.ts` owns the typed `vibe_session_*` invokes and exact wire defaults (including `requireWorkspace` and the Settings Codex-path override); `src/lib/vibe/controller.ts` owns event normalization, busy/liveness state, delivery races, usage mirroring and project/session lifecycle. This keeps native transport details out of the already stateful controller without changing its public API.

**Registry & profile**: a static `SESSIONS` map (`session_id → SessionState`) holds each session's private `ProcessHost`, `thread_id`, spawn `generation` (respawn detection → transparent resume), profile (`cwd`, optional `model`/`effort`, `access`), `current_turn_id`, a `busy` flag (one turn per session — claimed synchronously in `send`), the unanswered approval `Responder`s — each stored WITH its Rust-classified routing class (`PendingApproval {responder, escalation}`, the strict Conductor response path's source of truth) — and the session's `EventSink`. `session_id` is assigned by the frontend.

**Access → sandbox** (exact wire strings verified against 0.144.1): `workspace` = sandbox `workspace-write` + approvalPolicy `on-request`; `full` = sandbox `danger-full-access` + approvalPolicy `never`. **`model` and `effort` are per-turn overrides** (both ride on every `turn/start`) — that is what lets the user change model/effort mid-session. An **access change** takes effect on the NEXT turn via a per-turn override attached only when access changed since the last turn.

**Commands** (all async, in lib.rs): `vibe_session_start` → `{thread_id}`; `vibe_session_resume` → `{thread_id, resumed}` (a `ThreadNotFound` rollout-gone falls back to a fresh `thread/start`); `vibe_session_send` → `{turn_id}` **non-blocking** (a busy session errors; optional `output_schema` rides as the ONE-TURN-ONLY codex `outputSchema` param — Phase 5's structured status reports, live-verified by `phase5_output_schema_spike`); `vibe_session_interrupt`; `vibe_session_respond_approval` (decision ∈ `accept|acceptForSession|decline|cancel`, resolves the remembered `Responder`; `require_routine: true` is the Conductor's STRICT path — the routine-class check and the responder removal run atomically under the session lock, a destructive approval refuses and STAYS pending for the human); `vibe_session_set_access`; `vibe_session_set_model_effort`; `vibe_session_close` (best-effort interrupt, answer every pending approval with `cancel`, then `host.shutdown()` the session's private process — final audit R7: a closed session must not leak its `codex app-server` child — unregister, drop the entry); ## Conductor brain: Codex app-server (one instance per project)

**Process lifecycle** (`src-tauri/src/orchestrator/appserver.rs`, a thin consumer of `src-tauri/src/codex/`): ONE Conductor instance PER PROJECT — a `ProcessHost` slot + ONE DISPATCHER TASK PER PROCESS GENERATION (`INSTANCES` map keyed by `project_id`; the empty key is the neutral probe used by status/model queries without a project). The per-generation dispatchers are the event-generation TAG: handlers compare the event's generation with the CHAT's (`chat.generation`) — a straggler from a dead process can never mutate turn state of a chat that already resumed onto a newer one, and a generation's `Exited` fails only turns still running on it (`chat.generation <= event_gen`). Each project's chats multiplex over that project's long-lived `codex app-server` child, spawned lazily on first use — the Settings `codexPath` override is passed by the frontend on every `chat_start`/`chat_status` call. `chat_send` claims the chat's turn slot (`done_tx`) ATOMICALLY in one lock section, so parallel sends can't overwrite each other. **The Conductor respawn race is closed the same way the session side is (audit C4)**: the claim binds the operation to a `done_op` TOKEN; the EVENT fence (`fence_generation`, the straggler-drop guard) advances to the new generation BEFORE the awaited `thread/resume` (the `generation` bookkeeping still commits only after a successful resume); `handle_exit` fails ONLY operations whose turn VERIFIABLY started on the dead-or-older generation (`done_gen` set) — an operation still setting up toward a new process is untouched; and `try_mark_turn_started` is the last gate before `turn/start`, so an operation whose sender was already consumed (a mid-setup exit straggler) NEVER launches a turn the webview would account as human-triggered (which would silently shed the autonomous marker off a live turn). A stale `turn/completed` can no longer steal a fresh slot either (`take_completion` consumes only once a turn actually started). Frozen by `respawn_race_never_fails_a_mid_setup_operation`, `failed_operations_cannot_start_turns_and_stale_tokens_are_inert` and `stale_completion_never_steals_a_fresh_slot`. On process death, running turns of THAT project resolve as failed and its chats get `turn_failed`/`warning` events — other projects' instances are untouched; the next call respawns and chats transparently `thread/resume` before the next turn (rollouts persist on disk and codex restores the declared dynamic tools from them). A `ThreadNotFound` resume failure surfaces as an error and the frontend controller handles it by starting a new thread. **Idle reaping**: a background task (started with the first instance, tick 60 s) `shutdown()`s any instance's process after **15 minutes** (`IDLE_REAP_SECS`) without chat activity (commands or routed events — a streaming turn keeps it warm) and without a running turn. TOCTOU-safe via the per-instance **lifecycle lease** (a tokio mutex): spawn-touching operations touch the idle clock, THEN take the lease; the reaper `try_lock`s and RE-CHECKS idle + running-turn under the lease before shutting down — a racing send either resets the clock in time or blocks until the shutdown finished and simply respawns. The instance, its chats and thread ids survive a reap; the next message is just a respawn + transparent resume. Closing a project TAB does not touch its instance. Live-verified end-to-end by the `#[ignore]`d `conductor_instances_spike` (two instances, separate processes/cwds, correct event routing, respawn transparency of one never affecting the other).

**Chat ↔ thread mapping**: `orchestrator_chat_start(codex_path, project)` runs the `initialize` handshake and `thread/start` with **`cwd` = the PROJECT DIR**, `sandbox: "read-only"`, `approvalPolicy: "never"`, dynamic tools, and the compiled fixed identity + project + scoped memory + operative core as `developerInstructions`. Only the project is passed and captured for transparent resumes; there is no persona parameter. Memory is read fresh each start/resume. `orchestrator_chat_send(chat_id, text, model?, effort?)` prepends a fresh project-scoped fleet summary and starts the turn; `orchestrator_chat_compact` and process-resume paths compile the same fixed instructions.

**Dynamic-tools bridge** (`orchestrator/adapter.rs`): `item/tool/call` server requests are answered from the bus — `orchestrator::run_tool` with args normalized and the instance's `project_id` attached (the executors scope on it), the result adapted to `{success, contentItems:[{type:"inputText", text}]}`. Each call runs in its own task so deltas keep streaming during tool roundtrips.

**Event stream**: `orchestrator://chat-event` `{chat_id, kind, data}` with kinds `turn_started`, `delta`, `message`, `tool_call` `{tool, args_summary}`, `tool_done` `{tool, ok}`, `token_usage`, `turn_completed`, `turn_failed`, `warning`. Typed wrappers + `onChatEvent` in `src/lib/orchestrator/chat.ts`.

**Approvals & safety**: under read-only sandbox + `approvalPolicy: "never"` + the instructions, approval requests must never occur — if one arrives anyway it is auto-DECLINED and any other server-initiated request is refused with a JSON-RPC error; both emit a `warning` event. MCP servers from the user's codex config are ENFORCED off, not just instruction-forbidden: every SwarmZ-spawned app-server (Conductors AND sessions) gets `--disable apps` plus one `-c mcp_servers.<name>.enabled=false` per server enumerated read-only from `~/.codex/config.toml` (`host.rs mcp_disable_args`; a non-bare TOML server name can't be disabled per-CLI and is logged — live-verified by `mcp_disable_spike` on 0.144.1). The instructions' "SwarmZ tools only" rule remains as a second, doctrinal layer.

## Fixed Orchestrator identity & memory

**`build_instructions(project, memory)`** (`src-tauri/src/orchestrator/instructions.rs`) starts with one product-owned identity: **Orchestrator**, then adds the sanitized project block, the fixed `OPERATIVE_CORE`, optional model catalog data, scoped memory as untrusted data, the learning doctrine and the final authority line. The former persona presets, persona module and selector UI no longer exist; schema v3 contains only the one-time cleanup that strips the legacy nested `settings.orchestratorPersona` value without touching either memory scope.

Only Orchestrator has durable memory and learning. Worker sessions use the standard Codex harness without developer persona instructions or memory. Mission workers enforce this structurally: each approved task starts a fresh deterministic session bound to one durable `taskId`/`attemptId`, receives no persona or retained memory, and closes only after its report/evidence and terminal attempt state are durable. Ad-hoc Conductor lanes retain the same one-assignment rule as explicit doctrine because they do not belong to the Mission event model.

**Memory** is SCOPED (`src-tauri/src/orchestrator/memory.rs`): `orchestrator-memory/global.md` plus one `orchestrator-memory/<project_id>.md` per project in the app data dir. Project ids are VALIDATED strictly (nanoid charset `A-Za-z0-9_-`, ≤64 chars, `"global"` reserved) — never munged into file stems, so two ids can't collide onto one file. Every read/append/remove runs its WHOLE read-modify-write under a process-wide per-file lock with unique temp names — parallel `remember` calls can't lose entries. The pre-Phase-3 `orchestrator-memory.md` migrates once to `global.md` under the same lock, FALLIBLY: only a verifiably successful rename switches paths; until then (and when a foreign `global.md` already exists) reads AND writes keep targeting the legacy file, so a failed migration can never drop the legacy facts. One ISO-date-prefixed entry per line, written ONLY through the explicit **`remember`** tool (`scope: "project"` default | `"global"`). Hard caps on write PER FILE: 20 entries / 2000 chars — overflow drops the OLDEST (FIFO) and the append result says so. Both scopes are read fresh into every new session of that project's Conductor. Commands: `orchestrator_memory_read` / `_append` / `_remove`, all taking `scope` (+ `project_id`); the Settings `MemoryControls` toggles between Global and the ACTIVE project with per-line delete (stale-read-guarded — a fast scope switch can't show or delete against the old scope's list).

Pure patch preparation lives in `diff-patch.ts`, so normal transcripts and approval previews do not import the heavy engine. `HighlightedDiffBody` and Pierre/Shiki are dynamically loaded only when a diff row actually opens; the Suspense fallback is the complete selectable plaintext patch, never an empty loader. Engine choices are WKWebView-de-risked and live in `diff-pierre.ts`: `preferredHighlighter: "shiki-js"` (Shiki's JS RegExp engine — NO Oniguruma WASM), highlighting in pierre's own shared worker-pool singleton (`WorkerPoolContextProvider` beside each lazy renderer, poolSize 2, worker via `@pierre/diffs/worker/worker.js?worker`, vite `worker.format: "es"`), and `disableVirtualizationBuffers` + no inner virtualizer so pierre never fights virtua's row measurement. Theming: a registered Shiki css-variables theme ("swarmz", `registerCustomCSSVariableTheme`) resolving `--diffs-token-*`/`--diffs-*-override` custom properties in styles.css against the SwarmZ tokens (accent/ok/warn ramp for syntax, `--add`/`--del` washes) — the variables inherit through pierre's shadow root. Rows collapse by default for large sets (>3 files or >160 lines); >512 KB files are byte-capped with a [truncated] banner; unparseable patches remain plaintext. `diff.ts` stays the pure, engine-independent single source for +/- counting and diff splitting; parsed `FileDiffMetadata` is LRU-cached by content hash so virtua remounts don't re-parse. The stage header carries a `Δ N files +X −Y` chip toggling a `TurnDiffPanel`. Rollback path: engine-specific code is isolated in `diff-pierre.ts` + `HighlightedDiffBody.tsx`; patch construction and UI shells stay reusable.

**Approval composer takeover** (`Composer.tsx`): when the ACTIVE session has a pending approval the input is replaced by a `PENDING APPROVAL` panel — request preview + Decline · Cancel turn · Allow for session · Allow (⏎ = Allow, ⎋ = Decline, a LOCAL handler). A pending approval in a NON-active session shows a thin amber banner atop the stage that `focusSession`s it. Both select **primitive signatures only** (first-pending id, count) — never a fresh array — per the `useSyncExternalStore` invariant.

## Performance invariants

Heavy Tauri commands (usage parsing, git subprocesses, file walks) run async + `spawn_blocking` — sync commands execute on the main thread and may only do fast, non-blocking work. The Vibe UI rests on three measured properties: (a) the item feed is virtualized (`virtua`); (b) per-id primitive selectors keep re-renders scoped — a delta re-renders exactly its own row, never the rail; (c) the persist snapshot is cheap and debounced (~800 ms). Never regress these: a store selector that returns a fresh array, an un-memoized feed row, or a per-delta persist would each break one of them.

## Security

CSP is set in `tauri.conf.json` (no remote script/connect sources — the webview talks IPC only). The updater verifies minisign signatures against the pinned pubkey.

**Final-audit hardening (rounds R2–R13).** A closing security pass hardened the FS, classifier, host and autonomy surfaces; the details live in each subsystem's section above, indexed here:

- **Shared FS hardening — `src-tauri/src/fsx.rs`** (R3/R4/R5): the ONE home for fail-closed **path containment** (`fold_absolute`/`canonicalize_lenient`/`path_within`/`path_strictly_within` — "strictly inside this root after full symlink + `..` resolution?", any doubt → false) and unix-only **anchored no-follow directory handles** (`DirHandle` — every component opened `O_NOFOLLOW | O_DIRECTORY`, then `*at` syscalls on the anchored fd, so an agent swapping an intermediate path element for a symlink can't redirect the operation; plus `identity()` = the handle's `(dev, ino)` for verifying a path-resolved view still names the anchored directory, and `read_link`/`symlink`/`kind` for the no-follow env copy — audit C7). Consumed by `classify_approval` (foreign-cwd + operand + fileChange containment), `worktree.rs` (`.worktrees` removal confinement AND the fd-anchored create/verify/env-copy), `plans.rs` (the no-follow plan area) and `transcript.rs` (`project_docs` open with a `symlink_metadata` regular-file check + `O_NOFOLLOW`). Unit-tested with planted symlinks, symlinked dirs and FIFOs.
- **Approval classifier** (R2/R3, see *Approval routing*): `rg`'s program-executing flags (`--pre`/`--pre-glob`/`--hostname-bin`/`-z`) → destructive; every path-bearing operand of an otherwise-routine head (`cat`/`grep -f`/`cargo --manifest-path`/`tsc -p`) must resolve inside the session cwd (`operands_confined`).
- **Full access is human-only** (see *AGENTS.md* invariants): `spawn_agents`/`set_agent_config` refuse `access:"full"` on BOTH sides (Rust registry drops it from the schema; the webview `resolveAgentAccess` clamps to `workspace`). `danger-full-access` is reachable only via the human New-Session toggle / `vibe_session_set_access`.
- **Codex host** (`codex/host.rs`, `codex/sessions.rs`): MCP opt-out is FAIL-CLOSED (R12 — an enabled, non-disableable MCP server refuses the spawn; see *Codex app-server host*); the stdout/stderr line reader is BOUNDED (`next_line_bounded`, `MAX_LINE_BYTES` 64 MiB) and the event channel backpressures rather than growing unbounded; session events are **generation-fenced** (R6 — a request/notification/exit from a superseded process generation is answered but never mutates the resumed session's state, `generation_current`); `vibe_session_close` `host.shutdown()`s the child (R7 — no leak).
- **Identity & memory** (R10/R11, see *Fixed Orchestrator identity & memory*): identity is a product constant with no user-controlled fields; project name/dir are `sanitize_inline`d to single-line quoted literals (C1 + U+0085/U+2028/U+2029 → space); memory has a 500-char per-entry cap on top of the 20/2000-per-file FIFO.
- **Dev hook gated in release**: `orchestrator_run_tool` (the unscoped `window.__orch` tool surface) is refused in release builds via a `cfg!(debug_assertions)` gate in `lib.rs`.
- **Frontend fail-closed persistence**: sessions are never auto-evicted at the 30-lane archive hint; explicit close tears down the backend before dropping its visible/store identity, and Mission recovery closes only ownership-proven orphan Mission lanes. `loadAutonomyBudgets` THROWS on a genuine read error and hydrate then `latchAutonomyUnavailable()`s a GLOBAL breaker; Mission, outbox, Vibe and Runtime hydration/write health jointly gate Mission dispatch. `read_notes` is project-scoped (only the calling project's own folders); every untrusted string interpolated into an autonomous wire is single-line-flattened AND `JSON.stringify`-quoted as a delimited literal. Outward GitHub writes are additionally refused inside an autonomous turn unless `settings.autonomousGithubWrites` is on (default off) — see the autonomy invariant in `AGENTS.md`.
