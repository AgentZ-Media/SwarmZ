## Transport & backends

`src/lib/transport.ts` exports the native Tauri backend. Backend capabilities are declared in `backend-types.ts` and implemented in `backend-tauri.ts`. Native-only features may skip the backend interface and use `@tauri-apps/api` directly (e.g. `lib/worktree.ts`, `lib/orchestrator/native.ts`, `lib/vibe/controller.ts`, `lib/projects/store.ts` ŌĆö the `canonicalize_path` dedupe resolver).

## Store & persistence

`src/store.ts` holds the shared app state. Persisted: usage history, quick notes, and app settings (`AppSettings`: last used folder, codex/git path overrides, auto-update, worktree repo registry `worktreeRepos`, orchestrator prefs ŌĆö codex model/effort defaults, scan roots, persona, the Phase-5 `autoReviewFinishedLanes` toggle) ŌĆö edited in `SettingsDialog.tsx`, opened via the title-bar gear or `Ōīś,` (Tauri store `swarmz.json`). Always mutate settings through `updateSettings` (merges + persists the whole object). In-memory only: palette/notes/dashboard flags, the worktree panel scan, `quitConfirm`. Window size/position persists natively via `tauri-plugin-window-state`. Sessions live in their own store (`lib/vibe/session-store.ts`, key `vibeSessions`), orchestrator chats in `lib/orchestrator/chat-store.ts` (key `orchestratorChats`), project tabs in `lib/projects/store.ts` (key `projects`), Conductor timers in `lib/orchestrator/timers.ts` (key `conductorTimers`), autonomy budgets in `lib/orchestrator/autonomy.ts` (key `autonomyBudgets` ŌĆö serialize/hydrate stay pure in the module; store.ts wires the load/save + a debounced dirty sink via `registerAutonomyPersist`); all five are flushed by `flushAllPersists` at quit and hydrated from `hydrate()` in a FIXED order ŌĆö autonomy budgets FIRST (every autonomous delivery path gates on the projects store being hydrated, so restored budgets/breakers are guaranteed in place before any turn can pass the budget check) ŌåÆ projects ŌåÆ vibe sessions ŌåÆ orchestrator chats ŌåÆ conductor timers (the session migration assigns projectIds into the project store; the Phase-3 chat migration then resolves against both; timers fire missed ones and drop vanished-project entries ONLY when projects verifiably hydrated ŌĆö see *Conductor timers*).

The old pane-era keys (`grid`, `workspaces`, `workspacePresets`, `commandPresets`, `customCommands`, `profiles`) are DELETED once by the schema-v2 migration (`DEAD_STORE_KEYS` in `lib/schema-version.ts` ŌåÆ `deleteStoreKeys` in the backend).

The store file itself has a crash net: `storefile.rs` runs in Tauri `setup` before the webview loads ŌĆö a healthy `swarmz.json` refreshes `swarmz.json.bak`, a corrupt one (torn write, disk-full) is moved aside as `swarmz.json.corrupt-<ts>` and the backup restored, so the plugin's silent treat-unreadable-as-empty behavior can no longer factory-reset the app.

`swarmz.json` also carries a top-level **`schemaVersion`** key (a plain number, currently `2`) ŌĆö the migration anchor, deliberately WITHOUT a migration framework. The pure hooks live in `src/lib/schema-version.ts` (unit-tested): `normalizeSchemaVersion` (valid stored versions ŌĆö even NEWER ones ŌĆö are left untouched, so a downgraded build never clobbers a newer store's stamp) and `planSchemaMigration`, which `hydrate()` runs first thing: a raw version `>= 2` plans nothing; `1` or missing/invalid (a possibly pre-rebuild store) plans the one-time dead-key cleanup + the `2` stamp. The plan is idempotent. The actual VALUE migration (sessions ŌåÆ projects) does not run here ŌĆö it lives tolerantly in the vibe hydrator (see *Projects*). The `storefile.rs` rescue path is version-agnostic and stays untouched.

## Projects & swarm data model (Phase 2)

**Entity** (`Project` in `types.ts`, persisted key `projects` via `lib/projects/store.ts`): `{ id, dir, name, order, lastActiveAt, createdAt, closedAt? }`. `dir` is the CANONICAL folder path ŌĆö `openProject` resolves it through the Rust `canonicalize_path` command (symlinks, `/var` aliasing; falls back to the normalized input when the folder is gone) and dedupes on it, so the same folder always maps to the same project. `name` is the folder basename with a lowest-free numeric suffix on collision ("api", "api 2"). `order` is the tab position among OPEN projects; `activeProjectId` persists. Pure logic (dedupe key, naming, reorder patches, migration) lives in `lib/projects/core.ts` (unit-tested); the zustand store around it debounces persists (~500 ms, flushed by `flushAllPersists`).

**Open vs closed**: closing a project TAB only stamps `closedAt` ŌĆö the entity stays persisted and its sessions stay in the session store; reopening the same folder (picker, recents, a triage jump) flips the SAME project open again and the sessions reappear. `setActiveProject` auto-reopens.

**Sessions belong to projects** (schema v2): `VibeSession` carries `projectId`, `agentName` (the generated identity ŌĆö immutable, branch names derive from it in Phase 4), `spawnedBy: "user" | "conductor"` and `worktree: { root, branch, shared } | null` (carried now, filled by the WorktreePanel flow and Phase 4's tools). **Hydrate migration**: every hydrated session runs through `useProjects.adoptSessions` (pure core: `assignSessionsToProjects`) ŌĆö a valid `projectId` passes through; pre-v2 sessions and sessions whose project record was lost get projects derived from their `projectDir`s (deduped, matched against open AND closed projects, created OPEN otherwise); `agentName` defaults to the old display name, `spawnedBy` to "user". The stamped result persists immediately (`vibeSessions` writes `version: 2`). The session cap is **30 per project** (`MAX_SESSIONS_PER_PROJECT`, enforced on create and on hydrate).

**Agent names** (`lib/vibe/names.ts`, pure + unit-tested): a 130+ pool of short international first names. `pickAgentName(taken, rng?)` picks collision-free (case-insensitive; per project in the controller, globally strict in the New-Session dialog), suffixing "Maya 2" once the pool is exhausted; `branchSlugForAgent(name, task?)` ŌåÆ `swarm/maya-checkout` (NFKD diacritics-stripped ASCII slugs, task fragment capped at a word boundary). The dialog prefills a generated name with a ­¤Ä▓ reroll; `spawn_agents` sessions without a `name` get a pool name too.

**Scoping**: the SessionRail shows only the ACTIVE project's sessions (primitive joined-ids signature; no active project = show all, a degenerate pre-migration state). The Deck triage/counters stay GLOBAL across projects ŌĆö `focusSession` switches to the session's project tab first (reopening a closed one), then selects it; `activateProject` (tab click, Ōīś1ŌĆōŌīś9, palette) aligns the stage: selection restores the project's REMEMBERED session (`activeIdByProject`, persisted; stamped by `setActive`/`createSession`, validated on hydrate), else the project's newest; a session-less project lands on the Conductor. The orchestrator's `spawn_agents` attaches sessions to the Conductor's own project when `cwd` is omitted/equal to the project folder; an explicit foreign `cwd` resolves its own project (NOT the active tab) and never steals the active tab (`activate: false` for conductor spawns).

**TitleBar tabs** (interim look ŌĆö Phase 6 restyles): one tab per open project with a live dot (busy = muted, idle = faint), `busy/total` count, `ŌÜæn` needs-you badge, close X (middle-click too), double-click rename, drag reorder; the + button opens a folder picker / discovery-recents dropdown ("Open project" also lives in the ŌīśK palette, along with tab switching). Tabs are plain `<div>`s and opt out of the header drag region via `data-tauri-drag-region="false"`. **Closing with busy sessions** never blocks and never stops anything, but routes through `requestCloseProject` (vibe controller): N busy sessions raise the `CloseProjectConfirm` dialog (busy count + "they keep running in the background") before the tab hides.

## Quick notes

**Quick notes** (ŌīśN, title-bar note button, `QuickNotesPanel.tsx`): a right-side slide-in drawer holding lightweight checklists ŌĆö one **global** list plus one **per project folder** (store key `quickNotes` ŌåÆ `QuickNotesData`, folder lists keyed by repo root). The drawer opens in the scope of the active session's `projectDir` (fallback Global) and shows a chip per scope. Items (`NoteItem`) check off, edit inline, reorder by dragging the grip, and can toggle to plain text lines (`plain`); "Clear done" drops every checked item, emptied folder lists drop their key. Every mutation persists immediately. The drawer is a `role="dialog"` so the global-shortcut guard kicks in; ŌīśN closing it is special-cased in the dialog branch of the `App.tsx` keydown handler.

## Usage & limits

Codex usage is parsed from `~/.codex/sessions/**/*.jsonl` by `src-tauri/src/codex_usage.rs` (which also owns the shared usage types since the Claude parser was removed), using `session_meta`, `turn_context`, `event_msg/token_count`, `task_started` and `task_complete` rollout events for session id, cwd, model, tokens, context, limits and activity. The usage commands (`usage_for_dir`, `usage_for_session`, `usage_totals`) are async + `spawn_blocking` and codex-only; they currently have no frontend caller ŌĆö live session accounting runs frontend-side instead (see below). Codex ChatGPT-plan rollouts do not include reliable USD cost, so cost stays zero (there is no pricing table anymore).

Codex limit windows are **account-level**: `codex_account_limits` finds the newest `token_count.rate_limits` event across ALL of `~/.codex/sessions/**/*.jsonl` ŌĆö files are iterated newest-mtime-first, each only via a bounded 256 KiB tail read, stopping at the first hit ŌĆö and is polled every 60 s by `src/lib/limits.ts` (paused while hidden) into the Deck meters. `limits: null` = no data ever seen (the Deck shows a quiet `CX ŌĆö` placeholder); data older than 1 h gets an "as of HH:MM" tooltip suffix.

The **usage dashboard** (`UsageDashboard.tsx`, title-bar chart button) shows the persisted all-time history (store key `usageHistory`, Codex entries; the pre-rebuild Claude/shell entries are filtered from view). **Session accounting** feeds it live: the vibe controller mirrors each session's `token_usage` events into `recordUsageHistory` (keyed `codex:<session id>`, `runtime:"codex"`, `by_model` from the session's model override or "codex default") ŌĆö debounced ~2 s per session, flushed once more on close. The thread's `total` bucket is cumulative, so each write REPLACES the entry; cached input counts as cache reads and is subtracted from fresh input; `message_count` = user turns in the transcript; cost stays 0 (no pricing).

## Git status

`git_info` (`src-tauri/src/git.rs`) is a strictly read-only snapshot (repo, branch, ┬▒lines, untracked, `origin` URL) that shells out to the git binary (prefers the Settings `gitPath` override, else `/usr/bin/git`; every subprocess runs through `output_with_timeout`). There is no polling anymore ŌĆö the orchestrator's `git_status` tool and the worktree flows call it on demand.

## Git worktrees

`src-tauri/src/worktree.rs` + `src/lib/worktree.ts`, native-only direct `invoke` ŌĆö deliberately *writing* git, unlike the read-only git.rs. `worktree_add` creates `<repo>/.worktrees/<slug>` via `git worktree add -b` off the current HEAD (main root resolved via `--git-common-dir`), excludes `.worktrees/` via `.git/info/exclude` (never the tracked `.gitignore`), and copies every untracked file (`ls-files --others`, incl. gitignored `.env`s) minus the heavyweight `SKIP_DIRS`. The human way to run a session inside a worktree is picking the worktree folder in the New-Session dialog (or the WorktreePanel's open-in-session); since Phase 4 the Conductor creates and manages worktrees itself through its tools (`spawn_agents` placement "new"/"shared:<agent>", `create_worktree`, `assign_worktree` ŌĆö which retunes a live session via `thread/settings/update {cwd}` ŌĆö, `worktree_status`, `cleanup_worktree`), stamping every session's `worktree` meta (`{root, branch, shared}`).

**Management panel** (`WorktreePanel.tsx`): a title-bar `FolderGit2` button, visible only while Ōēź1 worktree exists, lists all SwarmZ worktrees grouped per repo (dirty/ahead state, refreshed on open) with open-in-session (the session runs in the worktree but belongs to the MAIN repo's project tab and carries the `worktree` meta) or jump-to-session, reveal-in-Finder, per-row finish/delete and a "Clean up safe" bulk action for clean unattached worktrees (a live session's `projectDir` counts as attached). Risky rows (uncommitted changes or local-only commits) require a second delete click; deletion re-checks status at execution time ŌĆö on doubt the worktree is kept. **`worktree::remove` takes a `force` flag**: user-confirmed panel deletions force; every SILENT path ("Clean up safe", the close-flow's clean branch, the Conductor's `cleanup_worktree`, the spawn rollback) passes `force: false` ŌĆö Rust then re-checks dirty/ahead INSIDE the removal call (an uncomputable ahead count ŌĆö `ahead_unknown` in `WorktreeStatus`/`WorktreeEntry` ŌĆö refuses like dirt; a git error is never silently 0) and runs `git worktree remove` WITHOUT `--force`, so git itself refuses work that appeared after the caller's check (the copied gitignored env files don't block a non-force removal ŌĆö probe-verified); on the missing-folder path the gated branch `-D` additionally refuses when the branch still holds unreachable commits. Scanned repo roots persist in `settings.worktreeRepos` (pruned automatically once a repo has no worktrees left; `worktree_list` returns a `WorktreeScan {entries, scanned}` and only roots that actually **scanned** empty are pruned ŌĆö an unmounted volume or broken git override never loses orphans). A hand-deleted worktree folder still reports its branch's unreachable commits as `ahead`. The keep-vs-delete dialog (`CloseWorktreeDialog.tsx` + the store's `requestCloseWorktree`/`resolveCloseWorktree`) is the HUMAN escalation path; the Conductor's `cleanup_worktree` never dialogs ŌĆö it re-checks status at execution time and REFUSES on dirty / local-only commits / unknown state / live occupants (forcing stays a human decision).

## Status reporting & quit guard

Quit guard (`src/lib/quit.ts`, started in `App.tsx`): closing the app raises `QuitConfirmDialog.tsx` when a session has a turn in flight (`vibeBusyIds` ŌĆö the run would be interrupted). Native covers both paths ŌĆö window close via `onCloseRequested` (every first close is preventDefault'ed, all debounced persists are flushed (`flushAllPersists`), then `close()` is re-issued past a module-level `confirmed` flag so the window-state plugin still saves), and ŌīśQ/menu quit via Rust (`lib.rs` run-callback: `ExitRequested` with `code.is_some()` ŌåÆ `prevent_exit()` + emit `app://quit-requested`; `prevent_exit` is a tauri-built-in no-op for the updater's `RESTART_EXIT_CODE`, and code `None` = last window closed passes, which is how a confirmed quit actually exits).

## Orchestrator sensing

Read-only sensing. `src-tauri/src/transcript.rs`: `transcript_read` returns actual message content from codex rollout files (found by the id suffix in the filename, head-scan fallback) ŌĆö the `event_msg` `user_message`/`agent_message` chat stream preferred, `response_item` messages (instruction wrappers filtered) as fallback. Options: `tail_messages` (default 20), `max_bytes` (default 1 MiB ŌĆö huge files are seek-tailed to the first complete line, never fully read) and `include_first_user_message` (from a separate capped head read). Texts are char-truncated (umlaut-safe) with an `ŌĆ”` marker. Note: the orchestrator's `read_agent` TOOL renders LIVE sessions from the vibe store (`renderSessionTranscript`) ŌĆö `transcript_read` reads rollout files on disk and currently only serves the dev hook. `project_docs` reads README.md/AGENTS.md/CLAUDE.md of a root (worktree paths collapse to the main repo; ~8 KB/file, ~20 KB total; missing files omitted). `src-tauri/src/projects.rs` (codex-only): `discover_projects` merges project folders from Codex `session_meta` cwds, frontend-known folders (project tabs ŌĆö open and closed ŌĆö, live session dirs, quick-note folder keys, `worktreeRepos`, `lastCwd` ŌĆö collected in `lib/orchestrator/native.ts`) and a shallow scan (depth Ōēż 2) for `.git` (dir or file); worktree paths collapse onto the repo root, entries dedupe and sort by last activity. It also feeds the "open project" recents (TitleBar + dialog). Both commands are async + `spawn_blocking`, uncached.

Frontend: `src/lib/orchestrator/` ŌĆö `snapshot.ts` (pure `sessionSnapshot`/`sessionCounts`/`fleetSummaryLine` over the vibe store's entries + busy map), `native.ts` (direct-`invoke` wrappers), `types.ts` (payload + tool-bus types), and `dev.ts`, a dev-only smoke-test hook exposing `window.__orch`.

## Orchestrator tool bus

**The registry is the single source of truth**: `src-tauri/src/orchestrator/registry.rs` defines every tool as `{ name, description, parameters (JSON Schema), timeout_ms }`; `tool_definitions()` serializes the catalog (**24 tools** since Phase 4, project-scoped) for the Codex app-server (`dynamicTools`). Per-tool timeouts follow the work behind the tool: 15 s default, `prompt_agent` 30 s, worktree ops 150 s, `spawn_agents` 600 s (the bus timeout can't CANCEL the webview batch ŌĆö a too-tight deadline would leave the batch running while the model retries into duplicate agents, so it must outlast the 8-agent worst case incl. shared-lane waits; the batch's name reservation additionally fail-fasts a same-names retry), `review_agent` 600 s. The `orchestrator_tools(persona, project)` command returns `{ instructions, tools }` ŌĆö the tools plus the compiled system instructions (`persona::build_instructions`). The frontend mirrors the *names* only (`ORCHESTRATOR_TOOL_NAMES` in `types.ts`, executor keys). Argument validation is a deliberately hand-rolled subset (required props, basic `type` tags, `enum`, array `minItems`/`maxItems`/`items`, applied recursively).

**Roundtrip bus** (`orchestrator/bus.rs` + `src/lib/orchestrator/bus.ts` + `executors.ts`): tools are defined in Rust but *execute in the webview*, where the Zustand stores live. `run_tool` validates against the registry, parks a tokio oneshot in a pending map, emits `orchestrator://tool-request` `{ id, tool, args, chat_id, project_id }` (`chat_id` = the triggering backend chat, `project_id` = the Conductor instance's project ŌĆö both null for dev/internal calls), and awaits the response with the tool's timeout (per the registry's per-tool budgets ŌĆö `spawn_agents` creates worktrees and spawns codex processes, `review_agent` runs a whole review turn). The webview bus (`startOrchestratorBus()`, registered once from `App.tsx`) looks up the executor, runs it with the `{chatId, projectId}` context and **always** answers via `orchestrator_tool_response(id, ok, payload)`. The bus core is tauri-free and unit-tested. `orchestrator_run_tool` is the dev-hook surface (`await __orch.tool("fleet_snapshot", {})` ŌĆö unscoped).

**Project scoping** (Phase 3): `requireSession` resolves `agent` arguments ONLY among the Conductor's project's sessions ŌĆö by raw id, or by session/agent NAME (optionally `@`-prefixed, case-insensitive) when unique in the project (ambiguity errors with the candidates; unknown ids error with the project's valid list). `fleet_snapshot` returns `{project, summary, sessions, worktrees, timers}` with only that project's data; the summary line Rust prepends to every turn uses the same scope. An unscoped call (dev hook) sees everything; project-scoped tools (spawn/worktrees/timers/plans) error without a project.

**The 24 tools** (executors in `src/lib/orchestrator/executors.ts`, catalog frozen by a registry test):

- **Sensing** ŌĆö `fleet_snapshot` **v2** (project header + scoped `sessionSnapshot` with exact status working/idle/pending-approval, per-session worktree `{branch, shared}`, `worktrees` occupancy (who works where, pure `worktreeOccupancy`), pending `ApprovalBrief`s with their routing class, and the project's timers) ┬Ę `read_agent` (live session items rendered compactly via `renderSessionTranscript`) ┬Ę `read_project_docs` (exactly one of `agent`/`path`) ┬Ę `read_notes` ┬Ę `git_status` (exactly one of `agent`/`path` ŌĆö worktree cwds work) ┬Ę `list_projects` (`scan_roots` omitted = the user's `orchestratorScanRoots`).
- **Agents** ŌĆö `spawn_agents` (1ŌĆō8 agents, each `{task, worktree, model?, effort?, access?, name?, expect_report?}`; `expect_report: true` (Phase 5) constrains the task turn's final message to the agent-report `outputSchema` and appends the report suffix to the task ŌĆö see *Autonomy loop*; placement `"new"` = a fresh worktree on a `swarm/<name>-<task>` branch (collision-suffixed, env copied, repo registered for the panel), `"shared:<agent>"` = join that agent's worktree (both marked `shared`; may reference an agent spawned earlier in the SAME batch), `"none"` = the project folder (analysis/read tasks); names auto from the pool with a CASE-INSENSITIVE batch reservation (an explicit name colliding with a live session or an earlier batch entry fails ITS spec ŌĆö duplicate names would make `shared:<name>` and every name-based tool ambiguous), effort defaults to `medium`, access to `workspace`, `spawnedBy:"conductor"`; sequential creation, per-entry errors never abort the batch, a FRESH worktree whose session fails to start is rolled back (gated non-force removal; an unremovable one is named in the error for `cleanup_worktree`); initial tasks kick off in parallel ACROSS lanes but strictly serialized WITHIN a shared worktree ŌĆö the next same-lane turn starts only after the previous agent's turn finished (`waitForSessionIdle`, bounded ~8 min; a still-busy lane hands the remaining tasks back as per-agent warnings instead of starting a second writer)) ┬Ę `prompt_agent` (idle ŌåÆ next turn; **BUSY ŌåÆ `turn/steer`** ŌĆö the running turn absorbs the text; the result reports `mode: "turn"|"steered"`; the 2 s double-prompt guard stays, the old busy-refusal is gone; `expect_report?` applies only when the prompt starts a FRESH turn ŌĆö a steered running turn keeps its own format) ┬Ę `interrupt_agent` ┬Ę `close_agent` (worktree stays ŌĆö the result says to clean it separately) ┬Ę `set_agent_config` (model/effort/access mid-session; `""` clears an override; applies next turn) ┬Ę `review_agent` (native detached `review/start` over the agent's work ŌĆö target `uncommitted` (default) / `branch:<base>` / `commit:<sha>`; returns the prioritized findings report) ┬Ę `decide_approval` (ROUTINE approvals only ŌĆö enforced SERVER-SIDE via `respond_approval(require_routine: true)`, the frontend class check is just the fast lane; decision accept|decline, optional `approval_id`, else the oldest pending; nothing marked optimistically ŌĆö see *Approval routing*).
- **Worktrees** ŌĆö `create_worktree` (fresh worktree + branch, env copy) ┬Ę `assign_worktree` (re-home an agent: refuses BUSY sessions, backend `thread/settings/update {cwd}` FIRST and the store meta only on its ack; co-tenants flip `shared`) ┬Ę `worktree_status` (project worktrees with dirty/ahead ŌĆö an uncomputable count surfaces as `ahead_unknown` ŌĆö + occupants) ┬Ę `cleanup_worktree` (**safe-gated**: refuses live occupants, re-checks dirty/ahead at execution time and treats `ahead_unknown` as "may hold work"). Entry lookup (`findWorktreeEntry`) is scoped to the CALLING Conductor's project repo only (canonicalized paths; never the cross-project `worktreeRepos` registry ŌĆö project A can't touch project B's worktrees), and `cleanup_worktree` + `assign_worktree` serialize per worktree through ONE async lock, so a cleanup's occupancy/status checks can never interleave with a re-homing. The removal itself goes through `worktree::remove(force: false)` ŌĆö Rust re-checks dirty/ahead/`ahead_unknown` inside the call and runs `git worktree remove` WITHOUT `--force`, so even work appearing after the checks refuses (never forces; forcing stays a human decision).
- **Timers** ŌĆö `set_timer` (exactly one of `delay_seconds`/`at_iso` + a note; per-project cap 20) ┬Ę `list_timers` ┬Ę `cancel_timer` (project-scoped).
- **Plans** ŌĆö `write_plan`/`list_plans`/`read_plan` ŌåÆ the Rust `plans.rs` commands (see *Conductor plan documents*).
- **Memory** ŌĆö `remember` (write-to-disk, ŌåÆ `orchestrator_memory_append` with `scope: "project"` (default) | `"global"`).

## Conductor plan documents (`src-tauri/src/plans.rs`)

The Conductor's ONE sanctioned write surface ŌĆö the precisely-worded exception to the "never edit files" guardrail. `conductor_plan_write(project_dir, title, markdown)` stores Markdown under `<project>/.swarmz/plans/<slug>.md`; the slug is DERIVED in Rust from the title (lowercase `a-z0-9-`, Ōēż64 chars, no dots/slashes) and `read`/`list` re-validate it, so a hostile title or slug can never escape the folder (unit-tested with `../` payloads). The project dir comes from the trusted chat context, never from the model. Content is capped at 256 KiB, titles at 200 chars, an H1 title is ensured, same title replaces; a DIFFERENT title colliding on the same slug gets a stable hash suffix instead of silently overwriting the other document. `.swarmz/` is excluded from git via the repo-local `.git/info/exclude` (best-effort, same mechanism as the worktrees ŌĆö never the tracked `.gitignore`), so plans don't dirty the user's `git status`. Foreign files in the folder are never listed/served.

**No-follow confinement** (Phase-4 security pass ŌĆö an AGENT can plant symlinks in its workspace, e.g. `.swarmz/plans/x.md ŌåÆ ../../src/main.rs` or `.swarmz/plans ŌåÆ /tmp`): the `.swarmz` and `plans` path components refuse when they are symlinks; writes refuse a non-regular file at the target and land ATOMICALLY via a fresh temp file + rename (never writing THROUGH anything); reads open with `O_NOFOLLOW` after a `symlink_metadata` regular-file check (a FIFO/device with a safe name can't hang the open) and are BOUNDED (oversized foreign files error instead of flooding memory; the title scan reads Ōēż4 KiB); `list` uses the no-follow `DirEntry::file_type()`, so a safe-named symlink to a host file is skipped, never leaked. All of it unit-tested with planted symlinks, a symlinked plans dir and a FIFO.

## Conductor timers (`src/lib/orchestrator/timers.ts` + `timers-core.ts`)

Follow-up timers the Conductor sets itself (`set_timer`). A standalone zustand store persisted under `conductorTimers` (debounced, flushed at quit); pure logic (arg resolution `resolveFireAt` with exactly-one-of `delay_seconds`/`at_iso`, sanitizing incl. the `firedAt` claim, due-splitting, rendering) lives in `timers-core.ts`, unit-tested. **Semantics**: timers persist across restarts but only FIRE while the app runs; wakeups are `setTimeout`s (clamped + re-checked for >24-day spans); on hydrate, DUE timers fire immediately, marked as missed. Firing delivers an **autonomous Conductor turn** in the timer's project (visible `ŌÅ░ Timer fired:` system marker + a `[timer firedŌĆ”]` wire text) via the controller's `deliverTimerTurn` (which passes the autonomy budget, and since Phase 5 runs INSIDE the project's autonomous serialization chain ŌĆö `runExclusiveAutonomous` ŌĆö so a timer turn never interleaves with an event-triggered turn in the same chat). The delivery + notice functions are REGISTERED from App.tsx (`registerTimerDelivery`/`registerTimerNotice`) instead of imported ŌĆö that keeps the import graph acyclic (store ŌåÆ timers ŌåÆ controller ŌåÆ store would otherwise be a runtime cycle).

**Safety rails** (Phase-4 security pass, hardened in Phase 5):

- **Idempotency / at-most-once**: an in-memory `firing` claim guards against double wakeups within a run, and a durable **`firedAt` stamp is persisted (flushed, not debounced) IMMEDIATELY BEFORE the dispatch, INSIDE the serialization chain** ŌĆö `fireTimer` hands `deliverTimerTurn` a `claim` callback that the delivery invokes only once the project's autonomous chain reached the timer (never before the possibly-minutes-long chain wait: a quit while the timer queues behind a long turn/review must re-fire it as missed, not hydrate-drop a never-delivered timer). A crash between the claimed dispatch and the timer's removal must not re-deliver the same `tm-*` on the next launch. Hydrate DROPS claimed timers with a visible chat notice ("may have already firedŌĆ” dropped to avoid a double delivery") instead of re-firing them; a "retry" outcome clears the stamp again (nothing ran).
- **Cancel re-check**: the `claim` callback re-checks that the timer still EXISTS ŌĆö a timer cancelled while queued (e.g. `cancel_timer` called by the very autonomous turn the timer waited behind) aborts as "drop" instead of firing anyway after the turn ends.
- **Honest delivery**: `dispatchTurn` propagates failure ŌĆö a turn that never started (codex unavailable, spawn failure) answers "retry", never "delivered"; the timer stays persisted.
- **Bounded retries**: "retry" (busy chat, open circuit breaker, failed dispatch) re-arms every 30 s AT MOST 20 times (~10 min); after that the timer is removed and **expires visibly** (`ŌÅ░ Timer expired undeliveredŌĆ”` system message via the notice sink). A quit mid-retry re-fires it as missed next launch.
- **Hydration honesty**: `loadConductorTimers` RETHROWS store read errors (null = only a genuinely missing key) ŌĆö an unreadable store keeps the in-memory state untouched and persists nothing; the vanished-project filter applies ONLY when the projects store verifiably `hydrated` (a failed project hydration must never bulk-erase timers ŌĆö the Phase-3 chat-readiness lesson), and `deliverTimerTurn` answers "retry" while projects are not hydrated.

## Approval routing (Phase 4)

Every approval request a session raises is CLASSIFIED in Rust (`codex/sessions.rs::classify_approval`, pure + unit-tested against an adversarial payload suite) and the `approval_request` event carries `escalation: "routine" | "destructive"` (stored on the approval item, hydrated tolerantly ŌĆö missing = destructive).

The classifier is **FAIL-CLOSED by construction** ("im Zweifel IMMER Mensch" ŌĆö a false destructive costs one human click, a false routine is unacceptable). **Commands**: a strict quote-aware tokenizer rejects the whole command on ANY shell metasyntax outside single quotes (`| & ; < > ( ) { } $ \` \\`, newlines, unterminated quotes ŌĆö no best-effort segment parsing; `ls || find . -delete`, `echo $(ŌĆ”)`, `cat /dev/null > x` are all human) and on any backslash escape (`r\m -rf` can't hide a head). Routine then requires the (bare or `/bin`//`/usr/bin`-prefixed ŌĆö never `./x` or other local paths) head to be on the tiny read-only allowlist ŌĆö `ls cat head tail grep rg wc pwd which echo` ŌĆö or one of the gated forms: `git` with a read-only subcommand (`status diff log show branch remote rev-parse describe blame`) AND read-only flags only (no `git -c`/`-C` global options, no `--output`/`-o`/`--ext-diff`, `branch` only in pure list forms, `remote` only bare/`-v`/`show`), `cargo test|check|clippy|build`, `pnpm test|run test`, `tsc`, `node --version`. NO interpreters (`python*`, `node -e/script`, `ruby`, shells as targets), NO `env` (rejected wholesale, never stripped ŌĆö `env` can rewrite PATH/DYLD and subvert an allowlisted head), NO `find`/`xargs`/`sed`/`tee`/`cp`/`mv`/`touch`/`mkdir`/`chmod`. Shell wrappers unwrap ONLY as exactly `sh|bash|zsh -c|-lc '<one script arg>'` (recursively classified, depth-capped); anything else is classified by its OUTER head. Sensitive substrings (`.env`, `id_rsa`, `.ssh`, `credentials`, `secret`, `token`, `/etc/`, ŌĆ”) force human regardless of head, and a request `cwd` outside the session's assigned directory (canonicalized `path_within`) is destructive even for read-only heads. **fileChanges**: routine only when EVERY change is a pure create (`add` of a not-yet-existing path) or an in-place edit (`update` without a move/rename), on a non-sensitive path STRICTLY inside the session's canonicalized cwd ŌĆö the containment check resolves symlinks (leaf AND ancestors; a link out of the tree classifies by its TARGET), rejects `..` lexically before any fs lookup, and fails closed on broken links/IO doubt; deletes, renames, overwriting adds and unknown kinds are always human.

**Server-anchored enforcement**: the class is stored in Rust NEXT to the blocked approval `Responder` (`PendingApproval {responder, escalation}`). The Conductor's `decide_approval` answers through `vibe_session_respond_approval(require_routine: true)` ŌĆö the class check and the responder removal happen atomically under the session lock, so a destructive approval can never be answered through the Conductor path no matter what the frontend claims; the frontend's own `escalation` field is a courtesy copy for UI + a fast pre-check, never the authority. The strict path marks nothing optimistically ŌĆö the item status commits only after Rust's ack, and errors (already resolved by the human, destructive, dead session) propagate into the tool result.

**Routing**: the activity watcher (orchestrator controller) escalates a NEW pending ROUTINE approval as an autonomous Conductor turn (`ŌÜæ Approval escalated` marker + a wire text asking for `decide_approval`; the request summary is flattened to a single quoted line ŌĆö untrusted data, see *Autonomy loop*), **gated on conductor-involvement like every loop event** ŌĆö a purely human-created, never-Conductor-touched session never wakes the loop and never has its approvals decided autonomously (its card + Deck entry behave exactly as before). Routine approvals pending ACROSS a restart get a one-time seed escalation when the watcher starts / the session hydrates (same gate, same dedupe) ŌĆö without it a hydrated pending approval never transitions again and the agent hangs until the human notices the Deck. Since Phase 5 the escalation routes through the **trigger router** (kind `approval`, subject = the approval id ŌĆö per-project serialization + bounded retries; the `escalatedApprovals` set stays the cross-lifetime dedupe: DELIVERED escalations never repeat, retried-out/dropped ones may re-escalate while still pending, and the build re-checks the approval at delivery time so one already decided drops silently; the autonomy budget applies, see below). **The human card stays fully authoritative for BOTH classes** ŌĆö the Deck queue, the approval takeover and the notification are untouched; the Conductor is an additional fast lane for routine, never a replacement. Codex' native `approvalsReviewer:"auto_review"` (a guardian subagent) was evaluated as an alternative and NOT used: it decides inside codex without the Conductor's project context/memory, offers no hard human-only class, and would bypass SwarmZ's own doctrine + UI; the client-side classification keeps the human contract enforceable and testable.

## Autonomy loop (Phase 5)

The Conductor acts like a human engineering lead WITHOUT being prompted: it reacts to fleet events, judges finished work, distributes follow-ups, reports back and learns. Five triggers exist, ALL funneled through one budget-gated core:

| trigger (`AutonomousTriggerKind`) | fires when | wire text |
|---|---|---|
| `agent-finished` | a conductor-involved session goes busy ŌåÆ idle (no pending approval); a FAILED/process-exited turn triggers too, but its wire says "ended its turn WITHOUT completing ŌĆ” do NOT report this as success" (no report parsing, no auto-review); an INTERRUPTED turn is a deliberate stop and never triggers | `[agent finished]` ŌĆö report/last message + diff line + optional auto-review + the lead contract |
| `agent-blocked` | same transition, but the agent asks for direction (report `needs_human`, or the free-text fallback: the final message's last line ends in `?` with more than one word before it ŌĆö a bare "Continue?" stays `agent-finished`) | `[agent needs direction]` ŌĆö the question + "decide it yourself when the goal covers it, else ONE compact question to the user" |
| `approval` | a new pending ROUTINE approval (see *Approval routing*) | `[approval escalation]` ŌĆö decide_approval or hand to the human |
| `timer` | a Conductor timer fires (see *Conductor timers*) | `[timer fired ŌĆ”]` |
| `idle` | a conductor-involved session sat idle Ōēź10 min with uncommitted work (non-empty turn diff), no pending approval ŌĆö ONE nudge per idle stretch (re-armed by the next busy transition; `lastBusyEndAt` is transient, so restarts never replay stale nudges) | `[idle check]` ŌĆö finish the lane or report the stall, "do NOT nudge again" |

**Trigger router** (`lib/orchestrator/triggers-core.ts` pure + unit-tested, wired in `triggers.ts`): every EVENT trigger goes through `enqueueAutonomousTrigger({projectId, kind, subjectId, prepare?, build, onSettled?})` ŌĆö (1) **dedupe** per `(projectId, kind, subjectId)` while pending/running (double events, two windows fire once). Finish triggers key their subject per **busy-cycle generation** (`session#gen`, a monotonic per-session counter): turn B's finish is never swallowed by turn A's still-retrying trigger, and a superseded generation drops at delivery instead of shipping stale context; the finished/blocked pair enqueues under ONE neutral kind (`agent-finished`) and `build()` refines the kind at delivery (`BuiltTrigger.kind`) ŌĆö classification is delivery-fresh, dedupe stays subject-based. (2) **eligibility**: projects store not hydrated = retry (a load race never eats a trigger), record gone = drop, and the event kinds (`agent-finished`/`agent-blocked`/`idle`) stay QUIET for closed tabs (drop ŌĆö the status ping still lands in the chat for later; timers and approval escalations keep delivering into closed projects: a timer is an explicit promise and a routine approval blocks a running agent); (3) **prepare + per-project serialization**: the optional `prepare()` phase (SLOW work ŌĆö the auto-review, up to minutes) runs OUTSIDE the chain so it never starves approvals/timers queued behind it; autonomous work itself runs strictly one at a time per project on a promise chain ŌĆö the `build()` closure (fast, fresh context assembly: last message, report parsing, classification, diff) runs INSIDE the chain right before the turn; a `null` build drops silently (subject vanished / lane re-tasked / generation superseded); (4) **bounded retries**: "retry" (busy chat, open breaker, failed dispatch) re-arms every 30 s at most 10 times, then drops ŌĆö the Phase-4 status ping remains as the fallback signal. The runner is REGISTERED from App.tsx (`registerAutonomousRunner(runAutonomousTurn)`) ŌĆö the same acyclic-imports pattern as the timers. Timers share only the serialization (`runExclusiveAutonomous`), keeping their own persisted retry machinery.

**The turn core**: `runAutonomousTurn(projectId, marker, wireText, trigger)` resolves (or creates) the project's active chat ŌĆö resolving also claims the project's per-launch fresh-chat slot (`freshenedProjects`), so autonomous content delivered before the stage first shows (missed timers, early finishes) stays visible in THAT chat instead of being buried behind the fresh-start chat ŌĆö refuses busy chats and sub-5 s gaps with "retry", passes the **autonomy budget**, then appends the visible system marker ŌĆö stamped **`autonomous: true` + `trigger`** on the message (persisted + hydrated; the interim ChatView renders a small `ŌÜĪ autonomous` chip, Phase 6 designs the real treatment) ŌĆö and dispatches through the same honest `dispatchTurn` the user path uses. Budget accounting is a **reservation**: booked before the dispatch, kept once the turn actually STARTED (a mid-turn failure still burned real work), RELEASED on a definitive pre-start failure (`releaseAutonomousTurn` ŌĆö five dead-codex attempts must not trip the breaker with "5 turns ran" while zero ran); a never-started attempt also keeps its ONE marker message and the retry reuses it instead of stacking `ŌÜĪ autonomous` markers.

**The autonomy budget** (`lib/orchestrator/autonomy.ts`, pure + unit-tested) is the HARD cap every autonomous turn passes first ŌĆö without it, a prompt-injected agent could spin an endless cascade (finished agent ŌåÆ autonomous turn ŌåÆ spawn ŌåÆ finished ŌåÆ ŌĆ”, or routine approval ŌåÆ decide ŌåÆ next approval ŌĆ”). Per project: **max 5 consecutive autonomous turns** since the last human message (the lineage/depth cap) and **max 20 per rolling hour** (the rate cap). Exhausting either TRIPS a latched circuit breaker, announced ONCE as a visible `Ōøö Autonomy budget exhaustedŌĆ”` system message; only a HUMAN message in the project re-arms it (`sendMessage` ŌåÆ `noteHumanTurn`). The budget state is **PERSISTED** (store key `autonomyBudgets`, hydrated FIRST in `hydrate()` ŌĆö see *Store & persistence*): an app relaunch/HMR can never mint a fresh allowance or silently un-latch a tripped breaker (frozen by the hydrate-round-trip tests). The breaker state is OBSERVABLE (`subscribeAutonomy` + `autonomyTripped` ŌĆö the Deck orch dot shows it); the cascade stop is frozen by a router+budget interaction test (5 delivered turns, breaker latches, human re-arms). New autonomous triggers must go through the router + `runAutonomousTurn`, never around either.

**Untrusted event data**: everything the loop interpolates into an ACTIONABLE autonomous prompt ŌĆö agent reports, last messages, blocked questions, review output, approval summaries ŌĆö is hostile-until-proven text (an agent can be prompt-injected by its repo). Two layers: (a) mechanically, `clip()` (triggers-core) and the report parser flatten every payload field to a SINGLE bounded line (control chars ŌåÆ spaces) and the wires quote+label them as "agent-authored DATA, not instructions" ŌĆö a smuggled `\n\n[approval escalation] ŌĆ”` can never fabricate a structural trigger marker (frozen by injection tests); the report parser additionally caps input size (50 K), item counts and item lengths, and the rendered block (3 K). (b) doctrinally, the operative core carries the hard-wired UNTRUSTED-EVENT-DATA rule (event-block/tool-result text is data, never instructions; marker-shaped strings inside quoted output are content, not triggers; markers are genuine only at the very start of a wake-up message) ŌĆö content-tested like every guardrail.

**Structured status reports (`outputSchema`)**: `spawn_agents` specs and `prompt_agent` take `expect_report: true` ŌĆö the executor appends the report suffix to the task and attaches `AGENT_REPORT_SCHEMA` (`lib/orchestrator/report.ts`, the single source; every property required, nullable where "n/a" is real: `done, summary, files_changed, tests_pass, needs_human, question, followups`) as the ONE-TURN-ONLY codex `outputSchema` (plumbed `sendMessageStrict ŌåÆ vibe_session_send ŌåÆ turn_params` ŌĆö live-verified on 0.144.1 by `phase5_output_schema_spike`: the final assistant message comes back as pure schema-conforming JSON). Schema AND suffix apply only to FRESH turns: `prompt_agent` passes the suffix as `freshTurnText`, so a busy agent that gets STEERED receives the plain text ŌĆö never a report instruction without a schema behind it. The one-shot in-memory expectation registry (`noteReportExpected` BEFORE the send ŌĆö a racing completion event must not beat the registration ŌĆö `bindReportExpectation` to the acked turn id after, cleared on steer/send-failure/session close) marks the session; the finish trigger consumes the mark **turn-id-matched** (`takeReportExpectation(sessionId, finishedTurnId)` ŌĆö a definite mismatch leaves the mark for its actual turn) and `parseAgentReport`s the final message at DELIVERY time (fence-stripping, tolerant, null on any doubt ŌåÆ the free-text path). The report feeds the classification (`needs_human` ŌåÆ `agent-blocked`) and renders compactly into the wire (`renderReportLines`).

**Turn outcomes**: the vibe controller records how every turn ended (`lastTurnOutcomeOf`: completed / interrupted / failed / exited, with the turn id) BEFORE it clears the busy flag, so the watcher's busyŌåÆidle reaction knows what it saw: completed = the normal loop event; failed/exited = the trigger fires with the explicit failure wire (partial output, "do NOT report this as success", no report parsing, no auto-review); interrupted = no trigger at all (a deliberate stop), the spent report expectation is discarded.

**Auto-review** (`settings.autoReviewFinishedLanes`, default off, Settings ŌåÆ Orchestrator): when ON and a finished (COMPLETED ŌĆö never failed) lane has a diff signal, the `agent-finished` trigger runs `reviewSession(id, "uncommitted")` (the detached codex review) in its `prepare()` phase ŌĆö OUTSIDE the serialization chain, so a 570 s review can't starve routine approvals/timers queued in the same project ŌĆö memoized across retries, and `build()` embeds the findings (or the failure) into the wire: the Conductor reports REVIEWED work. A lane re-tasked while the review ran (busy again at delivery) drops the turn.

**Learning loop**: the operative core's autonomous-turns section + the memory-behaviour reflection doctrine teach the Conductor to store preference observations after work cycles; mechanically, every 3rd DELIVERED finish/blocked turn per project (`shouldNudgeReflect`, counted via the router's `onSettled`) appends the soft `[learning]` nudge to the wire ŌĆö "store ONE concise observation via remember ŌĆ” skip entirely when nothing new emerged". No auto-write: memory still moves only through the explicit `remember` tool.

**Conductor-involvement**: only sessions the Conductor has business with wake the loop ŌĆö `spawnedBy: "conductor"`, or a chat of the project prompted them (`touchedPanes`). Purely human-driven sessions never trigger autonomous turns ŌĆö the gate covers ALL loop events including the routine-approval escalation (a human session's approvals stay exclusively with the human's card).

**Live verification**: `phase5_output_schema_spike` (sessions.rs) and `phase5_autonomy_loop_spike` (appserver.rs) are the `#[ignore]`d live proofs ŌĆö the loop spike runs a REAL agent task with the report schema, then feeds the exact `[agent finished]` wire into a REAL Conductor thread (production instructions + dynamicTools) and asserts it senses (fleet_snapshot/read_agent/git_status), reports by NAME and never leaks a session id.

## Codex app-server host (`src-tauri/src/codex/`)

The generic layer under every `codex app-server` consumer ŌĆö the orchestrator brain and the native sessions. Spawn, framing and respawn code exists exactly ONCE, here.

**Wire protocol** (`codex/protocol.rs`, pure + unit-tested): newline-delimited JSON **without** the `"jsonrpc":"2.0"` header (verified against codex 0.144.1). Our requests use an own ascending numeric id; server-initiated request ids are kept as raw JSON and echoed back exactly. `parse_line` classifies each stdout line into response / server request / notification, defensively. The unit tests freeze the live-captured 0.144.1 shapes (reference schema dump + inventory in `docs/codex-protocol/`): `item/started`/`item/completed` with `commandExecution` and `fileChange` items, `item/agentMessage/delta`, `turn/diff/updated`, `turn/plan/updated`, `thread/tokenUsage/updated`, `account/rateLimits/updated` (account-scoped ŌĆö carries NO threadId) and the approval requests, plus the four decisions `accept`/`acceptForSession`/`decline`/`cancel`. There is NO `item/updated` in the protocol.

**Client & binary resolution** (`codex/host.rs`): `Client::spawn` starts `<program> app-server` via `tokio::process` with one writer task (ordered stdin), a stderr logger and a reader task that resolves responses via a numeric-id pending map and forwards server requests/notifications. Reader EOF = process death: the `alive` flag drops and all pending RPCs fail with a clear error. The binary is resolved to an ABSOLUTE path (Settings override ŌåÆ current $PATH ŌåÆ well-known install dirs ŌåÆ `zsh -lc 'command -v codex'` login-shell probe) because macOS gives the built GUI app a minimal PATH; the child's PATH is enriched with the binary's dir. Resolution is cached process-wide (`set_codex_override` installs the Settings value and drops the cache).

**Thread registry & event routing**: a `Connection` (Client + `initialize` handshake + router task) holds a route table `threadId ŌåÆ EventSink`. The router forwards each server event to the sink registered for its `params.threadId`; routed server requests carry a `Responder` (owning client + echoed id). No sink responsible ŌåÆ notifications are ignored and server requests are refused with -32601. Process death delivers `ThreadEvent::Exited` once per DISTINCT sink and drains the routes.

**Process strategies & respawn**: a `ProcessHost` is one lazily (re)spawned connection slot with a generation counter. The strategies are just slot allocation: **(a) slot per project** ŌĆö each project's Conductor multiplexes its chat threads over its own slot (Phase 3; formerly one global slot); **(b) dedicated process per thread** ŌĆö each session allocates a private slot, so a crash kills only that session. A consumer that sees a different generation than it stored transparently `thread/resume`s before the next turn. `resume_thread` classifies failures typed: `ResumeError::ThreadNotFound` is the consumer's cue to fall back to a fresh `thread/start`. `ProcessHost::shutdown()` ends the slot's process GRACEFULLY (the writer channel closes ŌåÆ child stdin EOF ŌåÆ codex exits ŌåÆ the reader delivers `Exited` per sink), backed by a 5 s watchdog that force-kills a child ignoring the EOF (no zombies until app quit) ŌĆö the idle reaper's primitive; the next `ensure()` respawns with a bumped generation.

**Tests**: framing + frozen fixtures in `codex/protocol.rs`; pending map, resolution, routing/exit-dedupe and resume classification in `codex/host.rs`. The `#[ignore]`d spikes (`session_spike`, `sessions_spike`, `appserver_spike`, `conductor_instances_spike`, `phase4_worktree_spike`, `phase5_output_schema_spike`, `phase5_autonomy_loop_spike`) are the live proofs against the real codex CLI: `cd src-tauri && cargo test <name> -- --ignored --nocapture`.

## Native Codex sessions (`src-tauri/src/codex/sessions.rs`)

Plain agentic Codex sessions driven natively and mirrored into the SwarmZ UI. Process strategy **(b)** ŌĆö each session owns a PRIVATE `ProcessHost` slot. Unlike the orchestrator, a session keeps codex' STANDARD harness intact ŌĆö **NO `dynamicTools`, NO `developerInstructions`**.

**Registry & profile**: a static `SESSIONS` map (`session_id ŌåÆ SessionState`) holds each session's private `ProcessHost`, `thread_id`, spawn `generation` (respawn detection ŌåÆ transparent resume), profile (`cwd`, optional `model`/`effort`, `access`), `current_turn_id`, a `busy` flag (one turn per session ŌĆö claimed synchronously in `send`), the unanswered approval `Responder`s ŌĆö each stored WITH its Rust-classified routing class (`PendingApproval {responder, escalation}`, the strict Conductor response path's source of truth) ŌĆö and the session's `EventSink`. `session_id` is assigned by the frontend.

**Access ŌåÆ sandbox** (exact wire strings verified against 0.144.1): `workspace` = sandbox `workspace-write` + approvalPolicy `on-request`; `full` = sandbox `danger-full-access` + approvalPolicy `never`. **`model` and `effort` are per-turn overrides** (both ride on every `turn/start`) ŌĆö that is what lets the user change model/effort mid-session. An **access change** takes effect on the NEXT turn via a per-turn override attached only when access changed since the last turn.

**Commands** (all async, in lib.rs): `vibe_session_start` ŌåÆ `{thread_id}`; `vibe_session_resume` ŌåÆ `{thread_id, resumed}` (a `ThreadNotFound` rollout-gone falls back to a fresh `thread/start`); `vibe_session_send` ŌåÆ `{turn_id}` **non-blocking** (a busy session errors; optional `output_schema` rides as the ONE-TURN-ONLY codex `outputSchema` param ŌĆö Phase 5's structured status reports, live-verified by `phase5_output_schema_spike`); `vibe_session_interrupt`; `vibe_session_respond_approval` (decision Ōłł `accept|acceptForSession|decline|cancel`, resolves the remembered `Responder`; `require_routine: true` is the Conductor's STRICT path ŌĆö the routine-class check and the responder removal run atomically under the session lock, a destructive approval refuses and STAYS pending for the human); `vibe_session_set_access`; `vibe_session_set_model_effort`; `vibe_session_close` (best-effort interrupt, answer every pending approval with `cancel`, unregister, drop the entry); **Phase 4**: `vibe_session_steer` (`turn/steer` into the RUNNING turn with the race-safe `expectedTurnId` ŌĆö live-verified on 0.144.1, the turn absorbs the instruction; a lost race errors with a stable `steer-race:` prefix so the frontend falls back to a normal send ŌĆö including the EARLY no-turn case: Rust clears `current_turn_id` on `turn/completed` before the frontend busy flag clears, so that window is the same race), `vibe_session_set_cwd` (profile + live `thread/settings/update {cwd}` ŌĆö live-verified: the next turn runs in the new cwd), `vibe_session_review` (detached `review/start` on the session's thread ŌĆö resumes the parent thread first after a respawn; collects the review thread's events on a temporary route until `turn/completed`, returns `{status, review, review_thread_id}` where `review` = `exitedReviewMode.review`/final agentMessage; approvals on the review thread are cancelled).

## Conductor brain: Codex app-server (one instance per project)

**Process lifecycle** (`src-tauri/src/orchestrator/appserver.rs`, a thin consumer of `src-tauri/src/codex/`): ONE Conductor instance PER PROJECT ŌĆö a `ProcessHost` slot + ONE DISPATCHER TASK PER PROCESS GENERATION (`INSTANCES` map keyed by `project_id`; the empty key is the neutral probe used by status/model queries without a project). The per-generation dispatchers are the event-generation TAG: handlers compare the event's generation with the CHAT's (`chat.generation`) ŌĆö a straggler from a dead process can never mutate turn state of a chat that already resumed onto a newer one, and a generation's `Exited` fails only turns still running on it (`chat.generation <= event_gen`). Each project's chats multiplex over that project's long-lived `codex app-server` child, spawned lazily on first use ŌĆö the Settings `codexPath` override is passed by the frontend on every `chat_start`/`chat_status` call. `chat_send` claims the chat's turn slot (`done_tx`) ATOMICALLY in one lock section, so parallel sends can't overwrite each other. On process death, running turns of THAT project resolve as failed and its chats get `turn_failed`/`warning` events ŌĆö other projects' instances are untouched; the next call respawns and chats transparently `thread/resume` before the next turn (rollouts persist on disk and codex restores the declared dynamic tools from them). A `ThreadNotFound` resume failure surfaces as an error and the frontend controller handles it by starting a new thread. **Idle reaping**: a background task (started with the first instance, tick 60 s) `shutdown()`s any instance's process after **15 minutes** (`IDLE_REAP_SECS`) without chat activity (commands or routed events ŌĆö a streaming turn keeps it warm) and without a running turn. TOCTOU-safe via the per-instance **lifecycle lease** (a tokio mutex): spawn-touching operations touch the idle clock, THEN take the lease; the reaper `try_lock`s and RE-CHECKS idle + running-turn under the lease before shutting down ŌĆö a racing send either resets the clock in time or blocks until the shutdown finished and simply respawns. The instance, its chats and thread ids survive a reap; the next message is just a respawn + transparent resume. Closing a project TAB does not touch its instance. Live-verified end-to-end by the `#[ignore]`d `conductor_instances_spike` (two instances, separate processes/cwds, correct event routing, respawn transparency of one never affecting the other).

**Chat Ōåö thread mapping**: `orchestrator_chat_start(codex_path, persona, project)` runs the `initialize` handshake (`capabilities.experimentalApi: true`) and `thread/start` with **`cwd` = the PROJECT DIR** (fallback home when the folder is gone), `sandbox: "read-only"`, `approvalPolicy: "never"`, the whole registry adapted to the experimental `dynamicTools` field, and the compiled instructions (persona + project + scoped memory + operative core) as `developerInstructions`. Persona AND project are passed by the frontend on start/resume and captured on the chat (reused for transparent resumes); memory (global + project scope) is read fresh each start/resume. `orchestrator_chat_send(chat_id, text, model?, effort?)` prepends a fresh PROJECT-scoped fleet-summary line (`[fleet status: ŌĆ”]`, best-effort via `run_tool("fleet_snapshot")` with the chat's `project_id`) to the user text, starts the turn and resolves with `{status, text}` on `turn/completed`; `model`/`effort` are per-turn `turn/start` overrides. `thread/tokenUsage/updated` re-emits as a `token_usage` chat event for the context gauge. `orchestrator_chat_status(codex_path, project?)` reports `{running, version, account}`, reuses ANY alive instance before spawning, and never rejects for a dead process; `codex_list_models` does the same.

**Dynamic-tools bridge** (`orchestrator/adapter.rs`): `item/tool/call` server requests are answered from the bus ŌĆö `orchestrator::run_tool` with args normalized and the instance's `project_id` attached (the executors scope on it), the result adapted to `{success, contentItems:[{type:"inputText", text}]}`. Each call runs in its own task so deltas keep streaming during tool roundtrips.

**Event stream**: `orchestrator://chat-event` `{chat_id, kind, data}` with kinds `turn_started`, `delta`, `message`, `tool_call` `{tool, args_summary}`, `tool_done` `{tool, ok}`, `token_usage`, `turn_completed`, `turn_failed`, `warning`. Typed wrappers + `onChatEvent` in `src/lib/orchestrator/chat.ts`.

**Approvals & safety**: under read-only sandbox + `approvalPolicy: "never"` + the instructions, approval requests must never occur ŌĆö if one arrives anyway it is auto-DECLINED and any other server-initiated request is refused with a JSON-RPC error; both emit a `warning` event. Known gap: MCP servers from the user's codex config still attach to orchestrator threads ŌĆö the instructions forbid using non-SwarmZ tools, but that is instruction-level, not enforced.

## Performance invariants

Heavy Tauri commands (usage parsing, git subprocesses, file walks) run async + `spawn_blocking` ŌĆö sync commands execute on the main thread and may only do fast, non-blocking work. The Vibe UI rests on three measured properties: (a) the item feed is virtualized (`virtua`); (b) per-id primitive selectors keep re-renders scoped ŌĆö a delta re-renders exactly its own row, never the rail; (c) the persist snapshot is cheap and debounced (~800 ms). Never regress these: a store selector that returns a fresh array, an un-memoized feed row, or a per-delta persist would each break one of them.
