# v1.1.3 — Bounded autonomy and visible reviews

SwarmZ 1.1.3 makes autonomous work more configurable and predictable. It
adds user-owned budgets and command rules, keeps review work visible and
bounded, and tightens Orchestrator worker defaults around reasoning and Git.

## Configurable autonomy

- The autonomy budget is now enabled by default at 5 consecutive turns and
  20 turns per hour, with both values editable in Settings.
- The budget can be disabled completely for unlimited autonomous turns.
  Disabling clears existing counters and breaker latches.
- Worker reasoning now defaults to `high`. `xhigh` and `max` are capped back
  to high unless the Orchestrator explicitly identifies genuinely critical
  authentication, payment, cryptographic or irreversible-migration work.

## Persistent command permissions

- Supported command approvals now offer **Always allow**. The exact
  Codex-proposed argument rule is stored in SwarmZ and can be edited or
  deleted in Settings.
- Rule mutations are serialized, native-first and bounded. Rust revalidates
  every rule and only auto-accepts an exact future Codex proposal.
- Orchestrator-owned `swarm/*` worktrees receive a narrow Git lane for an
  exact hook/signing-disabled commit of staged changes and a non-force push
  to their own live branch. Rust binds the grant to the canonical worktree,
  gitdir, branch and request directory; staging and all other Git commands
  remain approval-gated.

## Review loops under control

- The automated code-review loop is off by default. Settings can enable it
  and choose a hard maximum number of iterations per feature worktree.
- Review counts persist independently of temporary workers and a worktree can
  have only one review in flight. Review findings must be fixed in the same
  worktree; the Orchestrator is explicitly forbidden from creating a new
  worktree for those fixes.
- Independent worktrees can be reviewed concurrently. Every acknowledged
  detached review appears as a temporary agent lane in Fleet, and running
  reviews are never evicted by the completed-review history cap.

## Safety and reliability

- Review event listeners and native thread routing are installed before a
  detached review becomes visible, closing fast-start and fast-finish races.
- Failed worktree reassignments tear down the unacknowledged backend instead
  of leaving native cwd and UI state split.
- Persistent approval rules are fully mirrored into Rust before Settings
  hydration is considered ready.
