# v1.0.3 — model-aware Conductor

SwarmZ's Conductor can now choose the right Codex model and reasoning effort
for each delegated task from the live catalog exposed by the installed Codex
app-server.

## Live model capabilities in every Conductor context

- New Conductor chats receive a bounded live model-catalog snapshot in their
  developer instructions, including exact model override values and each
  model's supported and default reasoning efforts.
- The new `list_models` tool refreshes that catalog on demand, so the
  Conductor does not rely on hardcoded or stale model names.
- Existing persisted chats transparently start a fresh backend context after
  the toolset upgrade while keeping their visible history in SwarmZ.

## Safe per-agent model routing

- `spawn_agents` and `set_agent_config` validate explicit model/effort pairs
  against the live catalog before creating a worktree or starting a session.
- The Conductor is instructed to use faster, economical models for clear
  research and mechanical work, and stronger models or higher effort for
  ambiguous, critical and architectural work.
- Omitting a model keeps the user's Codex configuration; an omitted spawn
  effort keeps SwarmZ's medium default.

## Ultra is not an effort level

`ultra` is a multi-agent execution mode, not a single-agent reasoning effort.
It is removed from every picker and catalog view and is refused at the
Conductor, session-backend and persisted-state boundaries. `max` remains a
normal catalog-driven effort when a model advertises it.

## Runtime choices are visible

Fleet cards now show each agent's current `model · effort`. Conductor activity
chips show the same runtime metadata, keep the original spawn choice as an
audit snapshot after an agent closes, and follow live retuning while the agent
is still running. The existing focus-stage chip remains available for direct
inspection and editing.
