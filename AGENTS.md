# AGENTS.md

SwarmZ ‚Äî run and monitor a swarm of native Codex agents (`codex app-server` sessions), organized in project tabs, with an AI Conductor PER PROJECT on top. React frontend + native macOS app (Tauri 2 / Rust). Deep per-subsystem details live in `docs/ARCHITECTURE.md` ‚Äî read the relevant section there (or the code) before changing a subsystem, and keep it in sync.

**A native Codex-only swarm manager.** SwarmZ is the codex-only application: the old terminal/PTY/grid subsystem, the Claude runtime, the OpenRouter brain and voice dictation were removed in the `rebuild/codex-only-v1` rebuild (git history and `main` preserve the former terminal-grid app; the rebuild plan is `docs/plans/rebuild-codex-only-v1.md`). What exists now is one coherent product:

- **Projects & swarm data model** ‚Äî sessions organized in per-folder project tabs (canonical-path dedupe), each session a named agent scoped to its project.
- **One Conductor per project** ‚Äî its own `codex app-server` process slot, its own chats, its own scoped memory and a project-scoped tool arsenal, on the swarm-doctrine operative core.
- **A fully capable engineering lead** ‚Äî the Conductor decomposes goals onto agents unprompted, steers/reviews them, manages worktrees, sets timers, writes plan documents, explores the project tree read-only (`list_files`/`read_file`), routes routine approvals, and selects agent runtimes from the live Codex `list_models` catalog ‚Äî a 34-tool arsenal. Large goals run as MISSIONS: one task-board plan under `.swarmz/plans/` is the durable state (checkbox lines, updated after every change, the reconciliation anchor after a breaker trip or restart).
- **Genuinely autonomous** ‚Äî a trigger router turns fleet events (agent finished/blocked, routine-approval escalation, timer, long-idle lanes, PR changes) into budget-gated autonomous Conductor turns; finish/blocked events that pile up behind a running turn COALESCE into one `[fleet events]` turn (one budget unit ‚Äî a 15-agent wave no longer trips the breaker at the 6th finish); agents deliver `outputSchema`-forced status reports (`expect_report`), finished lanes can auto-review (Settings toggle), and the learning doctrine stores preference observations via `remember`.
- **Vibe v3 UI** ‚Äî the design-token system + fleet-grid/focus-stage/Conductor-sidebar UI, diffs via @pierre/diffs (`DESIGN.md`).
- **GitHub over the LOCAL `gh` CLI (no OAuth ever, no token)** ‚Äî read-only detection + panel always; an OPT-IN master toggle arms the Conductor's 7 PR tools, the Rust write gate, the PR watcher (`github://pr-changed` ‚Üí Deck ticker + `pr-changed` autonomous turns for watched/auto-review PRs) and the routine-classification of the two sanctioned agent-run gh writes.

New features are native-only Codex; do not reintroduce terminal/PTY/web-mode paths. (Inline `(Phase N)` tags below are historical markers for where a subsystem entered ‚Äî the app is complete, not mid-rebuild.)

## Stack & commands

- React 19 + TypeScript + Tailwind v4 (dark only ‚Äî follow `DESIGN.md`), Zustand, pnpm
- Type-check: `./node_modules/.bin/tsc --noEmit` ¬∑ Build: `pnpm build` ¬∑ Dev: `pnpm tauri dev` ¬∑ Frontend unit tests: `pnpm test` (vitest, pure functions ‚Äî `src/**/*.test.ts`, no DOM) ¬∑ Rust tests: `cargo test` in `src-tauri/`

## Load-bearing invariants (don't break these)

- A `useSyncExternalStore`/zustand selector must never return a freshly-built array of fresh objects (even under `useShallow`) ‚Äî it never stabilizes across `getSnapshot` calls and infinite-loops. Select a **primitive signature** and rebuild the objects in `useMemo` (see the Deck's `useTriageRows`). The rail cards, feed rows, approval takeover and cross-session banner all obey this: each selects only its own item/session by id (or a first-pending-id/count signature), so a streaming delta on one session/item re-renders exactly that row ‚Äî never the rail or a sibling.
- Session transcript items are **normalized by id** (`items: Record<id, VibeItem>` + `order`); a streaming delta must replace ONLY its own item object so every other item keeps its reference (the identity-preservation lesson ‚Äî one row re-renders per delta batch). Don't rebuild the whole item array on a patch. Two authorship fields persist (hydration-tolerant, drive the who-did-what UI): a user item's `via: "conductor"` (a prompt the Conductor injected via `prompt_agent`/`spawn_agents`, rendered "// via Conductor"; the human `@session`/composer path is unmarked) and an approval item's `decidedBy: "conductor" | "human"` (`respondApprovalStrict` vs `respondApproval` ‚Äî the feed shows "‚üê Conductor ¬∑ routine" on auto-decided approvals).
- A native session runs **one turn at a time over one private `codex app-server` process** (`busy` claimed synchronously in `send`); a busy session refuses a second `turn/start` ‚Äî but since Phase 4 the orchestrator's `prompt_agent` STEERS a busy session instead (`turn/steer` with the race-safe `expectedTurnId`; a lost race falls back to a normal next-turn send ‚Äî live-verified on 0.144.1). The human composer still waits. On close, **every remembered approval `Responder` must be answered** (with `cancel`) before the entry is dropped, or the child process hangs waiting.

The budget state is PERSISTED (key `autonomyBudgets`, hydrated FIRST in `hydrate()` before the project hydration that gates every delivery path) ‚Äî an app relaunch/HMR must never mint a fresh allowance or un-latch a tripped breaker; budget booking is a reservation released only for turns that NEVER started. FAIL-CLOSED persistence: `loadAutonomyBudgets` THROWS on a genuine read error (only a missing key is null), and the hydrate then `latchAutonomyUnavailable()`s ‚Äî a GLOBAL latch that refuses every project's autonomous turns until a human message clears it (`autonomyUnavailable`, observable) rather than treating an unreadable/corrupt store as a fresh, un-tripped allowance. Since Phase 5 every EVENT trigger additionally goes through the trigger router (`triggers.ts`: dedupe per (project, kind, subject) ‚Äî finish triggers per busy-cycle generation `session#gen` ‚Äî, optional out-of-chain `prepare` for slow work, per-project serialization, bounded retries, closed-tab muting for the event kinds) ‚Äî new autonomous triggers must go through the router AND `runAutonomousTurn`, never around either; finish/blocked triggers COALESCE (`combineFleetEvents`): everything that accumulates behind a running turn delivers as ONE `[fleet events]` turn costing ONE budget unit (‚â§8 events per turn), so a busy fleet spends the 5-consecutive allowance per WAVE instead of per agent ‚Äî the breaker semantics themselves are unchanged; the autonomous marker message carries `autonomous: true` + `trigger` (persisted). Every loop event ‚Äî INCLUDING the routine-approval escalation ‚Äî is gated on conductor-involvement; purely human sessions never wake the loop. Everything interpolated into an autonomous wire (reports, messages, questions, review output, approval summaries, timer notes, agent names) is UNTRUSTED: single-line-flattened (`clip`/`inlineField`/report caps ‚Äî which also map the C1 range and the Unicode line/para separators U+0085/U+2028/U+2029 to a space, so no renderer can be tricked into a fresh line) AND quoted as a delimited literal via `JSON.stringify` (a `"`/`\` inside can't visually escape and pose as wire text), labeled as data, and doctrinally covered by the operative core's untrusted-event-data rule. OUTWARD-facing github writes (`create_pr`, `comment_pr`, `review_pr` with `post:true`) are REFUSED during an autonomous turn unless the user opted in (`settings.autonomousGithubWrites`, default off) ‚Äî the executors read `isAutonomousTurnInFlight(ctx.chatId)` (set by `runAutonomousTurn` around its dispatch); a human-triggered turn keeps them under the master toggle, so a prompt-injected autonomous cascade can never push/post/approve on the user's repo, only PROPOSE it.
- Global shortcuts early-return while a dialog is open (‚åòK may close its palette, ‚åòN its drawer); ‚åòW is always `preventDefault`ed and closes the focused session (busy ‚Üí confirm) ‚Äî never a native tab close.
- Sync Tauri commands run on the main thread ‚Äî anything blocking (subprocesses, file walks) must be async + `spawn_blocking`.
- Quit flushes **all** debounced persists (`flushAllPersists`); the quit/close choreography in `lib/quit.ts` + `lib.rs` looks redundant but every leg is load-bearing (window-state save, updater restart code ‚Äî the `ExitRequested` handler in `lib.rs` must stay as is).
- Title-bar dragging relies **only** on `data-tauri-drag-region="deep"` on the `<header>` in `TitleBar.tsx` ‚Äî WKWebView ignores `-webkit-app-region`, so the `drag-region`/`no-drag` CSS classes are no-ops on macOS. `"deep"` makes the whole subtree draggable; Tauri's `drag.js` auto-excludes real `<button>`/`<input>`/`role` elements. The project tabs are plain `<div>`s (activate + close buttons as siblings inside), so they opt out explicitly via `data-tauri-drag-region="false"` ‚Äî keep that attribute when touching the tab markup. Don't switch back to bare `data-tauri-drag-region` (drags only direct hits on the header itself = just the flex gaps).

## Releases & updates

- Auto-updater polls `latest.json` on GitHub Releases (`lib/updates.ts`; pill in `TitleBar.tsx`, toggle in Settings). About shows `__APP_VERSION__` (Vite define). Builds are ad-hoc signed only.
- Release checklist: bump version in `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml` (keep in sync) ‚Üí create `docs/release-notes/v<X.Y.Z>.md` (CI fails without it) ‚Üí tag `v<X.Y.Z>` + push ‚Üí `.github/workflows/release.yml` publishes incl. updater manifest.
- Updater signing key: `~/.tauri/swarmz_updater.key` (no password), repo secret `TAURI_SIGNING_PRIVATE_KEY`. Losing it breaks updates for existing installs.
