# AGENTS.md

SwarmZ — run, tile and monitor multiple Claude Code, Codex and shell agents in parallel. React frontend + native macOS app (Tauri 2 / Rust). Deep per-subsystem details live in `docs/ARCHITECTURE.md` — read the relevant section there (or the code) before changing a subsystem, and keep it in sync.

**Native-only app.** The old browser/Node web mode has been removed. New features are native-only; use Tauri/Rust commands for backend capabilities and do not reintroduce web-mode parity paths.

## Stack & commands

- React 19 + TypeScript + Tailwind v4 (dark only — follow `DESIGN.md`), Zustand, xterm.js, pnpm
- Type-check: `./node_modules/.bin/tsc --noEmit` · Build: `pnpm build` · Dev: `pnpm tauri dev` · Frontend unit tests: `pnpm test` (vitest, pure functions — `src/**/*.test.ts`, no DOM) · Rust tests: `cargo test` in `src-tauri/`

## Load-bearing invariants (don't break these)

- PTYs die exactly on store removal (`removeAgent`/`removeFloatingTerminal` → `destroyTerm`), **never** on React unmount — that's what lets panes remount across workspaces. All grids stay mounted; inactive ones are `visibility:hidden`. The `WorkspaceLayer` and `VibeLayer` are the same deal: both stay mounted as `absolute inset-0` siblings, the inactive one only `invisible pointer-events-none` — a `⌘⇧V` mode switch must never unmount either (PTYs + vibe scroll survive).
- A `useSyncExternalStore`/zustand selector must never return a freshly-built array of fresh objects (even under `useShallow`) — it never stabilizes across `getSnapshot` calls and infinite-loops. Select a **primitive signature** and rebuild the objects in `useMemo` (see the Deck's merged triage `useTriageRows`). The Vibe rail cards, feed rows, approval takeover and cross-session banner all obey this: each selects only its own item/session by id (or a first-pending-id/count signature), so a streaming delta on one session/item re-renders exactly that row — never the rail or a sibling.
- Vibe transcript items are **normalized by id** (`items: Record<id, VibeItem>` + `order`); a streaming delta must replace ONLY its own item object so every other item keeps its reference (the identity-preservation lesson — one row re-renders per delta batch). Don't rebuild the whole item array on a patch.
- A native Vibe session runs **one turn at a time over one private `codex app-server` process** (`busy` claimed synchronously in `send`); a busy session refuses a new turn (unlike a pane, which queues text) — both the controller and the orchestrator's session `prompt_pane` enforce this. On close, **every remembered approval `Responder` must be answered** (with `cancel`) before the entry is dropped, or the child process hangs waiting.
- Into-terminal text (drops, dictation, command inserts) goes through `term.paste()`, never raw `ptyWrite` — bracketed paste is what makes Claude attach image paths; submit is a **separate** `\r` write.
- `ConEmuANSI=ON` in the PTY env is the OSC 9;4 opt-in (pane status badges); don't remove. OSC 21337 is pre-wired but dormant.
- Persistence distinctions: `workspacePresets` `null` = seed, `[]` = stays empty · limits `null` = no usable login (hide meters) vs reject = transient (keep last values) · `Agent.resume` is transient, `--resume` is injected only at PTY spawn · `settings.orchestratorPersona` unset = the Maestro seed (never write the seed out; a "Reset" clears back to `undefined`).
- Orchestrator instructions come from ONE function, `orchestrator/persona.rs` `build_instructions(persona, memory)` (both provider paths). Its `OPERATIVE_CORE` block (tools + safety + delivery contract) is the WORD-FOR-WORD former `ORCHESTRATOR_INSTRUCTIONS` and is **not** persona-overridable — persona only prepends voice/self-image, it must never be able to weaken a guardrail. Extend the core (new tools/rules), never reword its existing sentences (content tests + the guardrail invariant depend on the text). Memory (`orchestrator-memory.md`, FIFO-capped 20/2000) is written only via the explicit `remember` tool — no silent auto-write; the model proposes uncertain facts and stores after confirmation.
- Custom agents: a pane/session runs AS an agent by carrying only its `agentSlug` (identity chrome looks the rest up from the Library by slug — the disk stays the source of truth). The compiled context is `<dir>/.compiled.md`, rewritten fresh on every start (frozen-at-start snapshot, like Maestro's memory); the fixed `context.rs::operative_block` is the agent guardrail (soul/agent.json set voice, never safety). Memory self-maintenance depends on the agent's folder being a sandbox writable root: for an agent Vibe session under workspace access, `vibe_session_start`/`_resume` MUST receive its `agent_slug` and `turn_params` MUST attach the object-form `sandboxPolicy` with `writable_roots` on every turn — drop either and the agent's `memory.md` writes need an approval again (guarded by the `memory_writable_root_spike` live test). Maestro reaches specialists ONLY through `list_agents` + `create_panes` `agent` (the catalog stays single-source in `registry.rs`).
- Global shortcuts early-return while a dialog is open (⌘K/⌘⇧K may close their palettes); ⌘W is always `preventDefault`ed; the ⌘⇧K branch stays before plain ⌘K.
- Sync Tauri commands run on the main thread — anything blocking (subprocesses, file walks, keychain, inference) must be async + `spawn_blocking`. `pty_write`/`pty_spawn`/`pty_resize` stay sync on purpose (ordering); writes are non-blocking via the per-session writer channel.
- Quit flushes **all** debounced persists (`flushAllPersists`); the quit/close choreography in `lib/quit.ts` + `lib.rs` looks redundant but every leg is load-bearing (window-state save, updater restart code).
- Title-bar dragging relies **only** on `data-tauri-drag-region="deep"` on the `<header>` in `TitleBar.tsx` — WKWebView ignores `-webkit-app-region`, so the `drag-region`/`no-drag` CSS classes are no-ops on macOS. `"deep"` makes the whole subtree draggable; Tauri's `drag.js` auto-excludes real `<button>`/`<input>`/`role` elements. The workspace tab is a plain `<div>` (activate + close buttons as siblings inside), so it opts out explicitly via `data-tauri-drag-region="false"` — keep that attribute when touching the tab markup. Don't switch back to bare `data-tauri-drag-region` (drags only direct hits on the header itself = just the flex gaps).

## Releases & updates

- Auto-updater polls `latest.json` on GitHub Releases (`lib/updates.ts`; pill in `TitleBar.tsx`, toggle in Settings). About shows `__APP_VERSION__` (Vite define). Builds are ad-hoc signed only.
- Release checklist: bump version in `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml` (keep in sync) → create `docs/release-notes/v<X.Y.Z>.md` (CI fails without it) → tag `v<X.Y.Z>` + push → `.github/workflows/release.yml` publishes incl. updater manifest.
- Updater signing key: `~/.tauri/swarmz_updater.key` (no password), repo secret `TAURI_SIGNING_PRIVATE_KEY`. Losing it breaks updates for existing installs.

## Ground rules

1. **Never start the dev server** or any long-running app process (`pnpm tauri dev`, `pnpm dev`, …) without explicit approval first. Type-checks, builds and `cargo test` are fine.
2. **Never commit or push on your own.** Only when explicitly asked or approved.
3. **Keep docs in sync.** If a change affects anything stated in `AGENTS.md`, `docs/ARCHITECTURE.md` or `README.md`, update them in the same change — proactively, without being asked.
