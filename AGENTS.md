# AGENTS.md

SwarmZ — run and monitor a swarm of native Codex agents (`codex app-server` sessions), organized in project tabs, with an AI Conductor PER PROJECT on top. React frontend + native macOS app (Tauri 2 / Rust). Deep per-subsystem details live in `docs/ARCHITECTURE.md` — read the relevant section there (or the code) before changing a subsystem, and keep it in sync.

**Codex-only rebuild (branch `rebuild/codex-only-v1`, Phase 3 done).** The old terminal/PTY/grid subsystem, the Claude runtime, the OpenRouter brain and voice dictation are REMOVED (Phase 1 of `docs/plans/rebuild-codex-only-v1.md`; git history and `main` preserve everything); the projects & swarm data model is in (Phase 2); the Conductor is one instance per project — own process slot, own chats, own memory, project-scoped tools, swarm-doctrine operative core (Phase 3). New features are native-only Codex; do not reintroduce terminal/PTY/web-mode paths.

## Stack & commands

- React 19 + TypeScript + Tailwind v4 (dark only — follow `DESIGN.md`), Zustand, pnpm
- Type-check: `./node_modules/.bin/tsc --noEmit` · Build: `pnpm build` · Dev: `pnpm tauri dev` · Frontend unit tests: `pnpm test` (vitest, pure functions — `src/**/*.test.ts`, no DOM) · Rust tests: `cargo test` in `src-tauri/`

## Load-bearing invariants (don't break these)

- A `useSyncExternalStore`/zustand selector must never return a freshly-built array of fresh objects (even under `useShallow`) — it never stabilizes across `getSnapshot` calls and infinite-loops. Select a **primitive signature** and rebuild the objects in `useMemo` (see the Deck's `useTriageRows`). The rail cards, feed rows, approval takeover and cross-session banner all obey this: each selects only its own item/session by id (or a first-pending-id/count signature), so a streaming delta on one session/item re-renders exactly that row — never the rail or a sibling.
- Session transcript items are **normalized by id** (`items: Record<id, VibeItem>` + `order`); a streaming delta must replace ONLY its own item object so every other item keeps its reference (the identity-preservation lesson — one row re-renders per delta batch). Don't rebuild the whole item array on a patch.
- A native session runs **one turn at a time over one private `codex app-server` process** (`busy` claimed synchronously in `send`); a busy session refuses a new turn — both the controller and the orchestrator's `prompt_pane` enforce this. On close, **every remembered approval `Responder` must be answered** (with `cancel`) before the entry is dropped, or the child process hangs waiting.
- Persistence distinctions: `schemaVersion` is `2` — stores below it get the one-time dead-key cleanup + stamp (`planSchemaMigration`; a NEWER stored version is never clobbered) · closing a project tab only stamps `closedAt` — NEVER delete its sessions (and never its Conductor chats or instance); reopening the same folder must resurface them under the SAME project id (dedupe by canonical dir) · pre-v2 sessions hydrate through `adoptSessions` (projects derived from `projectDir`s) — keep that path tolerant · orchestrator chats carry a `projectId` since Phase 3; chats without one migrate on hydrate via `assignChatsToProjects` (rule, in order: most recently touched session's project → last active project → "" invisible), which is why chats hydrate LAST (after projects + sessions) — the migration/collapse/persist runs ONLY when both hydrated cleanly AND ≥1 project exists, never downgrades a non-empty `projectId` to "", exempts ""-chats from the cap and heals them via a projects-store watcher · `settings.orchestratorPersona` unset = the Maestro seed (never write the seed out; "Reset" clears back to `undefined`) · orchestrator chats hydrate tolerantly (pre-rebuild `provider`/`wire` fields are dropped; an OpenRouter-era model stamp is cleared).
- **One Conductor instance per project**: `orchestrator/appserver.rs` keeps a `ProcessHost` slot PER `project_id` with ONE DISPATCHER PER PROCESS GENERATION — handlers compare the event's generation against `chat.generation`, so a straggler from a dead process never mutates a chat that already resumed onto a newer one, and a generation's `Exited` fails only turns still running on it. A chat belongs to exactly one instance (its `ProjectContext` is captured at start/resume, thread cwd = the project dir, sandbox stays read-only, approvalPolicy "never", approvals auto-declined); `chat_send` claims the chat's turn slot ATOMICALLY (parallel sends can't overwrite each other's done_tx). Tool calls carry the project — executors resolve sessions ONLY within it (`@name` must be unique in the project; project id `""` = unscoped, normalized in `run_tool`) and `fleet_snapshot`/the fleet-status prefix cover only that project. The idle reaper (15 min without chat activity or a running turn, checked every 60 s) `shutdown()`s only the PROCESS (graceful stdin-EOF, 5 s force-kill watchdog for a child ignoring it), TOCTOU-safe via the per-instance lifecycle lease: spawn paths touch the idle clock THEN take the lease, the reaper `try_lock`s and RE-CHECKS under it — instance, chats and thread ids survive, the next message respawns + transparently `thread/resume`s; one project's crash/reap must never touch another instance (live-verified by `conductor_instances_spike`).
- Conductor instructions come from ONE function, `orchestrator/persona.rs` `build_instructions(persona, project, memory)`. Its `OPERATIVE_CORE` block (v2, the swarm doctrine: the Conductor of THIS project decomposes goals onto agents UNPROMPTED — "delegating is your default") is **not** persona-overridable — persona only prepends voice/self-image, it must never be able to weaken a guardrail (never edit files, never push unprompted, the HUMAN holds final authority over agent approvals). Extend the core (new tools/rules — Phase 4 ADDS approval-routing sentences after the approval paragraph), never reword its existing sentences (content tests freeze them verbatim). Memory renders into the instructions as explicitly UNTRUSTED DATA (never instructions; the closing authority line is always the LAST content), the project block's name/dir are sanitized single-line quoted literals — neither a stored entry nor a folder name can weaken a guardrail. Memory is scoped — `orchestrator-memory/global.md` + `orchestrator-memory/<project_id>.md` (ids strictly validated, "global" reserved), FIFO-capped 20/2000 PER FILE, all ops under a per-file lock, the pre-Phase-3 single file migrates once (fallibly — a failed rename keeps serving the legacy file) — and is written only via the explicit `remember` tool (scope "project" default, "global" for cross-project preferences) — no silent auto-write; the model proposes uncertain facts and stores after confirmation.
- Worktree deletion is gated: silent removal only after a re-check at execution time; "Clean up safe" re-checks each worktree and never touches paths a live session works in; risky rows need a second armed click.
- Global shortcuts early-return while a dialog is open (⌘K may close its palette, ⌘N its drawer); ⌘W is always `preventDefault`ed.
- Sync Tauri commands run on the main thread — anything blocking (subprocesses, file walks) must be async + `spawn_blocking`.
- Quit flushes **all** debounced persists (`flushAllPersists`); the quit/close choreography in `lib/quit.ts` + `lib.rs` looks redundant but every leg is load-bearing (window-state save, updater restart code — the `ExitRequested` handler in `lib.rs` must stay as is).
- Title-bar dragging relies **only** on `data-tauri-drag-region="deep"` on the `<header>` in `TitleBar.tsx` — WKWebView ignores `-webkit-app-region`, so the `drag-region`/`no-drag` CSS classes are no-ops on macOS. `"deep"` makes the whole subtree draggable; Tauri's `drag.js` auto-excludes real `<button>`/`<input>`/`role` elements. The project tabs are plain `<div>`s (activate + close buttons as siblings inside), so they opt out explicitly via `data-tauri-drag-region="false"` — keep that attribute when touching the tab markup. Don't switch back to bare `data-tauri-drag-region` (drags only direct hits on the header itself = just the flex gaps).

## Releases & updates

- Auto-updater polls `latest.json` on GitHub Releases (`lib/updates.ts`; pill in `TitleBar.tsx`, toggle in Settings). About shows `__APP_VERSION__` (Vite define). Builds are ad-hoc signed only.
- Release checklist: bump version in `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml` (keep in sync) → create `docs/release-notes/v<X.Y.Z>.md` (CI fails without it) → tag `v<X.Y.Z>` + push → `.github/workflows/release.yml` publishes incl. updater manifest.
- Updater signing key: `~/.tauri/swarmz_updater.key` (no password), repo secret `TAURI_SIGNING_PRIVATE_KEY`. Losing it breaks updates for existing installs.

## Ground rules

1. **Never start the dev server** or any long-running app process (`pnpm tauri dev`, `pnpm dev`, …) without explicit approval first. Type-checks, builds and `cargo test` are fine.
2. **Never commit or push on your own.** Only when explicitly asked or approved.
3. **Keep docs in sync.** If a change affects anything stated in `AGENTS.md`, `docs/ARCHITECTURE.md` or `README.md`, update them in the same change — proactively, without being asked.
